# CPU 스케줄링 알고리즘

## CPU 스케줄링이란?

**CPU 스케줄러**는 준비 큐에서 다음에 CPU 코어를 사용할 프로세스를 결정합니다. 한 번에 하나의 프로세스만 CPU 코어를 사용할 수 있으므로 스케줄러는 이 결정을 지속적으로 내려야 합니다.

좋은 스케줄링은 매우 중요합니다: 대화형 프로그램이 얼마나 빠르게 반응하는지, 배치 작업이 얼마나 빠르게 완료되는지, 하드웨어가 얼마나 효율적으로 활용되는지를 결정합니다.

---

## 주요 용어와 지표

| 용어 | 정의 |
|---|---|
| **버스트 시간** | 프로세스가 필요로 하는 CPU 시간 (I/O 대기 제외) |
| **도착 시간** | 프로세스가 준비 큐에 들어온 시점 |
| **완료 시간** | 프로세스가 실행을 마친 시점 |
| **반환 시간** | 완료 시간 − 도착 시간 |
| **대기 시간** | 준비 큐에서 대기한 시간 (실행 중이거나 블록된 시간 제외) |
| **응답 시간** | 도착부터 첫 CPU 사용까지의 시간 |
| **처리량** | 단위 시간당 완료된 프로세스 수 |
| **CPU 사용률** | CPU가 유휴 상태가 아닌 시간의 비율 |

$$\text{반환 시간} = \text{완료 시간} - \text{도착 시간}$$
$$\text{대기 시간} = \text{반환 시간} - \text{버스트 시간}$$

---

## 스케줄링 유형

### 선점형 vs 비선점형

- **비선점형**: 프로세스가 시작되면 블록(I/O)되거나 종료될 때까지 실행됩니다. CPU를 빼앗을 수 없습니다. 단순하지만 대화형 시스템에 부적합합니다.
- **선점형**: OS가 실행 중인 프로세스를 중단하고 다른 프로세스로 전환할 수 있습니다 (예: 높은 우선순위 프로세스 도착 또는 타이머 만료). 실시간 및 대화형 시스템에 필수입니다.

---

## 1. FCFS (선입선처리)

**방법**: 도착 순서대로 프로세스를 실행합니다. 단순한 FIFO 큐. 비선점형.

### 예시

| 프로세스 | 도착 | 버스트 |
|---|---|---|
| P1 | 0 | 24 |
| P2 | 1 | 3 |
| P3 | 2 | 3 |

**간트 차트:**
```
| P1 (24) | P2 (3) | P3 (3) |
0        24       27       30
```

| 프로세스 | 완료 | 반환 | 대기 |
|---|---|---|---|
| P1 | 24 | 24 | 0 |
| P2 | 27 | 26 | 23 |
| P3 | 30 | 28 | 25 |
| **평균** | | **26** | **16** |

### 호위 효과 (Convoy Effect)

FCFS는 **호위 효과**로 어려움을 겪습니다: 짧은 프로세스들이 긴 프로세스 뒤에 갇힙니다. P2와 P3(각 3ms)는 P1(24ms)이 먼저 도착했다는 이유로 23ms와 25ms를 대기합니다.

**장점**: 구현이 단순하고 기아가 없습니다.
**단점**: 평균 대기 시간이 높고, 호위 효과, 대화형 시스템에 부적합합니다.

---

## 2. SJF (최단 작업 우선)

**방법**: **다음 CPU 버스트가 가장 짧은** 프로세스를 먼저 실행합니다. 비선점형 스케줄링에서 평균 대기 시간 최소화가 입증된 최적 알고리즘입니다.

### 비선점형 SJF

| 프로세스 | 도착 | 버스트 |
|---|---|---|
| P1 | 0 | 6 |
| P2 | 0 | 8 |
| P3 | 0 | 7 |
| P4 | 0 | 3 |

**간트 차트 (모두 0에 도착, 최단 순):**
```
| P4 (3) | P1 (6) | P3 (7) | P2 (8) |
0        3        9       16       24
```

| 프로세스 | 완료 | 반환 | 대기 |
|---|---|---|---|
| P4 | 3 | 3 | 0 |
| P1 | 9 | 9 | 3 |
| P3 | 16 | 16 | 9 |
| P2 | 24 | 24 | 16 |
| **평균** | | **13** | **7** |

### 선점형 SJF (SRTF — 최단 잔여 시간 우선)

새로 도착한 프로세스의 버스트가 현재 실행 중인 프로세스의 **잔여** 버스트보다 짧으면 즉시 선점합니다.

### 문제: 버스트 시간 예측

SJF는 다음 CPU 버스트를 미리 알아야 합니다 — 범용 스케줄링에서는 불가능합니다. 해결책: **지수 평균으로 예측**:

$$\tau_{n+1} = \alpha \cdot t_n + (1 - \alpha) \cdot \tau_n$$

여기서 $t_n$ = 실제 $n$번째 버스트, $\tau_n$ = 예측 $n$번째 버스트, $\alpha \in [0,1]$.

**장점**: 최적 평균 대기 시간 (비선점형), 최소 응답 시간 (SRTF).
**단점**: 실제 버스트 시간을 알 수 없음; 짧은 작업이 계속 들어오면 긴 작업에 기아 발생.

## 3. 라운드 로빈 (Round Robin)

**방법**: 각 프로세스는 **시간 퀀텀**(타임 슬라이스)이라는 고정된 시간만큼 CPU를 사용합니다. 보통 10~100ms. 퀀텀이 만료되면 프로세스가 선점되어 준비 큐의 뒤에 배치됩니다. FIFO 순서, 순환 회전.

**시분할 시스템**과 대화형 응답성을 위해 설계되었습니다.

### 예시

시간 퀀텀 = 4ms.

| 프로세스 | 도착 | 버스트 |
|---|---|---|
| P1 | 0 | 24 |
| P2 | 0 | 3 |
| P3 | 0 | 3 |

**간트 차트:**
```
| P1(4) | P2(3) | P3(3) | P1(4) | P1(4) | P1(4) | P1(4) | P1(4) |
0       4       7      10      14      18      22      26      30
```

P2는 t=7에 완료 (4ms 퀀텀 중 3ms만 필요).
P3는 t=10에 완료.
P1은 t=30에 완료.

### 시간 퀀텀 선택

**너무 크면** → FCFS로 퇴화. 긴 프로세스가 CPU를 독점. 응답 시간 저하.

**너무 작으면** → 잦은 컨텍스트 스위치가 지배. 퀀텀 = 1ms이고 컨텍스트 스위치 = 0.1ms이면 CPU 시간의 10%가 오버헤드로 낭비됩니다.

경험 법칙: **CPU 버스트의 80%가 퀀텀보다 짧아야 합니다.** 일반적인 값: 10~100ms.

**장점**: 공정하고 응답 시간이 좋으며 기아가 없습니다.
**단점**: SJF보다 평균 반환 시간이 높고 컨텍스트 스위치 오버헤드가 있습니다.

---

## 4. 우선순위 스케줄링

**방법**: 각 프로세스에 우선순위 번호를 할당합니다. CPU는 준비 큐에서 가장 높은 우선순위 프로세스를 실행합니다.

우선순위는 **정적** (생성 시 고정) 또는 **동적** (런타임에 조정)일 수 있습니다.

### 기아와 에이징

**기아**: 높은 우선순위 프로세스가 계속 도착하면 낮은 우선순위 프로세스는 절대 실행되지 않을 수 있습니다.

**에이징**: 오랫동안 대기한 프로세스의 우선순위를 점진적으로 높입니다. 기아를 방지합니다. 예: 15분마다 우선순위를 1씩 높임. 결국 모든 프로세스가 실행될 만큼 높은 우선순위를 얻습니다.

---

## 5. 다단계 피드백 큐 (MLFQ)

여러 큐로 나누고 프로세스의 동작에 따라 큐 간 이동을 허용합니다:

- 프로세스는 가장 높은 우선순위 큐에서 시작
- 전체 퀀텀을 사용하면 (CPU 집중적 동작) 낮은 큐로 이동 (더 큰 퀀텀, 낮은 우선순위)
- 퀀텀 만료 전에 CPU를 반납하면 (I/O 집중적 동작) 현재 큐 유지 또는 위로 이동
- 에이징: 오래 대기한 프로세스의 우선순위를 주기적으로 높임

MLFQ는 사전 지식 없이 프로세스 동작에 적응합니다. 실제 스케줄러의 기반입니다.

---

## 실세계: Linux CFS

Linux의 **완전 공정 스케줄러(CFS)**는 고정 시간 퀀텀이나 단순 우선순위 큐를 사용하지 않습니다.

**핵심 아이디어**: 모든 프로세스는 CPU 시간의 동등한 몫을 받을 자격이 있습니다. CFS는 각 프로세스가 받은 CPU 시간(`vruntime`)을 추적합니다. **가장 작은 vruntime**을 가진 프로세스가 항상 다음에 실행됩니다.

- `vruntime`은 실행 중에 증가하며 우선순위(nice 값)로 스케일됩니다: 낮은 우선순위 → vruntime이 더 빠르게 증가 → 더 빨리 선점됨
- **레드-블랙 트리**로 vruntime 순서로 정렬하여 O(log n) 선택
- 목표 지연: 설정 가능한 창 내에 모든 프로세스가 CPU를 받음 (기본 6ms)

```bash
# 프로세스의 스케줄러 통계 확인
cat /proc/<pid>/sched

# 스케줄링 정책 확인
chrt -p <pid>
```

Linux 스케줄링 클래스 (높은 우선순위 → 낮은 우선순위):
1. `SCHED_DEADLINE` — 데드라인이 있는 실시간 태스크의 EDF
2. `SCHED_FIFO` / `SCHED_RR` — POSIX 실시간 (고정 우선순위 1–99)
3. `SCHED_NORMAL` — 일반 프로세스용 CFS
4. `SCHED_BATCH` — 배치 작업용 CFS 변형
5. `SCHED_IDLE` — 다른 것이 없을 때만 실행

## 스케줄링 알고리즘 비교

| 알고리즘 | 선점형 | 기아 | 오버헤드 | 적합한 용도 |
|---|---|---|---|---|
| FCFS | 아니오 | 없음 | 매우 낮음 | 배치, 단순 시스템 |
| SJF (비선점) | 아니오 | 있음 | 낮음 | 버스트 시간을 아는 배치 작업 |
| SRTF | 예 | 있음 | 보통 | 평균 대기 최소화 |
| 라운드 로빈 | 예 | 없음 | 보통 | 시분할, 대화형 |
| 우선순위 | 둘 다 | 에이징 없으면 있음 | 낮음 | 우선순위 클래스가 있는 시스템 |
| MLFQ | 예 | 에이징 있으면 없음 | 높음 | 범용 |
| CFS | 예 | 없음 | 보통 | Linux 범용 |

## 6. 실시간 스케줄링

**실시간 시스템**은 타이밍 제약이 있습니다 — 태스크가 데드라인 전에 완료되어야 합니다.

### 하드 vs 소프트 실시간

| | 하드 실시간 | 소프트 실시간 |
|---|---|---|
| **데드라인 미스** | 시스템 실패 | 품질 저하 |
| **예시** | ABS 브레이크, 심박 조율기, 원전 | 비디오 스트리밍, VoIP, 게임 |
| **보장** | 모든 데드라인 충족 증명 필요 | 최선의 노력, 가끔 미스 허용 |

### Rate-Monotonic Scheduling (RMS)

주기적 태스크를 위한 **고정 우선순위** 알고리즘. 주기가 짧을수록 높은 우선순위를 할당합니다.

주기적 태스크 $T_i$의 주기 $P_i$와 최악 실행 시간 $C_i$가 주어졌을 때:

**CPU 사용률**: $U = \sum_{i=1}^{n} \frac{C_i}{P_i}$

**스케줄 가능성 테스트** (충분조건이지만 필요조건은 아님):

$$U \leq n \cdot (2^{1/n} - 1)$$

큰 $n$에서 이 값은 $\ln 2 \approx 0.693$으로 수렴합니다. 총 사용률 ≤ 69.3%이면 RMS가 모든 데드라인 충족을 보장합니다.

RMS는 고정 우선순위 알고리즘 중 **최적**입니다 — 어떤 고정 우선순위 알고리즘이 태스크 세트를 스케줄할 수 있다면 RMS도 가능합니다.

### Earliest Deadline First (EDF)

**동적 우선순위** 알고리즘. 매 스케줄링 시점에서 **가장 이른 절대 데드라인**을 가진 태스크가 가장 높은 우선순위를 받습니다.

**스케줄 가능성 테스트** (선점형 EDF의 필요충분조건):

$$U = \sum_{i=1}^{n} \frac{C_i}{P_i} \leq 1.0$$

EDF는 100% CPU 사용률을 달성하면서 모든 데드라인을 충족할 수 있습니다 — RMS의 69.3% 한계보다 우수합니다. 트레이드오프: 동적 우선순위는 스케줄러 오버헤드가 더 높고, 과부하 시 동작이 예측 불가능합니다.

### 실시간 우선순위 역전

**화성 패스파인더 (1997)**: 수저너 로버가 우선순위 역전으로 반복적인 시스템 리셋을 경험했습니다. 낮은 우선순위 기상 태스크가 높은 우선순위 버스 관리 태스크에 필요한 뮤텍스를 보유. 중간 우선순위 통신 태스크가 낮은 우선순위 태스크를 선점하여 뮤텍스를 해제하지 못하게 함. 워치독 타이머가 높은 우선순위 태스크의 정지를 시스템 행으로 해석하고 컴퓨터를 리셋했습니다.

**해결**: 우선순위 상속 (락 보유자의 우선순위를 임시로 높임) 또는 우선순위 천장 (각 락에 사용하는 가장 높은 우선순위 태스크의 우선순위를 사전 할당).

## 7. 멀티프로세서 스케줄링

### 대칭 멀티프로세싱 (SMP)

모든 CPU가 동등합니다. 각 CPU가 스케줄러를 독립적으로 실행합니다.

### 부하 분산

스케줄러가 CPU 간에 작업을 균등하게 분배해야 합니다:

- **푸시 마이그레이션**: 주기적 커널 스레드가 부하 불균형을 확인하고 과부하된 CPU에서 유휴 CPU로 프로세스 이동
- **풀 마이그레이션**: 유휴 CPU가 바쁜 CPU의 실행 큐에서 프로세스를 "훔침"

### 프로세서 어피니티

프로세스를 CPU 간 이동하면 L1/L2 캐시가 무효화됩니다. **소프트 어피니티**는 스케줄러가 같은 CPU에 프로세스를 유지하려는 선호. **하드 어피니티**는 사용자가 설정하는 필수 고정:

```c
cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(0, &cpuset);    // CPU 0에 고정
sched_setaffinity(pid, sizeof(cpuset), &cpuset);
```

### 갱 스케줄링 (Gang Scheduling)

병렬 프로그램(예: MPI)의 경우 프로세스의 모든 스레드를 다른 CPU에서 동시에 스케줄링합니다. 한 스레드가 디스케줄되면 다른 스레드가 동기화에서 스핀 대기하여 CPU를 낭비할 수 있습니다. 갱 스케줄링은 프로세스의 모든 스레드를 하나의 단위로 함께 스케줄링하여 이를 방지합니다.
