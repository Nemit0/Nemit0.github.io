# 데드락

## 데드락이란?

**데드락(교착상태)**은 프로세스 집합이 영구적으로 블록된 상황입니다. 각 프로세스가 집합 내 다른 프로세스가 보유한 자원을 기다리는 상태로, 어떤 프로세스도 진행할 수 없습니다.

고전적인 비유: 일차선 다리 양끝에서 각각 한 대의 차량이 들어온 상황. 상대방이 후진하지 않으면 전진할 수 없지만 둘 다 후진하지 않습니다.

```
프로세스 A: R1 보유 중, R2 대기 중
프로세스 B: R2 보유 중, R1 대기 중
→ 데드락
```

---

## 4가지 필요 조건 (코프만 조건)

Coffman 등(1971)은 데드락이 발생하려면 **4가지 조건이 모두 동시에** 성립해야 한다는 것을 발견했습니다. 하나라도 없으면 데드락은 불가능합니다.

### 1. 상호 배제 (Mutual Exclusion)

적어도 하나의 자원이 **비공유** 상태여야 합니다 — 한 번에 하나의 프로세스만 사용 가능합니다. 프로세스가 자원을 보유 중이라면 다른 프로세스는 대기해야 합니다.

예: 프린터, 뮤텍스, 파일에 대한 쓰기 락.

### 2. 점유와 대기 (Hold and Wait)

프로세스가 **적어도 하나의 자원을 보유**한 상태에서 **다른 프로세스가 보유한 추가 자원을 기다리고** 있어야 합니다.

프로세스가 자원을 보유하지 않은 상태에서만 요청할 수 있다면 이 조건은 성립하지 않습니다.

### 3. 비선점 (No Preemption)

자원을 프로세스로부터 강제로 빼앗을 수 없습니다. 자원은 해당 프로세스가 작업을 마친 후 **자발적으로** 해제됩니다.

OS가 자원을 선점할 수 있다면 데드락 사이클을 끊을 수 있습니다.

### 4. 순환 대기 (Circular Wait)

프로세스들이 **원형 체인**을 형성하여 각 프로세스가 체인의 다음 프로세스가 보유한 자원을 기다립니다.

```
P1 → R2 (P2가 보유)
P2 → R3 (P3가 보유)
P3 → R1 (P1이 보유)
```

이 사이클은 나머지 세 조건이 모두 충족됨을 의미합니다.

## 자원 할당 그래프 (RAG)

**자원 할당 그래프**는 자원 할당 상태를 시각화합니다:

- **원**: 프로세스
- **사각형**: 자원 유형 (내부 점은 자원 인스턴스)
- **요청 간선** P → R: 프로세스 P가 자원 R을 기다리는 중
- **할당 간선** R → P: 자원 R의 인스턴스가 P에 할당됨

**RAG로 데드락 탐지:**
- 그래프에 **사이클이 없으면** → 데드락 없음
- 모든 자원 유형이 단일 인스턴스일 때 사이클 → 데드락
- 자원 유형에 여러 인스턴스가 있을 때 사이클은 필요조건이지 충분조건이 아님

---

## 데드락 처리 전략

네 가지 주요 전략이 있습니다:

1. **예방 (Prevention)** — 코프만 조건 중 하나가 절대 성립하지 않도록 보장
2. **회피 (Avoidance)** — 자원 요청을 동적으로 검사하여 안전한 경우에만 허용
3. **탐지 및 복구 (Detection and Recovery)** — 데드락 발생을 허용하고, 탐지 후 해결
4. **무시 (Ostrich Algorithm)** — 데드락이 없는 척 (드물고 예방 비용이 클 때 유효)

## 1. 데드락 예방

4가지 코프만 조건 중 하나를 제거하여 데드락을 예방합니다.

### 상호 배제 제거

자원을 공유 가능하게 만들 수 있는 경우에 적용합니다. 읽기 락(공유 락)은 동시 접근을 허용합니다. 스풀링으로 프린터를 가상화하면 프로세스들이 스풀 파일에 쓰게 되어 프린터 직접 경쟁을 피합니다.

진정으로 독점적인 자원(쓰기 락, 물리 장치)은 상호 배제가 본질적으로 필요합니다.

### 점유와 대기 제거

**방법 A: 모든 자원을 한 번에 요청.**
프로세스 시작 전 필요한 모든 자원을 요청합니다. 하나라도 없으면 아무것도 보유하지 않은 채 대기합니다.

문제: 미래 자원 필요를 미리 알기 어렵습니다. 대기 중 자원이 유휴 상태로 낭비됩니다.

**방법 B: 요청 전 해제.**
새 자원을 요청하기 전에 현재 보유한 모든 자원을 해제합니다.

### 비선점 제거

자원을 보유한 프로세스가 얻을 수 없는 자원을 요청하면, 현재 보유한 자원을 모두 선점합니다. 프로세스는 필요한 모든 자원을 얻을 수 있을 때 재시작합니다.

CPU 레지스터나 메모리처럼 상태를 저장/복원할 수 있는 자원에 잘 작동합니다. 안전하게 선점할 수 없는 자원(작업 중인 프린터, 데이터 업데이트 중인 락)에는 좋지 않습니다.

### 순환 대기 제거

**모든 자원 유형에 전체 순서를 부여합니다.** 프로세스는 이 번호 순서대로 자원을 요청해야 합니다. $R_i$를 보유한 프로세스는 $j > i$인 $R_j$만 요청할 수 있습니다.

```c
// 항상 순서대로 잠금: mutex_A 먼저, 그 다음 mutex_B
pthread_mutex_lock(&mutex_A);
pthread_mutex_lock(&mutex_B);
// ... 임계 구역 ...
pthread_mutex_unlock(&mutex_B);
pthread_mutex_unlock(&mutex_A);
```

모든 스레드가 항상 같은 순서로 뮤텍스를 획득하면 순환 대기는 불가능합니다. 가장 실용적인 예방 전략입니다.

## 2. 데드락 회피

각 자원 요청을 동적으로 검사하여 결과 상태가 **안전**한 경우에만 허용합니다.

### 안전 상태

**안전 상태**는 **안전 순서(safe sequence)**가 존재하는 상태입니다 — 모든 프로세스를 순서대로 나열했을 때 각 프로세스가 현재 가용 자원과 앞선 프로세스들이 해제할 자원으로 필요한 모든 자원을 획득할 수 있는 순서.

```
안전 상태 → 데드락 불가능
불안전 상태 → 데드락 가능 (반드시 발생하는 건 아님)
```

### 뱅커 알고리즘 (Dijkstra, 1965)

가장 유명한 데드락 회피 알고리즘입니다. 은행이 모든 고객의 대출 요청을 처리할 수 있도록 보장해야 한다는 개념에서 유래했습니다.

**자료구조** ($n$개 프로세스, $m$개 자원 유형):

| 구조 | 크기 | 의미 |
|---|---|---|
| `Available[m]` | 벡터 | 각 자원의 가용 인스턴스 수 |
| `Max[n][m]` | 행렬 | 각 프로세스의 최대 요구량 |
| `Allocation[n][m]` | 행렬 | 현재 할당된 자원 |
| `Need[n][m]` | 행렬 | 남은 필요량: `Need[i][j] = Max[i][j] - Allocation[i][j]` |

**안전성 알고리즘:**

```
Work = Available
Finish[i] = false (모든 i에 대해)

반복:
  Finish[i] == false이고 Need[i] <= Work인 i 찾기
  찾으면:
    Work = Work + Allocation[i]
    Finish[i] = true
  못 찾으면:
    종료

모든 Finish[i] == true → 안전 상태
그렇지 않으면 → 불안전 상태
```

**제한사항:**
- 최대 자원 요구량을 미리 알아야 함 (현실에서 어려움)
- 요청마다 $O(n^2 \times m)$ 복잡도
- 고정된 프로세스 수와 자원 수 가정
- 실시간 시스템에서는 사용되지만 범용 OS에서는 드물게 사용됨

## 3. 데드락 탐지 및 복구

데드락이 발생하도록 허용합니다. 주기적으로 탐지 알고리즘을 실행합니다. 데드락 발견 시 해결합니다.

### 탐지 알고리즘

뱅커 알고리즘의 안전성 검사와 유사하지만 Max 행렬 없이 수행합니다.

**단일 인스턴스 자원:** 자원 할당 그래프에서 사이클 탐지. DFS로 $O(n^2)$.

**다중 인스턴스 자원:**

```
Work = Available
Allocation[i] != 0이면 Finish[i] = false, 아니면 true

반복:
  Finish[i] == false이고 Request[i] <= Work인 i 찾기
  찾으면:
    Work = Work + Allocation[i]
    Finish[i] = true
  못 찾으면:
    종료

Finish[i] == false인 프로세스들이 데드락 상태
```

### 탐지 주기

- 모든 요청 실패 후: 즉시 탐지하지만 비용이 높음
- 주기적으로 (N초마다): 데드락이 지속될 수 있음
- CPU 사용률이 임계값 이하로 떨어질 때: 데드락은 종종 낮은 CPU 사용률을 유발함

### 데드락 복구

**방법 A: 프로세스 종료**

- 데드락된 모든 프로세스를 종료 (간단하지만 작업 손실)
- 한 번에 하나씩 종료하고 탐지를 재실행 (데드락이 해결될 때까지)

**희생 프로세스 선택 기준:**
- 가장 낮은 우선순위
- CPU 시간을 가장 적게 사용한 프로세스 (손실 최소화)
- 자원을 가장 적게 보유한 프로세스
- 대화형 vs 배치 (배치 프로세스 종료 선호)

**방법 B: 자원 선점**

일부 프로세스에서 자원을 선점하여 다른 프로세스에 제공합니다.

- **희생자 선택**: 비용 최소화 (사용된 CPU 시간, 보유 자원)
- **롤백**: 희생자 프로세스를 안전한 상태로 롤백 (체크포인팅 필요)
- **기아 방지**: 항상 같은 프로세스가 희생자가 되지 않도록 롤백 횟수 추적

### 체크포인트와 롤백

자원 선점 복구가 작동하려면 프로세스를 일관된 상태로 되돌리는 메커니즘이 필요합니다.

**체크포인팅**: 프로세스의 완전한 상태(레지스터, 메모리, 열린 파일)를 안정 저장소에 주기적으로 저장합니다. 롤백이 필요하면 가장 최근 체크포인트에서 복원합니다.

체크포인팅은 비용이 높습니다 — 프로세스의 전체 메모리 상태를 저장하는 데 시간과 디스크 공간이 필요합니다.

**데이터베이스 방식**: 많은 데이터베이스 시스템은 트랜잭션 롤백으로 데드락을 처리합니다. DBMS는 대기 그래프에서 데드락 사이클을 탐지하고 가장 비용이 적은 트랜잭션을 중단(롤백)합니다. 애플리케이션은 중단된 트랜잭션을 재시도합니다.

```sql
-- PostgreSQL은 데드락을 자동 탐지합니다:
-- ERROR: deadlock detected
-- DETAIL: Process 123 waits for ShareLock on transaction 456;
--         blocked by process 789.
```

## 자원 유형별 데드락

### 재사용 가능 자원

반복적으로 사용할 수 있는 자원: CPU, 메모리, I/O 장치, 파일, 세마포어. 대부분의 데드락 이론은 재사용 가능 자원에 초점을 맞춥니다.

### 소비 가능 자원

생성(생산)되고 파괴(소비)되는 자원: 시그널, 메시지, 인터럽트, 버퍼의 데이터.

```
P1이 P2의 메시지를 대기
P2가 P1의 메시지를 대기
→ 데드락 (둘 다 상대방이 필요로 하는 메시지를 생산할 수 없음)
```

이 유형의 데드락은 자원이 아직 존재하지 않기 때문에 탐지가 더 어렵습니다.

## 분산 시스템에서의 데드락

분산 시스템에서는 데드락 탐지가 더 어렵습니다:
- 단일 노드가 모든 자원 할당의 완전한 뷰를 갖지 못함
- 통신 지연으로 전역 상태가 완벽하게 일관되지 않음
- 오래된 정보로 인해 팬텀 데드락이 잘못 탐지될 수 있음

### 분산 대기 그래프

각 노드가 로컬 대기 그래프를 유지합니다. **코디네이터**가 주기적으로 로컬 그래프를 수집하여 **전역** 대기 그래프를 구성하고 사이클을 확인합니다.

### 에지 체이싱

중앙 집중식 탐지의 대안입니다. 프로세스가 블록되면 대기 그래프의 간선을 따라 **프로브** 메시지를 보냅니다. 프로브가 출발점으로 돌아오면 사이클(데드락)이 존재합니다.

```
P1 (P2에 의해 블록) 프로브 전송 → P2 (P3에 의해 블록) 전달 → P3 (P1에 의해 블록) → 사이클 탐지
```

## 4. 타조 알고리즘

문제를 그냥 무시합니다. 재부팅하거나 사용자가 멈춘 프로세스를 직접 종료합니다.

다음 경우에 사용합니다:
- 데드락이 극히 드문 경우
- 예방/탐지 비용이 가끔 발생하는 데드락보다 큰 경우

대부분의 데스크탑 OS(Windows, macOS, Linux)는 일부 자원 유형에 이 방식을 사용하며 사용자나 감시 프로세스가 멈춘 프로세스를 탐지하고 종료하는 데 의존합니다.

## 데드락 vs 라이브락 vs 기아

| 상태 | 설명 |
|---|---|
| **데드락** | 프로세스들이 멈춤 — 아무도 진행 불가 |
| **라이브락** | 프로세스들이 활발히 실행되지만 진행하지 못함 (서로에게 반응하며 앞으로 나아가지 못함) |
| **기아** | 프로세스가 필요한 자원을 얻지 못함 (다른 프로세스들이 계속 먼저 얻음) |

**기아**는 **에이징(aging)**으로 해결합니다 — 대기 중인 프로세스의 우선순위를 점진적으로 높여 결국 서비스됩니다.

## 실제 코드에서의 데드락 예방

**락 순서 지정**: 코드베이스 전체에서 항상 같은 순서로 락을 획득합니다.

```c
// 나쁜 예: 스레드 1은 A→B, 스레드 2는 B→A → 데드락 위험
// 좋은 예: 둘 다 항상 A→B 순서로 잠금
```

**락 타임아웃**: `pthread_mutex_timedlock()`이나 `std::try_lock()`에 타임아웃을 설정합니다. N밀리초 내에 락을 획득하지 못하면 물러서고 재시도합니다.

**std::lock (C++)**: 데드락 없는 알고리즘으로 여러 락을 동시에 획득합니다:

```cpp
std::lock(mutex_a, mutex_b);  // 데드락 없음
std::lock_guard<std::mutex> la(mutex_a, std::adopt_lock);
std::lock_guard<std::mutex> lb(mutex_b, std::adopt_lock);
```

**도구**: ThreadSanitizer(TSan)는 런타임에 잠재적 데드락과 경쟁 조건을 탐지합니다.

```bash
# ThreadSanitizer로 컴파일
g++ -fsanitize=thread -g -o program program.cpp
./program
```
