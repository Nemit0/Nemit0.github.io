# 프로세스 vs 스레드

## 프로세스란?

**프로세스**는 실행 중인 프로그램의 인스턴스입니다. 프로그램을 실행하면 OS는 메모리를 할당하고 실행 파일을 로드하여 런타임 환경을 구성하는 프로세스를 생성합니다.

프로세스는 자원 소유의 기본 단위입니다. 각 프로세스는 다음을 가집니다:

- 자체 **가상 주소 공간** (코드, 데이터, 힙, 스택 세그먼트)
- 자체 **파일 디스크립터** 및 오픈 핸들
- 자체 **환경 변수**
- 보안 자격 증명 (Unix의 UID, GID)
- 최소 하나의 **실행 스레드**

프로세스는 서로 **격리**되어 있습니다. 한 프로세스는 OS가 중재하는 IPC 없이는 다른 프로세스의 메모리에 직접 접근할 수 없습니다. 이 격리성이 프로세스를 안전하지만 비용이 높게 만듭니다.

### 프로세스 상태

프로세스는 생명주기 동안 다양한 상태를 거칩니다:

```
새로 생성 → 준비 → 실행 → 종료
              ↑↓
           대기 (I/O, 슬립, 신호 등 대기)
```

| 상태 | 설명 |
|---|---|
| **New (새로 생성)** | 프로세스가 생성되는 중 |
| **Ready (준비)** | 메모리에 있지만 CPU를 기다리는 중 |
| **Running (실행)** | CPU에서 현재 실행 중 |
| **Waiting (대기)** | I/O, 락, 신호 등을 기다리며 블록됨 |
| **Terminated (종료)** | 실행 완료, 자원 해제 중 |

단일 코어 CPU에서는 한 번에 하나의 프로세스만 **Running** 상태입니다. 나머지는 Ready(실행 가능) 또는 Waiting(아직 실행 불가) 상태입니다.

### PCB (Process Control Block)

OS는 **프로세스 제어 블록(PCB)**이라는 자료구조로 모든 프로세스를 추적합니다. PCB는 프로세스를 관리하고 재개하는 데 필요한 모든 정보를 담고 있습니다:

| PCB 필드 | 내용 |
|---|---|
| 프로세스 ID (PID) | 고유 식별자 |
| 프로세스 상태 | New, Ready, Running, Waiting, Terminated |
| 프로그램 카운터 (PC) | 다음 실행할 명령어 주소 |
| CPU 레지스터 | 범용 레지스터, 스택 포인터, 플래그 |
| 메모리 관리 정보 | 페이지 테이블, 세그먼트 테이블 |
| I/O 상태 | 열린 파일 목록, 대기 중인 I/O |
| 스케줄링 정보 | 우선순위, 버스트 시간, 큐 포인터 |
| 회계 정보 | 사용된 CPU 시간, 경과 시간 |
| 부모 PID | 이 프로세스를 생성한 프로세스의 PID |

CPU가 한 프로세스에서 다른 프로세스로 전환(컨텍스트 스위치)할 때, OS는 현재 PCB를 저장하고 다음 PCB를 로드합니다.

### 프로세스 생성

Unix/Linux에서는 `fork()`로 프로세스를 생성합니다:

```c
#include <unistd.h>
#include <stdio.h>

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        // 자식 프로세스
        printf("자식: PID = %d\n", getpid());
    } else if (pid > 0) {
        // 부모 프로세스
        printf("부모: PID = %d, 자식 PID = %d\n", getpid(), pid);
    }
    return 0;
}
```

`fork()`는 현재 프로세스의 **완전한 복사본**을 생성합니다. 자식은 부모의 주소 공간을 복제받습니다 (실제로는 Copy-on-Write). 두 프로세스는 반환값을 기준으로 분기합니다.

### Copy-on-Write (CoW)

`fork()`는 실제로 주소 공간 전체를 복사하지 않습니다. 부모와 자식은 처음에 같은 물리 페이지를 **공유**하며, 읽기 전용으로 표시됩니다. 어느 쪽이든 페이지에 **쓰기**를 시도하면 CPU가 페이지 폴트를 발생시키고, OS가 그 페이지만 복사하여 쓰기 프로세스에 개인 사본을 제공합니다.

```
쓰기 전:
  부모 페이지 테이블 → [공유 페이지 A] ← 자식 페이지 테이블  (읽기 전용)

자식 쓰기 후:
  부모 페이지 테이블 → [공유 페이지 A]  (원본, 여전히 읽기 전용)
  자식 페이지 테이블 → [페이지 A 사본]  (이제 쓰기 가능)
```

이것이 성능에 결정적입니다. `fork()` 후 `exec()`는 페이지 테이블만 복제하면 됩니다 — `exec()`가 주소 공간을 완전히 대체하기 때문에 실제 데이터는 복사할 필요가 없습니다.

### 프로세스 종료

프로세스가 종료되면 OS는 프로세스의 자원(메모리, 열린 파일, 락)을 회수합니다. 그러나 PCB 항목은 부모가 `wait()`를 호출하여 종료 상태를 수집할 때까지 남아있습니다.

### 좀비 프로세스

자식 프로세스가 종료되었지만 부모가 아직 `wait()`를 호출하지 않으면 자식은 **좀비**(defunct 프로세스)가 됩니다. 메모리나 CPU를 점유하지 않지만 종료 상태를 보관하기 위해 프로세스 테이블에 PCB 항목이 남습니다.

```bash
# 좀비는 ps 출력에서 'Z'로 표시됨
ps aux | grep 'Z'
```

좀비 자체는 무해하지만 수천 개가 쌓이면 프로세스 테이블(PID 공간)을 소진할 수 있습니다. 해결책은 부모가 `wait()` 또는 `waitpid()`를 적절히 호출하는 것입니다:

```c
int status;
pid_t child = wait(&status);      // 자식이 종료할 때까지 블록
waitpid(pid, &status, 0);         // 특정 자식 대기
waitpid(-1, &status, WNOHANG);    // 논블로킹으로 아무 자식 대기
```

### 고아 프로세스

부모가 자식보다 먼저 종료하면 자식은 **고아**가 됩니다. OS는 고아 프로세스를 `init` (PID 1) 또는 서브리퍼 프로세스로 **재입양**합니다. 새 부모가 고아가 종료될 때 `wait()`를 호출하여 영구적인 좀비가 되는 것을 방지합니다.

### 프로세스 계층과 그룹

Unix 프로세스는 `init` (PID 1)에 뿌리를 둔 **트리**를 형성합니다. 모든 프로세스에는 부모가 있습니다 (`init` 제외).

```bash
pstree -p
# systemd(1)─┬─sshd(800)───sshd(1200)───bash(1201)───vim(1500)
#             ├─cron(500)
#             └─nginx(900)─┬─nginx(901)
#                          └─nginx(902)
```

프로세스는 **프로세스 그룹**과 **세션**으로 조직됩니다:

- **프로세스 그룹**: 관련 프로세스의 집합 (예: 파이프라인 `ls | grep | sort`의 모든 프로세스). 그룹 전체에 시그널을 보낼 수 있습니다.
- **세션**: 프로세스 그룹의 집합으로 보통 터미널 로그인에 연결됩니다. **세션 리더**(보통 셸)와 최대 하나의 **전경 프로세스 그룹**을 가집니다.

이 계층은 셸의 **작업 제어**의 기반입니다 — `Ctrl+C`는 전경 프로세스 그룹에 `SIGINT`를, `Ctrl+Z`는 `SIGTSTP`를 보내며, `bg`와 `fg`는 배경/전경 그룹을 관리합니다.

### 시그널

**시그널**은 프로세스에 전달되는 소프트웨어 인터럽트입니다. Unix에서 가장 기본적인 IPC 형태입니다.

| 시그널 | 기본 동작 | 설명 |
|---|---|---|
| `SIGINT` (2) | 종료 | 키보드 인터럽트 (Ctrl+C) |
| `SIGTERM` (15) | 종료 | 정중한 종료 요청 |
| `SIGKILL` (9) | 종료 | **잡거나 무시할 수 없음** |
| `SIGSEGV` (11) | 코어 덤프 | 세그멘테이션 폴트 |
| `SIGSTOP` (19) | 정지 | **잡을 수 없음** — 프로세스 일시 중지 |
| `SIGCONT` (18) | 계속 | 정지된 프로세스 재개 |
| `SIGCHLD` (17) | 무시 | 자식 프로세스 종료 또는 정지 |

프로세스는 기본 동작을 재정의하기 위해 **시그널 핸들러**를 설치할 수 있습니다:

```c
#include <signal.h>

void handler(int signum) {
    // 시그널 전달 시 비동기적으로 호출됨
    // 여기서는 async-signal-safe 함수만 사용 (write, _exit 등)
    // printf, malloc 등은 사용 불가
    write(STDOUT_FILENO, "시그널 감지\n", 17);
}

int main() {
    struct sigaction sa;
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;
    sigaction(SIGINT, &sa, NULL);   // Ctrl+C 핸들러 설치
}
```

시그널은 **비동기적**입니다 — 실행 중 어느 시점에서든 도착할 수 있어 올바른 시그널 핸들러 작성이 까다롭습니다.

### 프로세스 간 통신 (IPC)

프로세스가 격리되어 있으므로 통신하려면 OS가 중재하는 메커니즘이 필요합니다:

| IPC 메커니즘 | 유형 | 설명 | 속도 |
|---|---|---|---|
| **파이프** | 바이트 스트림 | 단방향, 부모-자식 간 | 빠름 |
| **명명된 파이프 (FIFO)** | 바이트 스트림 | 관계 없는 프로세스 간 | 빠름 |
| **메시지 큐** | 메시지 | 커널이 관리하는 메시지 버퍼 | 보통 |
| **공유 메모리** | 메모리 영역 | 가장 빠른 IPC — 직접 메모리 접근 | 가장 빠름 |
| **소켓** | 바이트 스트림 / 데이터그램 | 네트워크 또는 로컬 (Unix 도메인) | 보통 |
| **시그널** | 알림 | 비동기 알림, 데이터 페이로드 없음 | 빠름 |

**파이프** — 가장 단순한 IPC. 한쪽 끝에 쓰면 다른 쪽에서 읽을 수 있습니다:

```c
int fd[2];
pipe(fd);       // fd[0] = 읽기 끝, fd[1] = 쓰기 끝

if (fork() == 0) {
    close(fd[1]);                    // 자식은 쓰기 끝 닫기
    read(fd[0], buf, sizeof(buf));   // 부모로부터 읽기
} else {
    close(fd[0]);                    // 부모는 읽기 끝 닫기
    write(fd[1], "hello", 5);       // 자식에게 쓰기
}
```

**공유 메모리** — 커널을 통해 데이터를 복사하지 않아 가장 빠른 IPC:

```c
int fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 4096);
void *ptr = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
// 두 프로세스가 *ptr을 직접 읽고 쓸 수 있음
// 경쟁 조건 방지를 위해 동기화(세마포어/뮤텍스) 필요
```

### 스레딩 모델: 1:1, M:1, M:N

유저 레벨 스레드와 커널 레벨 스레드 간의 매핑이 스레딩 모델을 정의합니다:

**1:1 (일대일)**: 각 유저 스레드가 하나의 커널 스레드에 매핑됩니다. Linux (NPTL), Windows, 현대 macOS에서 사용. 멀티코어에서 진정한 병렬성. 단점: 커널 스레드 생성 오버헤드로 수백만 스레드까지 확장이 어렵습니다.

**M:1 (다대일)**: 모든 유저 스레드가 하나의 커널 스레드에 매핑됩니다. 유저 공간 스케줄링이 빠르지만 병렬성 없음 — 블로킹 시스템 콜이 모든 스레드를 블록합니다.

**M:N (다대다)**: M개의 유저 스레드가 N개의 커널 스레드에 매핑됩니다 (N ≤ M). 진정한 병렬성과 가벼운 유저 공간 스케줄링의 장점을 결합합니다. Go의 고루틴 스케줄러(work-stealing 포함)가 사용합니다.

```
유저 스레드:   T1  T2  T3  T4  T5  T6
                \  |  /    \  |  /
커널 스레드:     K1          K2
```

Go의 런타임은 고루틴(유저 스레드)을 더 작은 OS 스레드 풀에 다중화합니다. 고루틴이 시스템 콜에서 블록되면 런타임은 다른 고루틴을 다른 OS 스레드로 이동시킵니다.

## 스레드란?

**스레드**는 OS가 스케줄링하는 가장 작은 실행 단위입니다. 스레드는 프로세스 *내부*에 존재하며, 프로세스의 자원을 공유하지만 각자의 실행 컨텍스트를 가집니다.

모든 프로세스는 하나의 스레드(메인 스레드)로 시작합니다. 동일한 프로세스 내에서 추가 스레드를 생성할 수 있습니다.

같은 프로세스의 스레드들이 **공유**하는 것:
- **가상 주소 공간** (코드, 힙, 전역 데이터)
- **파일 디스크립터**
- **신호 핸들러**
- **작업 디렉토리**

각 스레드가 **독자적으로** 가지는 것:
- **스택** (지역 변수, 함수 호출 프레임)
- **프로그램 카운터**
- **CPU 레지스터**
- **스레드 ID (TID)**
- **신호 마스크**

### 스레드 생성 (POSIX)

```c
#include <pthread.h>
#include <stdio.h>

void *worker(void *arg) {
    printf("스레드 실행 중, arg = %d\n", *(int *)arg);
    return NULL;
}

int main() {
    pthread_t tid;
    int value = 42;

    pthread_create(&tid, NULL, worker, &value);
    pthread_join(tid, NULL);  // 스레드 완료 대기

    return 0;
}
```

### 유저 레벨 스레드 vs 커널 레벨 스레드

| | 유저 레벨 스레드 | 커널 레벨 스레드 |
|---|---|---|
| **관리 주체** | 유저 공간 라이브러리 | OS 커널 |
| **컨텍스트 스위치 비용** | 낮음 (커널 모드 전환 없음) | 높음 (시스템 콜 오버헤드) |
| **블로킹 I/O** | 전체 프로세스 블록 | 호출한 스레드만 블록 |
| **진정한 병렬성** | 없음 (단일 커널 스레드) | 있음 (여러 코어 활용) |

## 프로세스 vs 스레드: 비교표

| 항목 | 프로세스 | 스레드 |
|---|---|---|
| **주소 공간** | 별도 (격리) | 공유 (프로세스 내) |
| **생성 비용** | 높음 (전체 복제) | 낮음 (최소한의 오버헤드) |
| **컨텍스트 스위치 비용** | 높음 (TLB 플러시, 페이지 테이블 교체) | 낮음 (동일 주소 공간) |
| **통신** | IPC (파이프, 소켓, 공유 메모리, 신호) | 직접 메모리 접근 (동기화 필요) |
| **결함 격리** | 한 프로세스 충돌이 다른 프로세스에 영향 없음 | 스레드 하나 충돌이 전체 프로세스 종료 가능 |
| **메모리** | 각자 힙과 스택 | 힙 공유, 스택은 별도 |
| **스케줄링 단위** | 스레드 (프로세스는 컨테이너) | 스레드 |
| **생성 시스템 콜 (Linux)** | `fork()` | `clone()` (공유 플래그 포함) |

> Linux에서 `fork()`와 `pthread_create()` 모두 결국 `clone()`을 호출합니다. 차이는 *어떤 자원*을 공유하느냐입니다. 프로세스는 기본적으로 아무것도 공유하지 않고, 스레드는 거의 모든 것을 공유합니다.

---

## 언제 스레드를, 언제 프로세스를?

**스레드를 쓸 때**: 단일 작업 내에서 성능을 극대화하고 싶을 때. 웹 서버가 스레드 풀로 요청을 동시 처리하거나, 비디오 인코더가 프레임을 여러 코어에 분산할 때.

**프로세스를 쓸 때**: 격리가 필요할 때. 브라우저는 탭마다 별도 프로세스를 사용하여 탭 하나가 충돌해도 전체 브라우저가 다운되지 않도록 합니다.

**스레드의 위험**: 공유 메모리는 경쟁 조건(race condition)을 유발합니다. 버그는 스레드 스케줄링이 비결정적이기 때문에 재현하기 어렵습니다.

### 스레드 풀 패턴

새 작업마다 스레드를 생성하는 대신(비용이 높음), 미리 생성된 스레드 풀을 유지하여 큐에서 작업을 가져갑니다:

```
[메인 스레드] → [작업 큐] → [스레드 1]
                           → [스레드 2]
                           → [스레드 3]
```

스레드 생성 비용을 분산시키고 자원 사용량을 제한합니다.

## Linux 구현 세부사항

Linux에서 프로세스와 스레드는 모두 **태스크**(`struct task_struct`)로 표현됩니다. 커널은 이 둘을 진정으로 구분하지 않습니다. 차이는 `clone()` 플래그를 통해 어떤 자원을 공유하느냐입니다:

| `clone()` 플래그 | 공유 자원 |
|---|---|
| `CLONE_VM` | 가상 주소 공간 (힙, 전역 변수) |
| `CLONE_FILES` | 파일 디스크립터 테이블 |
| `CLONE_SIGHAND` | 신호 핸들러 |
| `CLONE_THREAD` | 스레드 그룹 (동일 PID) |

`pthread_create()`는 `CLONE_VM | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD` 등을 전달합니다.
`fork()`는 이 플래그들을 전달하지 않아 자식이 모든 것을 복사받습니다.

```bash
# 프로세스의 스레드 목록 확인
ls /proc/<pid>/task/

# 프로세스 상태 확인
cat /proc/<pid>/status
```

---

## 정리

- **프로세스**는 자체 주소 공간과 자원을 가진 실행 중인 프로그램입니다. 생성 비용이 높고 강하게 격리됩니다.
- **스레드**는 프로세스 내부의 실행 단위입니다. 가볍고, 메모리를 공유하며, 동기화가 필요합니다.
- OS는 **스레드**를 스케줄링합니다. 프로세스는 자원 컨테이너입니다.
- 단일 작업의 성능을 위해서는 스레드를, 작업 간 격리를 위해서는 프로세스를 사용합니다.
- 스레드의 공유 메모리라는 유연성의 대가는 동기화의 올바른 구현 부담입니다.
