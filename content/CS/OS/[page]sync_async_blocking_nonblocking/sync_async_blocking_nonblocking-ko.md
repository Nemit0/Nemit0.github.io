# 동기 vs 비동기 / 블로킹 vs 논블로킹

이 두 개념은 자주 혼동되지만 **직교**합니다 — 어떤 조합도 가능합니다. 상호작용을 보여주기 전에 각각을 정확하게 정의합니다.

---

## 동기 vs 비동기

이 차원은 **작업이 완료되는 시점에 대해 호출자가 언제 제어권을 돌려받는가**에 관한 것입니다.

### 동기 (Synchronous)

호출자는 작업이 **완료될 때까지 대기한 후 계속 진행**합니다. 연산을 호출한 후 결과가 준비된 후에만 다음 코드 줄이 실행됩니다.

```c
// 동기 파일 읽기
char buf[1024];
ssize_t n = read(fd, buf, sizeof(buf));
// n이 결과. read()가 반환된 후에만 이 줄이 실행됨.
printf("읽은 바이트 수: %zd\n", n);
```

제어 흐름이 선형입니다: 호출 → 대기 → 반환 → 다음 명령어.

### 비동기 (Asynchronous)

호출자는 **작업을 시작하고 즉시 계속 진행**합니다. 결과는 나중에 콜백, 이벤트, 신호, 또는 future/promise를 통해 전달됩니다.

```javascript
// 비동기 파일 읽기 (Node.js)
fs.readFile('/etc/hosts', (err, data) => {
    // 나중에, 데이터가 준비된 후 실행됨
    console.log('읽은 바이트 수:', data.length);
});
console.log('이 줄은 즉시 실행됨, 파일 읽기 전에');
```

제어 흐름이 분기됩니다: 호출 → 계속 → ... → 완료 시 콜백 호출.

## 블로킹 vs 논블로킹

이 차원은 **자원을 기다리는 동안 스레드가 무엇을 하는가**에 관한 것입니다.

### 블로킹 (Blocking)

블로킹 호출은 작업이 완료될 때까지 **호출 스레드를 중단**합니다. 스레드는 OS에 의해 스케줄에서 제외(슬립)되고 결과가 준비되면 깨어납니다. 대기 중에 CPU를 사용하지 않지만 다른 작업을 할 수 없습니다.

```c
// 블로킹 read: 데이터가 준비될 때까지 스레드가 슬립
ssize_t n = read(fd, buf, sizeof(buf));
// I/O 대기 중에 스레드가 슬립 상태였음
```

### 논블로킹 (Non-Blocking)

논블로킹 호출은 작업 성공 여부에 관계없이 **즉시 반환**합니다. 결과가 아직 준비되지 않았다면 대기하는 대신 오류 코드(보통 `EAGAIN` 또는 `EWOULDBLOCK`)를 반환합니다.

```c
// fd를 논블로킹 모드로 설정
int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

// 논블로킹 read
ssize_t n = read(fd, buf, sizeof(buf));
if (n == -1 && errno == EAGAIN) {
    // 데이터가 아직 준비되지 않음 — 다른 작업 후 나중에 재시도
}
```

스레드는 절대 슬립하지 않습니다. 즉시 데이터를 얻거나 나중에 다시 시도하라는 신호를 받습니다.

## 2×2 매트릭스

두 차원이 결합되어 네 가지 모드가 됩니다:

|  | **동기** | **비동기** |
|---|---|---|
| **블로킹** | 스레드 대기, 결과 반환 | (드물게 의미 있음 — 모순적) |
| **논블로킹** | 즉시 반환 (성공 또는 오류), 호출자가 폴링 | 즉시 반환, 콜백/이벤트로 결과 전달 |

### 동기 + 블로킹 (가장 일반적)

C에서의 일상적인 `read()`, `write()`, `connect()`. 스레드가 함수를 호출하고 슬립하다가 완료 시 깨어나며 결과가 직접 반환됩니다.

단순한 프로그래밍 모델. 스레드당 한 번에 하나의 연산. 확장성이 좋지 않음 — 동시 요청 10,000개에 10,000개의 스레드 필요.

### 동기 + 논블로킹 (폴링)

즉시 반환합니다. 결과가 준비되지 않으면 호출자가 루프에서 재시도합니다.

```c
fcntl(fd, F_SETFL, O_NONBLOCK);

while (1) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n > 0) {
        // 데이터 획득
        break;
    } else if (errno == EAGAIN) {
        // 아직 준비 안 됨, 잠시 다른 작업
        usleep(1000);
        continue;
    }
}
```

### 비동기 + 블로킹 (드묾)

비동기 연산을 시작하지만 스레드가 여전히 완료를 기다립니다 — 비동기의 목적을 무색하게 합니다. 동기 코드를 비동기로 점진적으로 변환할 때 과도기적 패턴으로 나타납니다.

### 비동기 + 논블로킹 (고성능 I/O)

즉시 반환하고 호출자는 대기하지 않습니다. 결과는 비동기적으로 전달됩니다. 다음에서 사용되는 모델입니다:
- **epoll** / **kqueue** / **IOCP**
- **io_uring** (Linux 5.1+)
- **Node.js** 이벤트 루프
- **Python asyncio**
- **Rust async/await**

```c
// epoll 기반 서버 (단순화)
int epfd = epoll_create1(0);
struct epoll_event ev = { .events = EPOLLIN, .data.fd = sockfd };
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);

struct epoll_event events[64];
while (1) {
    int n = epoll_wait(epfd, events, 64, -1);  // 어떤 fd가 준비될 때까지 블록
    for (int i = 0; i < n; i++) {
        // 준비된 각 fd를 논블로킹 read로 처리
        handle_event(events[i].fd);
    }
}
```

단일 스레드로 수천 개의 연결을 관리할 수 있습니다.

## 상세 I/O 모델

POSIX는 다섯 가지 I/O 모델을 정의합니다:

### 1. 블로킹 I/O

기본 모드. 단순합니다. 동시 연산당 하나의 스레드.

```
애플리케이션              커널
   |── read(fd) ─────────>|
   |   (스레드 슬립)       | [데이터 대기]
   |                       | [데이터 도착]
   |<── 데이터 반환 ───────|
   | 다음 명령어            |
```

### 2. 논블로킹 I/O (폴링)

```
애플리케이션              커널
   |── read(fd) ─────────>|
   |<── EAGAIN ───────────|  (데이터 미준비)
   | [다른 작업]           |
   |── read(fd) ─────────>|
   |<── 데이터 반환 ───────|  (데이터 준비됨)
```

슬립 없음, 하지만 자주 폴링하면 CPU 낭비.

### 3. I/O 멀티플렉싱 (select / poll / epoll)

```
애플리케이션              커널
   |── epoll_wait({fd..})->|
   |   (스레드 슬립)       | [여러 fd 감시]
   |                       | [fd1 읽기 가능]
   |<── fd1 준비 반환 ─────|
   | 논블로킹 read(fd1)    |
```

하나의 스레드가 많은 파일 디스크립터를 감시합니다. 적어도 하나가 준비될 때까지 슬립합니다.

- `select()`: O(n) 스캔, 최대 1024 fd 제한. 레거시.
- `poll()`: O(n) 스캔, 제한 없음. select보다 약간 좋음.
- `epoll()` (Linux): O(1) 알림, 커널이 감시 목록 유지. 수백만 fd로 확장 가능.
- `kqueue()` (BSD/macOS): epoll과 유사.

### 엣지 트리거 vs 레벨 트리거

epoll과 kqueue에 적용되는 구분입니다:

**레벨 트리거** (기본): `epoll_wait()`는 조건이 참인 동안 fd를 준비 상태로 반환합니다. 가용 데이터를 모두 읽지 않으면 다음 호출에서 다시 보고됩니다. 관대함 — 이벤트를 놓치기 어렵습니다.

**엣지 트리거** (`EPOLLET`): `epoll_wait()`는 상태가 **변경**될 때만 fd를 반환합니다 (예: 새 데이터 도착). 가용 데이터를 모두 읽지 않으면 더 많은 데이터가 도착할 때까지 다시 알림을 받지 않습니다. 각 알림에서 전체 버퍼를 비워야 합니다.

```c
// 엣지 트리거: EAGAIN까지 읽어야 함, 아니면 데이터가 조용히 손실됨
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &(struct epoll_event){.events = EPOLLIN | EPOLLET});

// epoll_wait 반환 시:
while (1) {
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n == -1 && errno == EAGAIN) break;  // 버퍼 완전히 비워짐
    process(buf, n);
}
```

| | 레벨 트리거 | 엣지 트리거 |
|---|---|---|
| 알림 | 조건이 유지되는 동안 | 상태 변경 시에만 |
| 부분 읽기 | OK (재알림) | 반드시 비워야 함 (아니면 이벤트 손실) |
| 프로그래밍 | 더 단순 | 주의 깊게 처리 필요 |
| 성능 | 시스템 콜 더 많음 | 시스템 콜 더 적음 |

고성능 서버 (nginx, Redis)는 엣지 트리거 모드를 사용합니다.

### 4. 비동기 I/O (POSIX AIO / io_uring)

```
애플리케이션              커널
   |── aio_read(fd, buf) ->|  (즉시 반환)
   | [다른 작업]           | [I/O 진행 중]
   |                       | [I/O 완료]
   |<── SIGIO/콜백 ────────|
   | 결과 처리             |
```

커널이 백그라운드에서 모든 I/O를 수행합니다. 완료 시 애플리케이션에 알림. 애플리케이션 스레드는 I/O를 기다리며 슬립하지 않습니다.

**io_uring** (Linux 5.1+)은 현대적인 고성능 AIO 인터페이스입니다 — 공유 메모리의 제출/완료 링, 제로 복사, 배치 시스템 콜.

## 실용적 비교

### 연결당 스레드 (동기 + 블로킹)

```
클라이언트 1 ──→ 스레드 1 (블로킹 read/write)
클라이언트 2 ──→ 스레드 2 (블로킹 read/write)
클라이언트 3 ──→ 스레드 3 (블로킹 read/write)
```

**장점**: 단순한 코드. 자연스러운 순차적 흐름.
**단점**: 클라이언트 10,000개 = 스레드 10,000개. 각 스레드 스택 기본 8MB. 10K 스레드 = 스택만 80GB RAM. 컨텍스트 스위치 오버헤드 폭발.

### 이벤트 루프 (비동기 + 논블로킹)

```
클라이언트 1 ──→ [이벤트 루프] ──→ fd 1 처리 (논블로킹)
클라이언트 2 ──→ [이벤트 루프] ──→ fd 2 처리 (논블로킹)
...
모든 클라이언트가 단일 스레드에서 처리됨
```

**장점**: 단일 스레드로 수백만 연결 처리. 연결 간 컨텍스트 스위치 오버헤드 없음.
**단점**: 단일 스레드 — 느린 연산 하나가 모든 것을 블록. CPU 집중 작업은 오프로드 필요. 콜백 지옥 / 복잡한 제어 흐름.

## 언어 수준 비동기 모델

### 콜백 (JavaScript, C)

```javascript
readFile('a.txt', (err, a) => {
    readFile('b.txt', (err, b) => {
        writeFile('c.txt', a + b, (err) => {
            // 콜백 지옥
        });
    });
});
```

### 프로미스 / 퓨처

```javascript
readFile('a.txt')
    .then(a => readFile('b.txt').then(b => [a, b]))
    .then(([a, b]) => writeFile('c.txt', a + b))
    .catch(err => console.error(err));
```

### Async/Await (프로미스/퓨처에 대한 문법적 설탕)

```python
async def process():
    a = await read_file('a.txt')  # 논블로킹 — 이벤트 루프에 양보
    b = await read_file('b.txt')  # 논블로킹 — 이벤트 루프에 양보
    await write_file('c.txt', a + b)
```

동기처럼 보이지만 논블로킹입니다. `await`는 대기 중에 이벤트 루프에 제어권을 넘깁니다 — 다른 코루틴이 실행될 수 있습니다.

```rust
// Rust async/await — 제로 비용 추상화
async fn process() -> Result<(), Error> {
    let a = read_file("a.txt").await?;
    let b = read_file("b.txt").await?;
    write_file("c.txt", &(a + &b)).await?;
    Ok(())
}
```

## 정리: 언제 무엇을 사용할까

| 시나리오 | 권장 모델 |
|---|---|
| 단순 스크립트, 낮은 동시성 | 동기 + 블로킹 (단순 스레드) |
| 고동시성 네트워크 서버 | 비동기 + 논블로킹 (이벤트 루프, io_uring) |
| I/O + CPU 혼합 작업 | 비동기 I/O + CPU용 스레드 풀 |
| 실시간, 지연 민감 | 논블로킹 + 세심한 타이밍 |
| 하드웨어 폴링 (임베디드) | 논블로킹 폴링 루프 |
| Linux 파일 I/O (전통) | 동기 블로킹 (비동기 파일 I/O는 복잡) |
| Linux 파일 I/O (현대, 고성능) | io_uring |

핵심 통찰: **블로킹 vs 논블로킹**은 OS/시스템 콜 인터페이스의 속성입니다. **동기 vs 비동기**는 그 위에 쌓인 프로그래밍 모델의 속성입니다. 블로킹 I/O 위에 비동기 프로그래밍 모델을 구축할 수 있고(스레드 전용), 논블로킹 I/O를 동기적으로 사용할 수도 있습니다(폴링).

고성능 시스템은 이 두 관심사를 분리합니다: OS 수준에서 논블로킹 I/O를 사용하고, 그 위에 깔끔한 비동기 프로그래밍 모델(코루틴, async/await, 퓨처)을 구축하여 코드 가독성을 유지합니다.

## 리액터와 프로액터 패턴

이벤트 기반 I/O 시스템 구축을 위한 두 가지 주요 아키텍처 패턴입니다.

### 리액터 패턴

애플리케이션이 I/O 이벤트(읽기 가능, 쓰기 가능)에 대한 관심을 등록하고 콜백을 제공합니다. 리액터(이벤트 루프)가 이벤트를 대기하고 이벤트 발생 시 콜백을 디스패치합니다. 애플리케이션이 콜백 내에서 실제 I/O(논블로킹 읽기/쓰기)를 수행합니다.

```
[이벤트 루프 / 리액터]
    |
    ├── epoll_wait() → fd1 읽기 가능 → on_read(fd1) 호출
    |                                  → 애플리케이션이 read(fd1) 수행
    └── ...
```

리액터가 이벤트를 **역다중화**하고, 애플리케이션이 **처리**합니다. Node.js (libuv), Python (asyncio), Java NIO, nginx, Redis에서 사용합니다.

### 프로액터 패턴

애플리케이션이 비동기 I/O 연산을 시작하고 완료 콜백을 제공합니다. 프로액터(OS 또는 런타임)가 백그라운드에서 I/O를 수행합니다. I/O 완료 시 결과가 이미 준비된 상태로 완료 콜백을 호출합니다.

```
[애플리케이션]
    |── async_read(fd1, buf, callback) → [OS가 백그라운드에서 I/O]
    |
[OS가 I/O 완료]
    └── callback(fd1, bytes_read) 호출 → 데이터가 이미 buf에 있음
```

애플리케이션이 `read()`나 `write()`를 직접 호출하지 않습니다. Windows IOCP, Linux io_uring에서 사용합니다.

| | 리액터 | 프로액터 |
|---|---|---|
| I/O 수행 주체 | 애플리케이션 (콜백 내) | OS (백그라운드) |
| 이벤트 알림 | "fd가 I/O 준비됨" | "I/O가 완료됨" |
| OS 지원 | epoll, kqueue, poll | IOCP, io_uring |

### io_uring: 프로액터

io_uring은 진정한 프로액터입니다: 애플리케이션이 **제출 큐 (SQ)**에 I/O 요청을 제출하고, 커널이 완료 후 **완료 큐 (CQ)**에 결과를 게시합니다. 두 큐 모두 공유 메모리에 있어 시스템 콜 없이 제출과 완료가 가능합니다.

## 코루틴과 구조적 동시성

### 스택풀 vs 스택리스 코루틴

**스택풀 코루틴** (파이버, 그린 스레드): 각 코루틴이 자체 스택을 가집니다. 호출 스택의 어느 깊이에서든 양보할 수 있습니다. 코루틴당 더 많은 메모리 (~4-64 KB 스택). 예: Go 고루틴, Lua 코루틴, Java 가상 스레드 (Project Loom).

**스택리스 코루틴**: 코루틴이 상태 머신입니다. `await` 지점에서만 양보 가능합니다. 별도 스택 불필요 — 지역 변수를 보관하는 작은 구조체만. 더 메모리 효율적 (~수백 바이트). 예: Rust async/await, C++20 코루틴, Python async/await.

| | 스택풀 | 스택리스 |
|---|---|---|
| 코루틴당 메모리 | ~4-64 KB | ~100-500 바이트 |
| 양보 위치 | 호출 스택 어디서든 | `await` 지점에서만 |
| 최대 동시성 | ~100K-1M | ~1M-100M |
| 예시 | Go, Java Loom, Lua | Rust, C++20, Python asyncio |

### 구조적 동시성

전통적인 동시성(스레드/태스크를 임의로 생성)은 fire-and-forget 패턴으로 이어져 태스크가 부모 범위를 벗어나 살아남고, 오류 처리와 취소가 어려워집니다.

**구조적 동시성**은 동시 태스크가 부모에 범위가 지정되도록 보장합니다 — 부모 태스크는 모든 자식 태스크가 완료될 때까지 완료될 수 없습니다. 자식이 실패하면 부모와 형제에게 알림됩니다.

```python
# Python: TaskGroup을 사용한 구조적 동시성 (Python 3.11+)
async with asyncio.TaskGroup() as tg:
    task1 = tg.create_task(fetch("url1"))
    task2 = tg.create_task(fetch("url2"))
# 여기서 모든 태스크가 완료 보장
# 어떤 태스크가 실패하면 나머지가 자동 취소됨
```

장점:
- 누수 태스크 없음 (모든 태스크에 소유자가 있음)
- 자동 취소 전파
- 오류 처리가 호출 스택을 따름
- 결정적 자원 정리

## 백프레셔

비동기 시스템에서 생산자가 소비자의 처리 속도보다 빠르게 작업을 생성할 수 있습니다. **백프레셔** 없이는 버퍼가 무한히 커집니다 → 메모리 고갈.

백프레셔 메커니즘:
- **유한 채널/큐**: 큐가 가득 차면 생산자가 블록 (또는 오류 수신)
- **TCP의 흐름 제어**: TCP의 슬라이딩 윈도우가 수신자가 따라잡지 못할 때 자동으로 송신자를 느리게 함
- **리액티브 스트림**: 소비자가 생산자에게 처리할 수 있는 항목 수를 알림 (`request(n)`)

```
백프레셔 없이:
  생산자 (빠름) → [버퍼가 영원히 커짐] → 소비자 (느림) → OOM

백프레셔 있음:
  생산자 → [유한 버퍼, 크기 N] → 소비자
  버퍼 가득? → 생산자 블록/감속 → 시스템 안정
```

프로덕션 시스템에서 이것은 매우 중요합니다. 빠른 데이터베이스 쿼리가 느린 네트워크 연결에 결과를 쏟아부을 때, 또는 빠른 생산자가 느린 소비자에게 채널을 통해 공급할 때 — 백프레셔 없이는 중간 버퍼가 모든 가용 메모리를 소비합니다.
