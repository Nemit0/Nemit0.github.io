# 컨텍스트 스위칭

## 컨텍스트 스위치란?

**컨텍스트 스위치**는 CPU가 한 프로세스(또는 스레드)의 실행을 멈추고 다른 프로세스를 시작하는 메커니즘입니다. OS는 현재 실행 상태를 저장하여 나중에 재개할 수 있도록 하고, 다음 프로세스의 저장된 상태를 로드합니다.

멀티태스킹이 가능한 이유가 바로 이것입니다. 단일 코어 CPU에서도 수십 개의 프로세스가 동시에 실행되는 것처럼 보이는 건, CPU가 초당 수천 번 프로세스를 전환하기 때문입니다.

---

## "컨텍스트"란 무엇인가?

프로세스의 **컨텍스트**는 CPU가 중단된 지점에서 실행을 재개하는 데 필요한 모든 정보입니다:

- **프로그램 카운터 (PC)** — 다음 명령어 주소
- **CPU 레지스터** — 범용 레지스터, 스택 포인터, 플래그 레지스터
- **메모리 관리 레지스터** — 페이지 테이블 기저 레지스터 (x86의 CR3)
- **커널 스택** — 프로세스의 커널 모드 스택
- **부동소수점 / SIMD 상태** — FPU, SSE, AVX 레지스터

이 모든 것이 프로세스의 **PCB(Process Control Block)**에 저장되고, 프로세스가 재개될 때 로드됩니다.

---

## 컨텍스트 스위치가 발생하는 시점

컨텍스트 스위치는 특정 이벤트에 의해 트리거됩니다:

### 1. 타이머 인터럽트 (선점형 스케줄링)
OS는 하드웨어 타이머를 N밀리초마다 인터럽트가 발생하도록 설정합니다. 인터럽트가 발생하면 CPU가 커널 모드로 전환되고 스케줄러가 전환 여부를 결정합니다. 가장 일반적인 트리거입니다.

### 2. I/O 요청 (자발적 양보)
프로세스가 `read()`, `write()` 등을 호출하지만 데이터가 준비되지 않은 경우. OS는 프로세스를 **Waiting** 상태로 전환하고 Ready 상태의 프로세스로 전환합니다.

### 3. 시스템 콜
프로세스가 블로킹되는 시스템 콜을 호출합니다 (예: `sleep()`, `wait()`, 잠긴 뮤텍스에서의 `pthread_mutex_lock()`).

### 4. 높은 우선순위 프로세스가 준비됨
I/O 작업 완료로 고우선순위 프로세스가 Ready 상태가 되면 선점형 시스템에서 즉시 전환될 수 있습니다.

### 5. 명시적 양보
프로세스가 `sched_yield()` (Linux) 또는 `SwitchToThread()` (Windows)를 호출하여 자발적으로 CPU를 반납합니다.

## 컨텍스트 스위치 절차

프로세스 A에서 프로세스 B로 전환할 때 OS가 수행하는 단계:

### 1단계: 프로세스 A의 컨텍스트 저장

CPU가 커널 모드에 있습니다. OS 커널은:

1. 인터럽트를 잠시 비활성화합니다
2. 모든 CPU 레지스터를 A의 PCB에 저장합니다
3. 프로그램 카운터를 저장합니다
4. 메모리 관리 상태 (페이지 테이블 포인터)를 저장합니다
5. A가 FPU/SIMD를 사용했다면 그 상태도 저장합니다

```
A의 PCB:
  PC  = 0x4005a2   ← A가 실행 중이던 위치
  RAX = 0x0000001c
  RSP = 0x7ffe9120
  CR3 = 0x1a3000   ← A의 페이지 테이블 기저주소
  ...
```

### 2단계: 프로세스 A의 상태 업데이트

- A의 상태를 **Running** → **Ready** (선점된 경우) 또는 **Waiting** (블록된 경우)로 변경
- A를 적절한 큐에 다시 넣습니다

### 3단계: 다음 프로세스 선택 (스케줄링)

**스케줄러**가 준비 큐에서 다음 실행할 프로세스를 선택합니다. 선택 알고리즘은 스케줄링 정책에 따라 다릅니다 (FCFS, Round Robin, Priority 등).

### 4단계: 프로세스 B의 컨텍스트 로드

1. B의 페이지 테이블 포인터(CR3) 로드 → TLB 플러시 발생
2. PCB에서 B의 CPU 레지스터 로드
3. B가 FPU/SIMD를 사용한다면 그 상태 로드
4. B의 프로그램 카운터 로드
5. B의 상태를 **Ready** → **Running**으로 변경

### 5단계: 프로세스 B 재개

커널이 `iret` (x86) 또는 `eret` (ARM) 같은 명령어를 실행하여 B의 컨텍스트에서 반환합니다. CPU는 B가 중단된 지점에서 실행을 재개합니다.

## 컨텍스트 스위치의 비용

컨텍스트 스위치는 **무료가 아닙니다**. 여러 층위의 비용이 있습니다:

### 직접 비용: 상태 저장/복원

레지스터 저장, PCB 업데이트, 스택 조작. 현대 CPU에서 수십~수백 나노초.

FPU/SIMD 상태 (AVX-512: 2048비트 = 256바이트)는 저장/복원 비용이 높습니다. OS는 **지연 FPU 저장**을 사용합니다 — 다른 프로세스가 실제로 FPU를 사용할 때만 저장합니다.

### 간접 비용: TLB 플러시

OS가 새 페이지 테이블(CR3)을 로드하면 CPU의 **TLB(Translation Lookaside Buffer)**를 플러시해야 합니다. TLB는 가상-물리 주소 변환을 캐시합니다. 플러시 후에는 TLB가 다시 워밍업될 때까지 모든 메모리 접근에서 TLB 미스가 발생합니다.

이것이 종종 프로세스 전환의 **주된 비용**입니다. TLB 미스 하나당 약 100ns의 메모리 지연이 추가됩니다.

**스레드 전환은 더 저렴합니다** — 같은 프로세스의 스레드들은 동일한 페이지 테이블을 공유하므로 TLB 플러시가 필요 없습니다.

### 간접 비용: 캐시 오염

CPU 캐시(L1, L2, L3)는 실행 중인 프로세스의 명령어와 데이터를 보관합니다. B로 전환하면 A의 캐시된 데이터는 그대로 남아 있지만, B의 명령어와 데이터를 로드하면서 A의 항목이 축출됩니다. 이 **캐시 오염**으로 새 프로세스에서 캐시 미스가 증가합니다.

### 대략적인 비용 (현대 x86)

| 작업 | 대략적 비용 |
|---|---|
| 커널 모드 진입/퇴출 (시스템 콜) | ~100–300 ns |
| 스레드 전환 (같은 프로세스) | ~1–5 μs |
| 프로세스 전환 (다른 주소 공간) | ~5–30 μs |
| 전환 후 TLB 워밍업 | 수백 μs (워크로드 의존) |

## 모드 스위치 vs 컨텍스트 스위치

흔한 혼동: **모드 스위치**는 **컨텍스트 스위치**와 다릅니다.

**모드 스위치** (권한 레벨 변경): 유저 모드와 커널 모드 간의 전환. 모든 시스템 콜과 인터럽트에서 발생합니다. CPU가 권한 레벨을 변경하고 커널 스택으로 전환하지만, 같은 프로세스가 여전히 실행 중입니다.

**컨텍스트 스위치**: OS가 실행 중인 태스크 전체를 교체합니다. 모드 스위치 + 모든 상태 저장/복원 + 잠재적으로 주소 공간 교체를 포함합니다.

```
시스템 콜: 유저 모드 → 커널 모드 → 유저 모드   (같은 프로세스)
                      [모드 스위치만]

선점:     유저 모드 → 커널 모드 → [A 저장, B 로드] → 유저 모드
                     [모드 스위치]  [컨텍스트 스위치]
```

모든 컨텍스트 스위치는 모드 스위치를 포함하지만, 모든 모드 스위치가 컨텍스트 스위치는 아닙니다. 모드 스위치는 저렴하고 (~100-300 ns), 컨텍스트 스위치는 비쌉니다 (~1-30 μs).

## 인터럽트 처리와 컨텍스트 스위칭

인터럽트는 컨텍스트 스위치가 트리거되는 방식의 핵심입니다.

### 인터럽트 생명주기

1. **장치가 인터럽트 발생** (하드웨어 IRQ) 또는 CPU가 폴트/트랩 감지
2. CPU가 현재 명령어를 마치고, 최소 상태(PC, 플래그)를 **커널 스택**에 저장
3. CPU가 **인터럽트 디스크립터 테이블 (IDT)**에서 핸들러 주소 조회
4. CPU가 **ring 0** (커널 모드)로 전환하고 핸들러로 점프
5. 인터럽트 핸들러 실행
6. 인터럽트 반환 전, 커널이 **재스케줄**이 필요한지 확인 (Linux의 `need_resched` 플래그)
7. 재스케줄 필요 시 → **컨텍스트 스위치** 발생
8. 인터럽트 반환 (`iret`)으로 (다른 프로세스일 수 있는) 프로세스에서 실행 재개

### 상반부 vs 하반부 (Linux)

인터럽트 지연 시간을 최소화하기 위해 Linux는 인터럽트 처리를 분할합니다:

- **상반부** (hardirq): 인터럽트 컨텍스트에서 즉시 실행. 최소한의 필수 작업만 — 인터럽트 확인, 장치에서 데이터 읽기, 지연 작업 스케줄. 슬립 불가.
- **하반부** (softirq / tasklet / workqueue): 나중에 인터럽트 활성화 상태에서 실행. 대부분의 처리 수행.

## 멀티코어와 NUMA 고려사항

멀티코어 시스템에서 컨텍스트 스위칭은 캐시 토폴로지 및 메모리 아키텍처와 상호작용합니다.

### 코어별 스케줄링

각 CPU 코어는 자체 **실행 큐**를 가집니다. 컨텍스트 스위치는 각 코어에서 독립적으로 발생합니다.

### 캐시 계층 효과

현대 CPU는 코어별 전용 L1/L2 캐시와 공유 L3 캐시를 가집니다:

```
코어 0: [L1d] [L1i] [L2]  ←  전용
코어 1: [L1d] [L1i] [L2]  ←  전용
         └───────┬───────┘
              [L3 공유]
              [DRAM]
```

프로세스가 **같은 코어**에서 재개되면 L1/L2 데이터가 여전히 웜할 수 있습니다. **다른 코어**에서 재개되면 L1/L2가 완전히 콜드입니다 — 모든 데이터를 L3 또는 DRAM에서 가져와야 합니다.

이것이 **CPU 어피니티**가 중요한 이유입니다.

### NUMA (비균등 메모리 접근)

멀티 소켓 서버에서 각 CPU 소켓은 자체 로컬 메모리를 가집니다. 원격 메모리 접근(다른 소켓에 연결된)은 약 2배 느립니다:

```
소켓 0: [코어0..코어7] ←→ [로컬 RAM 0]    (빠름: ~80ns)
               ↕ QPI/UPI 인터커넥트         (느림: ~150ns)
소켓 1: [코어8..코어15] ←→ [로컬 RAM 1]   (빠름)
```

프로세스를 소켓 0에서 소켓 1로 이동하는 컨텍스트 스위치는 치명적입니다: 모든 캐시된 메모리 참조가 원격이 됩니다. Linux의 NUMA 인식 스케줄러는 프로세스를 원래 NUMA 노드에 유지하려 합니다.

```bash
# NUMA 토폴로지 확인
numactl --hardware

# 프로세스를 NUMA 노드 0에 고정
numactl --membind=0 --cpunodebind=0 ./program
```

## 유휴 프로세스

실행 가능한 프로세스가 CPU 코어에 없을 때, 커널은 **유휴 프로세스**를 실행합니다. 유휴 프로세스는 **바쁜 대기를 하지 않습니다**. 하드웨어 절전 명령어를 사용합니다:

| 아키텍처 | 유휴 명령어 | 효과 |
|---|---|---|
| x86 | `hlt` | 다음 인터럽트까지 CPU 중지 |
| x86 | `mwait` | 메모리 주소 모니터, 쓰기까지 중지 |
| ARM | `wfi` (Wait For Interrupt) | 인터럽트까지 CPU 중지 |

현대 CPU는 점진적으로 절전 수준이 높아지고 깨어나는 지연도 증가하는 여러 **C 상태**(전력 상태)를 가집니다:

| C 상태 | 이름 | 절전 | 깨어남 지연 |
|---|---|---|---|
| C0 | Active | 없음 (실행 중) | 0 |
| C1 | Halt | 낮음 | ~1 μs |
| C3 | Sleep | 높음 (L1/L2 플러시) | ~50 μs |
| C6 | Deep Sleep | 매우 높음 (전압 감소) | ~100 μs |

더 깊은 C 상태는 더 많은 전력을 절약하지만 다음 컨텍스트 스위치의 지연을 증가시킵니다. 지연에 민감한 애플리케이션(실시간, 고빈도 거래)의 경우 깊은 C 상태를 비활성화할 수 있습니다.

## 자발적 vs 비자발적 컨텍스트 스위치

Linux는 프로세스별로 두 종류를 추적합니다:

```bash
cat /proc/<pid>/status | grep ctxt
# voluntary_ctxt_switches:    1024
# nonvoluntary_ctxt_switches:  57
```

- **자발적**: 프로세스가 CPU를 반납 (I/O, 락, 슬립에서 블록됨)
- **비자발적**: 스케줄러가 프로세스를 선점 (타이머 만료, 고우선순위 프로세스 준비 완료)

시스템 전체 컨텍스트 스위치 모니터링:

```bash
vmstat 1
# in  cs 항목 참조 (초당 인터럽트, 컨텍스트 스위치)
```

## 컨텍스트 스위치 오버헤드 최소화

실제로 사용되는 전략들:

**스레드 풀**: 고정된 수의 스레드를 유지하고 재사용합니다. 스레드 생성/소멸 오버헤드를 제거합니다.

**CPU 어피니티**: `sched_setaffinity()` (Linux)로 스레드/프로세스를 특정 CPU 코어에 고정합니다. 스레드의 데이터를 같은 코어의 캐시에 유지하여 캐시 오염을 줄입니다.

**협력적 스케줄링 / 코루틴**: 유저 공간에서의 태스크 전환 (Go의 goroutine, Rust의 async/await, Python의 asyncio). 커널에 진입하지 않고 전환하므로 약 10배 저렴합니다. 단점: 블로킹 시스템 콜은 전체 OS 스레드를 블록합니다.

**대용량 페이지 (Huge Pages)**: TLB 압력 감소. 2MB 대용량 페이지는 4KB 페이지보다 512배 더 많은 범위를 커버하므로 전환 후 TLB 미스가 줄어듭니다.

**io_uring (Linux)**: 단일 시스템 콜로 많은 I/O 작업을 제출하여 커널 모드 전환 빈도를 줄입니다.

## 풀이 예제: 두 프로세스 간 단계별 문맥 전환

프로세스 A (PID 1001)는 피보나치 수를 계산하는 반복문을 실행 중입니다. 프로세스 B (PID 1002)는 문자열 파싱 루틴을 실행하다 선점된 후 준비 큐에서 대기 중입니다. 명령어 주소 `0x4005a2`에서 타이머 인터럽트가 발생합니다.

정확히 무슨 일이 일어나는지 단계별로 살펴봅니다:

**1단계 — 타이머 인터럽트 발생**

하드웨어 APIC 타이머가 IRQ를 발생시킵니다. CPU는 현재 명령어 경계가 끝나는 시점에 이를 감지합니다. 프로세스 A의 유저 공간 코드 실행이 루프 도중 일시 정지됩니다.

**2단계 — CPU가 커널 모드로 전환**

CPU가 자동으로:
- 권한 레벨을 **Ring 3** (유저) → **Ring 0** (커널)으로 변경
- 스택 포인터를 프로세스 A의 **커널 스택**으로 전환 (TSS의 RSP0 필드에서 로드)
- 최소 인터럽트 프레임을 커널 스택에 푸시: `SS`, `RSP`, `RFLAGS`, `CS`, `RIP`

프로세스 A의 유저 공간 실행이 동결됩니다. CPU는 커널 모드에서 타이머 인터럽트 핸들러를 실행 중입니다.

**3단계 — 프로세스 A의 전체 레지스터 상태를 커널 스택에 저장**

인터럽트 핸들러(및 스케줄러의 `save_context()` 루틴)가 나머지 모든 범용 레지스터를 A의 커널 스택에 푸시합니다:

```
A의 커널 스택 (상단부터):
  RIP    = 0x4005a2    ← A가 인터럽트된 정확한 위치
  CS     = 0x0033      ← 유저 모드 코드 세그먼트
  RFLAGS = 0x00000246  ← carry=0, zero=1, interrupt-enable=1
  RSP    = 0x7ffe9120  ← A의 유저 공간 스택 포인터
  SS     = 0x002b      ← 유저 모드 스택 세그먼트
  RAX    = 0x00000262  ← 현재 피보나치 값 fib(15)=610 (0x262)
  RBX    = 0x0000000f  ← 루프 카운터 (n=15)
  RCX    = 0x00000001
  RDX    = 0x00000000
  RSI    = 0x7ffe9100
  RDI    = 0x0000000f
  RBP    = 0x7ffe9110
  R8–R15 = ...
  CR3    = 0x1a3000    ← A의 페이지 테이블 기저 주소 (별도 저장)
```

**4단계 — 저장된 상태를 프로세스 A의 PCB에 복사**

커널은 저장된 레지스터 값을 A의 커널 스택에서 메모리의 A의 **PCB(Process Control Block)**로 복사합니다:

```
PID 1001 (프로세스 A)의 PCB:
  cpu_state.rip    = 0x4005a2
  cpu_state.rsp    = 0x7ffe9120
  cpu_state.rax    = 0x2329
  cpu_state.rbx    = 0x0000000f
  cpu_state.rflags = 0x00000246
  cpu_state.cr3    = 0x1a3000
  fpu_state        = [필요 시 지연 저장]
  kernel_stack_ptr = 0xffff8800_12340000
```

PCB는 이제 A의 실행 상태에 대한 완전한 스냅샷입니다.

**5단계 — 프로세스 A의 상태 변경: Running → Ready**

스케줄러가 A의 태스크 디스크립터의 상태 필드를 업데이트합니다:

```
A.state = TASK_RUNNING  →  TASK_READY
```

프로세스 A는 CPU의 활성 슬롯에서 제거되고 준비 큐에 다시 배치됩니다 (블록된 것이 아니라 선점된 것이므로 Waiting이 아닌 Ready 상태가 됩니다).

**6단계 — 스케줄러가 실행되어 프로세스 B를 선택**

OS 스케줄러(예: Linux CFS)가 준비 큐를 검사합니다. 프로세스 B (PID 1002)는 가상 실행 시간 우선순위가 가장 높습니다 — 가장 오래 대기했습니다. 스케줄러가 B를 선택합니다.

```
스케줄러 결정:
  next = pick_next_task(rq)  →  PID 1002 (프로세스 B)
```

**7단계 — 프로세스 B의 PCB 상태 로드**

커널이 프로세스 B의 PCB를 읽습니다:

```
PID 1002 (프로세스 B)의 PCB:
  cpu_state.rip    = 0x401234   ← B가 마지막으로 인터럽트된 위치
  cpu_state.rsp    = 0x7ffd4080 ← B의 유저 공간 스택
  cpu_state.rax    = 0x00000041 ← ASCII 'A', 문자열 파싱 중
  cpu_state.rbx    = 0x00000003 ← 문자열 버퍼의 인덱스
  cpu_state.rflags = 0x00000202
  cpu_state.cr3    = 0x2b5000   ← B의 페이지 테이블 기저 주소
```

**8단계 — 프로세스 B의 페이지 테이블 포인터(CR3) 쓰기 → TLB 플러시 발생**

커널이 B의 CR3 레지스터를 로드합니다:

```c
mov cr3, 0x2b5000   // B의 페이지 디렉터리 기저 주소 로드
```

이 단일 명령어가 **전체 TLB를 플러시**합니다(PCID/ASID를 사용하지 않는 경우). 프로세스 A를 위해 캐시된 모든 가상-물리 주소 매핑이 즉시 무효화됩니다. B가 다음에 가상 주소에 접근할 때마다 TLB가 다시 워밍업될 때까지 CPU는 메모리에서 페이지 테이블을 탐색해야 합니다.

**9단계 — 프로세스 B의 CPU 레지스터 로드**

커널이 B의 커널 스택에서 저장된 레지스터 값을 물리적 CPU 레지스터에 팝합니다:

```
레지스터 로드 후 CPU:
  RAX    ← 0x00000041
  RBX    ← 0x00000003
  RCX, RDX, RSI, RDI, RBP, R8–R15 ← B의 저장된 상태에서
```

**10단계 — 프로세스 B의 상태 변경: Ready → Running**

```
B.state = TASK_READY  →  TASK_RUNNING
```

B는 이제 이 CPU 코어의 활성 태스크로 표시됩니다.

**11단계 — B의 컨텍스트에서 인터럽트 복귀**

커널이 `iret` (x86 인터럽트 복귀)를 실행합니다. 이 명령어는 원자적으로:
- B의 커널 스택에서 `RIP`, `CS`, `RFLAGS`, `RSP`, `SS`를 팝
- 권한을 Ring 0 → Ring 3으로 강등
- 스택 포인터를 B의 유저 공간 RSP (`0x7ffd4080`)로 전환

CPU는 이제 프로세스 B의 주소 공간에서 유저 모드로 실행 중입니다.

**12단계 — 프로세스 B가 마지막 PC에서 재개**

CPU가 `0x401234` 주소에서 명령어를 가져옵니다 — 프로세스 B가 인터럽트된 정확한 위치입니다. B의 문자열 파싱 루프가 아무 일도 없었던 것처럼 계속됩니다. B는 자신이 일시 정지되었다는 사실을 전혀 알지 못합니다.

### 커널 `context_switch()` 의사 코드

```c
// 단순화된 커널 context_switch(prev, next)
void context_switch(struct task *prev, struct task *next) {
    // 1. prev의 FPU/SIMD 상태 저장 (지연: prev가 FPU를 사용한 경우만)
    if (prev->used_fpu)
        fxsave(&prev->pcb.fpu_state);

    // 2. 페이지 테이블 전환 (주소 공간)
    //    TLB를 플러시함 (PCID 최적화를 사용하지 않는 경우)
    write_cr3(next->pcb.cr3);

    // 3. TSS에서 커널 스택 포인터 전환
    //    (다음 인터럽트/시스템 콜이 `next`의 커널 스택을 사용하도록)
    tss.rsp0 = next->kernel_stack_top;

    // 4. prev의 범용 레지스터 저장, next의 레지스터 로드
    //    (아키텍처별 어셈블리)
    switch_to(prev, next);
    //   ↑ 내부적으로:
    //     모든 GPR을 prev의 커널 스택에 푸시
    //     prev의 RSP를 prev->pcb.kernel_rsp에 저장
    //     next->pcb.kernel_rsp를 RSP에 로드
    //     모든 GPR을 next의 커널 스택에서 팝
    //     ret (next의 커널 스택에 저장된 RIP로 점프)

    // 5. next의 FPU 상태 복원 (지연: next가 FPU를 사용하는 경우만)
    if (next->used_fpu)
        fxrstor(&next->pcb.fpu_state);

    // 6. 현재 태스크 포인터 업데이트
    current = next;

    // 실행이 이제 `next`의 컨텍스트에서 계속됩니다.
    // `prev`는 미래의 context_switch()가 다시 선택할 때까지
    // 위의 switch_to 호출 지점에서 동결된 상태입니다.
}
```

`switch_to()`가 반환되면 CPU는 실제로 프로세스 B의 컨텍스트에서 실행 중입니다. 프로세스 A가 나중에 다시 스케줄되면, `switch_to()`에서 해당 함수가 정상적으로 반환된 것처럼 재개됩니다.

## 성능 비용: 구체적인 계산

문맥 전환 오버헤드는 무시할 수 있는 수준이라고 치부되는 경우가 많지만, 규모가 커지면 전체 CPU 시간의 측정 가능한 비율이 됩니다.

**기본 가정:**
- 각 문맥 전환 비용 **5 μs** (직접 비용: 레지스터 저장/복원 ≈ 1 μs + TLB 워밍업 및 캐시 미스 페널티 ≈ 4 μs)
- 이는 적당한 워킹셋을 가진 현대 x86 서버에서 프로세스 전환의 현실적인 평균값입니다

**시나리오 1 — 초당 1,000회 전환 (가벼운 부하):**

```
비용  = 5 μs × 1,000 = 5,000 μs = 5 ms/초
오버헤드 = 5 ms / 1,000 ms = 0.5%의 CPU 시간
```

매 CPU-초의 0.5%가 유용한 작업 없이 — 단순한 부기 작업에 소비됩니다.

**시나리오 2 — 초당 10,000회 전환 (보통 부하):**

```
비용  = 5 μs × 10,000 = 50,000 μs = 50 ms/초
오버헤드 = 50 ms / 1,000 ms = 5%의 CPU 시간
```

CPU 코어의 5%가 이제 문맥 전환만을 위해 사용됩니다.

**시나리오 3 — 초당 100,000회 전환 (고부하 I/O 서버):**

```
비용  = 5 μs × 100,000 = 500,000 μs = 500 ms/초
오버헤드 = 500 ms / 1,000 ms = 50%의 CPU 시간
```

이 수준에서는 시스템이 애플리케이션 코드를 실행하는 것보다 전환하는 데 더 많은 시간을 씁니다. 이것이 "문맥 전환 스래싱" 상태입니다 — 전환 횟수를 줄이거나 프로세스 대신 스레드를 사용해야 합니다.

### 문맥 전환 횟수 vs 오버헤드 비교표

| 초당 전환 횟수 | 초당 손실 시간 | CPU 오버헤드 |
|---|---|---|
| 100 | 0.5 ms | 0.05% |
| 1,000 | 5 ms | 0.5% |
| 5,000 | 25 ms | 2.5% |
| 10,000 | 50 ms | 5% |
| 50,000 | 250 ms | 25% |
| 100,000 | 500 ms | 50% |
| 200,000 | 1,000 ms | 100% (CPU 포화) |

*전환당 5 μs 가정. 스레드 전환(1–2 μs)은 동일한 횟수에서 ~3–5배 낮은 오버헤드를 보입니다.*

**실용적 시사점:**
- 요청당 새 프로세스를 생성하는 웹 서버가 초당 10,000 요청을 처리한다면, 문맥 전환만으로 5%의 오버헤드가 발생
- 스레드 풀을 사용하면 (스레드 전환 ≈ 1 μs) 이를 ~1%로 줄임
- 코루틴을 사용한 비동기 I/O (유저 공간 전환 ≈ 0.1 μs)는 ~0.1%로 줄임

Linux에서 현재 문맥 전환 횟수를 확인할 수 있습니다:

```bash
# 시스템 전체 초당 전환 횟수 (cs 열)
vmstat 1

# 프로세스별 합계
cat /proc/<pid>/status | grep ctxt_switches
```

## 프로세스 vs 스레드 문맥 전환 비교

프로세스 전환과 스레드 전환의 핵심 차이는 단 하나의 레지스터에서 비롯됩니다: **CR3** (페이지 테이블 기저 주소). 이 레지스터가 변경되는지 여부에서 모든 결과가 파생됩니다.

### 프로세스 전환이 비싼 이유

프로세스 A에서 프로세스 B로 전환할 때:

1. **CR3가 변경되어야 함** — A와 B는 서로 다른 가상 주소 공간을 가지므로, 페이지 테이블 기저 레지스터가 B의 페이지 디렉터리를 가리키도록 업데이트되어야 합니다.
2. **TLB가 플러시됨** — 새 CR3를 로드하면 (PCID 없이) 모든 TLB 항목이 무효화됩니다. TLB는 ~1,000–4,000개의 캐시된 주소 변환을 보유하는데 이것들이 모두 무효가 됩니다.
3. **TLB 콜드 스타트 페널티** — B의 워킹셋이 TLB를 다시 채울 때까지 모든 메모리 접근에서 하드웨어 페이지 테이블 탐색이 발생합니다 (각각 ~100–200 사이클).
4. **L1/L2 캐시가 콜드** — 프로세스 A의 명령어 및 데이터 캐시 라인은 B에 대해 모두 유효하지 않습니다. B의 코드와 데이터는 L3 또는 DRAM에서 가져와야 합니다.
5. **분기 예측기 상태가 무효화됨** — CPU의 분기 예측기는 A의 코드 패턴에 맞게 훈련되어 있어서, B의 패턴을 학습할 때까지 B의 코드에서 잘못된 예측을 하게 됩니다.

### 스레드 전환이 더 저렴한 이유

**같은 프로세스** 내의 스레드 T1과 T2 간 전환 시:

1. **CR3가 변경되지 않음** — 두 스레드는 동일한 가상 주소 공간과 페이지 테이블을 공유합니다. CR3는 그대로 유지됩니다.
2. **TLB 플러시 없음** — 페이지 테이블이 변경되지 않으므로 캐시된 모든 TLB 항목이 유효하게 유지됩니다.
3. **L1/L2 캐시가 여전히 워밍 상태일 수 있음** — T1과 T2가 서로 다른 데이터에서 작업하지만 공유 캐시 내에 데이터가 들어간다면, T2의 데이터가 이미 L1/L2에 있을 수 있습니다.
4. **저장/복원할 상태가 더 적음** — 메모리 관리 상태 변경 없이 GPR과 스택 포인터만 전환하면 됩니다.

### 저장/로드되는 항목: 프로세스 vs 스레드

| 상태 구성요소 | 프로세스 전환 | 스레드 전환 (같은 프로세스) |
|---|---|---|
| 범용 레지스터 (RAX–R15) | ✅ 저장/복원 | ✅ 저장/복원 |
| 프로그램 카운터 (RIP) | ✅ 저장/복원 | ✅ 저장/복원 |
| 스택 포인터 (RSP) | ✅ 저장/복원 | ✅ 저장/복원 |
| 플래그 레지스터 (RFLAGS) | ✅ 저장/복원 | ✅ 저장/복원 |
| FPU / SIMD 상태 (xsave) | ✅ 저장/복원 (지연) | ✅ 저장/복원 (지연) |
| 페이지 테이블 기저 (CR3) | ✅ **변경됨 → TLB 플러시** | ❌ **변경 없음 — 플러시 없음** |
| TLB 항목 | ❌ 모두 무효화 | ✅ 유효 유지 |
| L1/L2 명령어 캐시 | ❌ 새 프로세스를 위해 콜드 | ✅ 워밍 상태 유지 가능 (공유 코드) |
| L1/L2 데이터 캐시 | ❌ 새 프로세스를 위해 콜드 | ✅ 워밍 상태 유지 가능 (공유 힙) |
| 커널 스택 포인터 (TSS RSP0) | ✅ 업데이트 | ✅ 업데이트 |
| 메모리 매핑 파일, 공유 라이브러리 | ❌ 재매핑 (새 CR3) | ✅ 이미 매핑됨 (같은 CR3) |

### 구체적인 비용 비교

| 항목 | 스레드 전환 | 프로세스 전환 |
|---|---|---|
| CR3 쓰기 + TLB 플러시 | 없음 | ~200–500 사이클 |
| TLB 워밍업 (첫 N회 접근) | 없음 (TLB 여전히 유효) | 미스당 ~100–200 ns × N 회 |
| L1/L2 캐시 워밍업 | 부분적 (공유 데이터 유지 가능) | 완전히 콜드 |
| 총 일반적 지연 시간 | **~1–5 μs** | **~5–30 μs** |
| 비율 | 1× (기준) | 5–30× 느림 |

프로세스 전환에 걸리는 ~5–30 μs는 단일 작업에 소비되는 것이 아닙니다 — 전환 후 처음 수백 번의 메모리 접근에 걸쳐 발생하는 TLB 미스와 캐시 미스의 상각 비용입니다. 전환 자체(레지스터 저장/복원)는 ~1 μs만 소요되며 나머지 비용은 이후 수 마이크로초에 걸쳐 지연 지불됩니다.

### 실용적 시사점

이것이 고성능 서버(Nginx, Redis, Go 런타임)가 연결당 새 프로세스를 포크하는 대신 **스레드** 또는 **코루틴**을 사용하는 이유입니다. 연결당 프로세스 방식으로 초당 10,000개 연결을 처리하는 서버는 문맥 전환만으로 ~5–30 μs × 10,000 = **50–300 ms/초** (CPU의 5–30%)를 소비합니다. 스레드 풀을 사용하면 동일한 워크로드의 비용이 **1–5 μs × 10,000 = 10–50 ms/초** (~1–5%)로 줄어듭니다.
