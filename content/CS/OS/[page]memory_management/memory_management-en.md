# Memory Management

## Why Memory Management?

Multiple processes share a single physical RAM. Memory management solves three problems:

1. **Protection** — one process shouldn't read/write another's memory
2. **Relocation** — a process should run regardless of where in RAM it's loaded
3. **Efficient utilization** — RAM is limited; pack in as many processes as possible

The OS provides each process an **abstraction**: a private, contiguous address space starting at 0. Underneath, the physical layout can be fragmented, partially on disk, or shared with other processes.

---

## Address Types

- **Logical (Virtual) Address**: address generated by the CPU / used in the program. What your code sees.
- **Physical Address**: actual address in RAM hardware.

The **Memory Management Unit (MMU)** is hardware inside the CPU that translates logical → physical addresses at runtime.

---

## Early Approaches (Before Paging)

### Contiguous Allocation

The OS loads each process into a contiguous block of physical memory. Base and Limit registers define the boundaries:

```
Physical Memory:
[OS] [Process A] [Process B] [hole] [Process C] [hole]
```

**External fragmentation**: free memory exists but is split into small non-contiguous chunks too small for new processes.

**Compaction**: move all processes together to create one large free block. Expensive (must pause processes and rewrite all addresses).

---

## Paging

**Paging** eliminates external fragmentation by dividing both physical memory and logical address space into fixed-size blocks:

- Physical memory → **frames** (physical page frames)
- Logical address space → **pages**
- Pages and frames are the same size (typically 4 KB, sometimes 2 MB or 1 GB for huge pages)

A process's pages can be scattered across any free frames — they don't need to be contiguous.

### Address Translation

A logical address is split into two fields:

```
| Page Number (p) | Page Offset (d) |
```

For a 32-bit address with 4 KB pages:
- Page offset: 12 bits (2^12 = 4096 = 4 KB)
- Page number: 20 bits (2^20 = 1M pages per process)

The **page table** maps page numbers → frame numbers:

```
Logical: page 3, offset 100
Page table[3] = frame 7
Physical: frame 7, offset 100 → address = 7 * 4096 + 100
```

Each process has its own page table, stored in memory. The CPU's **page table base register** (PTBR / CR3 on x86) points to the current process's page table.

### Page Table Entry (PTE)

Each entry stores more than just the frame number:

| Bit | Name | Meaning |
|---|---|---|
| Frame number | — | Physical frame address |
| P | Present/Valid | Is this page currently in RAM? |
| R/W | Read/Write | Is write access allowed? |
| U/S | User/Supervisor | Accessible in user mode? |
| A | Accessed | Set by CPU on any read or write |
| D | Dirty | Set by CPU on write (page has been modified) |
| NX | No-Execute | Instructions cannot be fetched from this page |

The P bit is central to virtual memory — if 0, the page is not in RAM (it's on disk), and accessing it triggers a **page fault**.

### Internal Fragmentation

Paging has no external fragmentation but has **internal fragmentation**: if a process needs only 100 bytes but a full 4 KB page is allocated, 3996 bytes are wasted. On average, half a page per allocation is wasted.

---

## Translation Lookaside Buffer (TLB)

Naive paging requires **two memory accesses** per instruction: one to read the page table, one to access the actual data. This doubles memory latency.

The **TLB** is a fast hardware cache inside the MMU that stores recent page-number-to-frame-number translations:

```
CPU generates logical address
    ↓
TLB lookup (page number as key)
    ↓
TLB hit (>95% of the time) → get frame number in ~1 cycle
    ↓
TLB miss → walk page table in memory (hundreds of cycles) → store in TLB
```

**TLB reach** = TLB entries × page size. If TLB reach < process working set, TLB thrashing occurs.

On a context switch (different process = different page table), the TLB must be **flushed** (all entries invalidated) unless the CPU supports ASIDs/PCIDs (see Context Switching post).

### TLB Performance

- Modern TLBs: 32–2048 entries, organized in L1/L2 levels
- Hit rate: typically 95–99%
- Effective Access Time (EAT):

$$\text{EAT} = \alpha \cdot (T_{TLB} + T_{mem}) + (1 - \alpha) \cdot (T_{TLB} + 2 \cdot T_{mem})$$

Where $\alpha$ = TLB hit rate, $T_{TLB}$ ≈ 1 ns, $T_{mem}$ ≈ 100 ns.

At 99% hit rate: EAT ≈ 0.99 × 101 + 0.01 × 201 ≈ 103 ns (vs 200 ns without TLB).

## Multi-Level Page Tables

A 32-bit process with 4 KB pages needs a page table with $2^{20}$ = 1M entries. At 4 bytes each, that's **4 MB per process** just for the page table — even if most of the address space is unused.

**Multi-level page tables** use a tree structure: only allocate page table memory for portions of the address space that are actually used.

### Two-Level (x86-32):

```
Virtual address: | PD index (10) | PT index (10) | offset (12) |
```

- **Page Directory (PD)**: 1024 entries, each pointing to a Page Table
- **Page Table (PT)**: 1024 entries, each pointing to a frame

Only Page Tables for actually-used regions are allocated. Most PD entries are NULL.

### Four-Level (x86-64):

64-bit addresses use a 4-level structure (PML4 → PDPT → PD → PT) covering 48-bit virtual address space (256 TB per process).

```
Virtual address: | PML4 (9) | PDPT (9) | PD (9) | PT (9) | offset (12) |
```

Each table walk: 4 memory accesses. Without TLB, every memory access costs 5 memory accesses total.

### Inverted Page Table

A normal page table has one entry per virtual page — which means huge tables for 64-bit address spaces. An **inverted page table** flips the mapping: one entry per **physical frame**.

```
Normal page table:  virtual page → physical frame   (one table per process)
Inverted page table: physical frame → (PID, virtual page)   (one global table)
```

Size: fixed at `number_of_physical_frames` entries — independent of virtual address space size.

Lookup: given (PID, page_number), search the table for a matching entry. Linear search is too slow ($O(n)$ per access), so a **hash table** is used:

```
hash(PID, page_number) → bucket → search chain for exact match → frame number
```

Average lookup: $O(1)$ with good hash function and low collision rate.

Used in: IBM PowerPC, UltraSPARC, IA-64 (Itanium). Not used in x86-64 (which uses hierarchical page tables instead).

**Drawback**: sharing pages between processes is harder — each physical frame maps to only one (PID, virtual page) pair. Sharing requires special handling (aliasing entries).

### Hashed Page Table

A generalization of the inverted page table. Each entry in the hash table stores (virtual_page, frame_number, next_pointer). Multiple entries can chain from the same hash bucket.

```
hash(virtual_page) → chain: [vp1 → frame5] → [vp2 → frame12] → NULL
```

Common in 64-bit address spaces where hierarchical tables would be sparse and wasteful. The hash table can be sized proportionally to physical memory rather than virtual address space.

## Segmentation

**Segmentation** is an alternative to paging where address space is divided into **variable-size logical segments** corresponding to program structure:

- Code segment
- Data segment
- Stack segment
- Heap segment
- Shared library segments

A logical address has two parts: **(segment number, offset)**.

The **segment table** maps each segment to a physical base address and limit:

| Segment | Base | Limit |
|---|---|---|
| 0 (code) | 4096 | 1024 |
| 1 (data) | 8192 | 512 |
| 2 (stack) | 16384 | 2048 |

Translation: `physical = base[segment] + offset` if `offset < limit`, else segmentation fault.

### Segmentation vs Paging

| | Paging | Segmentation |
|---|---|---|
| Division | Fixed-size pages | Variable-size segments |
| External fragmentation | None | Yes |
| Internal fragmentation | Yes (last page) | None |
| Protection | Per-page | Per-segment (matches logical structure) |
| Sharing | Per-page | Per-segment (easy to share code) |

### Segmentation with Paging (x86-64 Reality)

Modern x86-64 Linux/Windows use **flat segmentation** (all segments cover the full address space, effectively disabling segmentation) combined with paging. Pure segmentation is rarely used in modern OSes. The concept is important for understanding legacy systems and how protection rings work.

## Virtual Memory

**Virtual memory** decouples the logical address space from physical RAM. Not all of a process's address space needs to be in RAM at once. Pages not currently needed can be stored on disk (the **swap space** or **page file**).

Benefits:
- Run programs larger than physical RAM
- More processes fit simultaneously (each uses only the pages it's actively using)
- Faster process startup (load only the first few pages)
- On-demand loading of code pages (only execute the parts actually called)

### Demand Paging

Pages are loaded into RAM **only when accessed**. At startup, a process gets a minimal set of pages. When an unloaded page is accessed, a **page fault** occurs.

### Page Fault Handling

1. CPU references a virtual address
2. MMU checks the PTE: P (present) bit = 0
3. CPU raises a **page fault exception**
4. OS page fault handler runs:
   a. Check if the address is valid (in VMA — virtual memory area). If not → segmentation fault (SIGSEGV)
   b. Find a free frame in physical RAM
   c. If no free frame: **page replacement** (choose a victim page, write it to disk if dirty)
   d. Read the needed page from disk into the frame
   e. Update the PTE: set frame number and P bit = 1
   f. Resume the faulting instruction (CPU retries the access)
5. Process continues

Page faults are **expensive** — disk I/O takes milliseconds. RAM access takes nanoseconds. A page fault is ~100,000× slower than a cache hit.

### Page Replacement Algorithms

When all frames are full and a new page must be loaded, the OS must **evict** an existing page. Which one?

#### FIFO (First-In First-Out)

Evict the page that has been in memory the longest.

```
Reference string: 1 2 3 4 1 2 5 1 2 3 4 5
Frames: 3

1 2 3 | fault→1 in  (evict nothing)
1 2 3 | fault→2 in  ...
1 2 3 | fault→3 in
4 2 3 | fault→4 in  (evict 1 — oldest)
4 1 3 | fault→1 in  (evict 2)
4 1 2 | fault→2 in  (evict 3)
5 1 2 | fault→5 in  (evict 4)
...
```

Simple but suffers from **Belady's anomaly**: more frames can cause more page faults with FIFO.

#### Optimal (OPT)

Evict the page that will not be used for the longest time in the future. Impossible to implement (requires future knowledge) but used as a benchmark.

#### LRU (Least Recently Used)

Evict the page that hasn't been used for the longest time. Approximates OPT by using past behavior as a predictor.

LRU doesn't suffer Belady's anomaly.

**Implementation challenges**: exact LRU requires updating a timestamp on every access — expensive. Hardware provides **Accessed (A)** and **Dirty (D)** bits, which the OS reads periodically.

**Clock Algorithm (Second Chance)**: FIFO with a second chance. Pages get a reference bit. When choosing a victim, scan in circular order — if reference bit = 1, clear it and skip (give second chance); if 0, evict. Cheap approximation of LRU.

**Enhanced Clock (NRU)**: Two bits: reference (R) and modified (M). Prefer evicting:
1. Not referenced, not modified (cheapest to evict)
2. Not referenced, modified (must write to disk)
3. Referenced, not modified
4. Referenced, modified (most expensive — in use and dirty)

#### Comparison

| Algorithm | Fault Rate | Implementation | Belady's Anomaly |
|---|---|---|---|
| FIFO | High | Simple | Yes |
| OPT | Lowest (theoretical) | Impossible in practice | No |
| LRU | Near-optimal | Expensive (exact) | No |
| Clock / NRU | Good | Moderate | No |

## Thrashing

**Thrashing** occurs when a process (or the entire system) spends more time swapping pages in and out than executing useful instructions.

Cause: too many processes competing for too little physical RAM. Each process gets fewer frames than its working set. Every memory access faults. The system spends all time in I/O.

```
CPU utilization drops → OS loads more processes to increase utilization
→ each process gets even fewer frames → more page faults → CPU utilization drops further
→ spiral into thrashing
```

### Working Set Model (Denning, 1968)

The **working set** of a process is the set of pages it actively uses in a recent time window Δ.

$$W(t, \Delta) = \{ \text{pages referenced in interval } (t - \Delta, t] \}$$

The OS tracks working sets and only schedules processes when the system has enough free frames to hold their working sets. If total working set demand > available frames → suspend some processes (reduce degree of multiprogramming).

### Page Fault Frequency (PFF) Control

Monitor each process's page fault rate:
- Too high → give it more frames
- Too low → take away frames
- If a process needs more frames but none are available → suspend it

## Memory Allocation: Kernel Structures

### Slab Allocator (Linux)

For kernel objects (inodes, task_structs, etc.), Linux uses a **slab allocator**: pre-allocates pools of identically-sized objects. Allocation is fast (no need to search for a free block), and fragmentation is minimized since objects always fit their slab exactly.

### Buddy System

Linux uses the **buddy system** for physical frame allocation. Memory is divided into power-of-2 blocks. When splitting a block, you create two equal "buddies". When both buddies are free, they merge back into a larger block. Fast allocation/deallocation with limited fragmentation.

## Memory Layout of a Process

A typical Linux process memory layout (low to high address):

```
0x0000000000000000   (unmapped — NULL pointer trap)
Text segment          (code, read-only)
Data segment          (initialized global/static variables)
BSS segment           (uninitialized globals, zero-filled)
Heap                  (grows upward via brk/mmap)
...  (empty space)
Memory-mapped files   (shared libraries, mmap'd files)
...  (empty space)
Stack                 (grows downward)
Kernel space          (0xFFFF800000000000 and above — x86-64)
```

Key syscalls:
- `brk()` / `sbrk()` — extend heap (legacy)
- `mmap()` — map file or anonymous memory at arbitrary address
- `munmap()` — unmap region
- `mprotect()` — change protection on a region (R/W/X flags in PTEs)

## Copy-on-Write (CoW)

**Copy-on-Write** is a memory optimization used heavily by `fork()`. When a process forks, the child doesn't get a full copy of the parent's memory. Instead, both processes share the same physical pages, marked **read-only** in both page tables.

When either process writes to a shared page:
1. CPU triggers a **protection fault** (write to read-only page)
2. OS page fault handler recognizes this as a CoW page
3. OS copies the page to a new physical frame
4. Updates the writing process's PTE to point to the new frame with write permission
5. The other process keeps the original page

```
After fork():
  Parent PTE → [Page X, read-only]
  Child PTE  → [Page X, read-only]   (same physical page)

After child writes:
  Parent PTE → [Page X, read-only]   (original)
  Child PTE  → [Page X', read-write] (new copy)
```

Pages that are never written are never copied. This makes `fork()` nearly instantaneous regardless of process size — only the page tables are duplicated, not the actual data. A `fork()` + `exec()` pattern copies almost nothing.

CoW is also used by `mmap()` with `MAP_PRIVATE` — modifications are local to the process and don't affect the underlying file.

## Memory-Mapped Files

`mmap()` maps a file (or anonymous memory) directly into a process's virtual address space. Instead of using `read()`/`write()` syscalls, the process accesses file contents through normal memory operations.

```c
#include <sys/mman.h>

int fd = open("data.bin", O_RDWR);
void *addr = mmap(NULL, file_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// Access file contents directly through memory
int *data = (int *)addr;
data[0] = 42;          // Writes directly to the file (eventually)

munmap(addr, file_size);
```

**How it works**: The OS sets up PTEs for the mapped region but doesn't load the data. When the process accesses a page, a page fault occurs. The OS reads the corresponding file block into a frame and maps it. Subsequent accesses hit the page cache — no syscall needed.

**Flags**:
- `MAP_SHARED` — changes are visible to other processes and written back to the file
- `MAP_PRIVATE` — CoW semantics, changes are local to this process
- `MAP_ANONYMOUS` — no file backing, zero-filled (used for heap allocation)
- `MAP_HUGETLB` — use huge pages (2MB/1GB)
- `MAP_FIXED` — map at exact address (dangerous: can overwrite existing mappings)

**Advantages**: No `read()`/`write()` syscall overhead; file data is shared through the page cache (multiple processes mapping the same file share the same physical pages); random access is natural (no seeking).

**Shared libraries** (.so files) are loaded via `mmap()` — the code pages are shared read-only across all processes using that library.

## NUMA (Non-Uniform Memory Access)

On multi-socket servers, each CPU socket has its own local memory bank. Accessing local memory is fast (~80ns); accessing another socket's memory goes through an interconnect (QPI/UPI) and is slower (~150ns).

```
Socket 0: [Core 0-7] ←→ [Local RAM 0]    (fast)
               ↕ QPI/UPI interconnect      (slow)
Socket 1: [Core 8-15] ←→ [Local RAM 1]   (fast)
```

The OS must be NUMA-aware:
- **Memory allocation**: allocate pages from the NUMA node where the process is running
- **Process scheduling**: keep processes on the same node as their memory
- **Page migration**: if a process is moved to another node, consider migrating its pages too

Linux NUMA policies:
```c
#include <numaif.h>

// Set memory policy for current process
set_mempolicy(MPOL_BIND, &node_mask, max_node);  // Allocate only from specific nodes
set_mempolicy(MPOL_INTERLEAVE, &node_mask, max_node);  // Round-robin across nodes

// Move pages between nodes
move_pages(pid, count, pages, nodes, status, flags);
```

```bash
# View NUMA topology
numactl --hardware

# Run with specific NUMA policy
numactl --membind=0 --cpunodebind=0 ./program

# Check NUMA statistics
numastat -p <pid>
```

## OOM (Out-of-Memory) Killer

When the system runs critically low on memory and all recovery mechanisms fail (page replacement, swap), Linux's **OOM killer** terminates processes to free memory.

The OOM killer selects victims based on an **OOM score** (0–1000) computed from:
- Memory consumption (RSS — Resident Set Size)
- Process age (younger processes score higher)
- Whether it's a privileged (root) process (lower score)
- Per-process `oom_score_adj` (-1000 to 1000, tunable)

```bash
# View OOM score of a process
cat /proc/<pid>/oom_score

# Protect a critical process from OOM killer
echo -1000 > /proc/<pid>/oom_score_adj    # Never kill

# Make a process the most likely OOM victim
echo 1000 > /proc/<pid>/oom_score_adj
```

The OOM killer sends `SIGKILL` to the selected victim. This is a last resort — proper memory management (swap, memory limits via cgroups) should prevent OOM conditions.

**Overcommit**: Linux allows `malloc()` to succeed even when total allocations exceed physical RAM + swap (overcommit mode). The assumption is that not all allocated memory is used simultaneously. If processes actually try to use more than available, the OOM killer activates.

```bash
# Overcommit modes (vm.overcommit_memory):
# 0 = Heuristic overcommit (default)
# 1 = Always overcommit (never fail malloc)
# 2 = Don't overcommit (strict: total allocations ≤ swap + ratio*RAM)
sysctl vm.overcommit_memory
```

## Swap Space Management

**Swap space** extends physical memory to disk. When RAM is full, the OS moves less-used pages to swap to free frames for more active pages.

### Swap Area Layout

Linux supports swap **partitions** and swap **files**:

```bash
# Create swap file
dd if=/dev/zero of=/swapfile bs=1M count=4096
mkswap /swapfile
swapon /swapfile

# View swap usage
swapon --show
free -h
```

### Swappiness

The `vm.swappiness` parameter (0–200, default 60) controls how aggressively the kernel moves pages to swap:

- **swappiness = 0**: Only swap to avoid OOM
- **swappiness = 100**: Equally prefer reclaiming page cache and swapping anonymous pages
- **swappiness = 200**: Strongly prefer swapping anonymous pages

```bash
sysctl vm.swappiness=10    # Less aggressive swapping (good for desktops)
```

### Page Cache vs Anonymous Pages

Linux distinguishes:
- **Page cache** (file-backed pages): Contents from files. Can be evicted and re-read from disk. Cheap to reclaim.
- **Anonymous pages** (heap, stack, mmap MAP_ANONYMOUS): No file backing. Must be written to swap before eviction. Expensive to reclaim.

The kernel maintains **active/inactive lists** for both types. Pages start on the inactive list and move to the active list on re-access. Reclamation preferentially targets the inactive list.

### SSD Considerations

Traditional swap on HDDs is painfully slow (seek time ~10ms). SSDs make swap much more viable (~0.1ms access). However, excessive swapping to SSD shortens its lifespan due to write amplification. **zswap** compresses pages in RAM before writing to swap — often 2–3× compression ratio reduces actual swap I/O.
