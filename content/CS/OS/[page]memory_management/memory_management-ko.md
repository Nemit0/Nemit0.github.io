# 메모리 관리

## 왜 메모리 관리가 필요한가?

여러 프로세스가 단일 물리 RAM을 공유합니다. 메모리 관리는 세 가지 문제를 해결합니다:

1. **보호** — 한 프로세스가 다른 프로세스의 메모리를 읽거나 쓰지 못하도록
2. **재배치** — 프로세스가 RAM의 어느 위치에 로드되더라도 실행될 수 있도록
3. **효율적 활용** — RAM은 제한적이므로 최대한 많은 프로세스를 수용

OS는 각 프로세스에게 **추상화**를 제공합니다: 0부터 시작하는 개인적이고 연속적인 주소 공간. 실제 물리적 배치는 단편화되어 있거나 부분적으로 디스크에 있거나 다른 프로세스와 공유될 수 있습니다.

---

## 주소 유형

- **논리(가상) 주소**: CPU가 생성/프로그램에서 사용하는 주소. 코드에서 보이는 주소.
- **물리 주소**: 실제 RAM 하드웨어의 주소.

**MMU(Memory Management Unit)**는 CPU 내부의 하드웨어로, 런타임에 논리 → 물리 주소를 변환합니다.

---

## 페이징

**페이징**은 물리 메모리와 논리 주소 공간을 모두 고정 크기 블록으로 나눠 외부 단편화를 제거합니다:

- 물리 메모리 → **프레임** (물리 페이지 프레임)
- 논리 주소 공간 → **페이지**
- 페이지와 프레임은 같은 크기 (보통 4 KB, 때로는 2 MB나 1 GB의 대용량 페이지)

프로세스의 페이지들은 임의의 빈 프레임에 분산될 수 있습니다 — 연속적일 필요가 없습니다.

### 주소 변환

논리 주소는 두 필드로 분리됩니다:

```
| 페이지 번호 (p) | 페이지 오프셋 (d) |
```

4 KB 페이지를 가진 32비트 주소:
- 페이지 오프셋: 12비트 (2^12 = 4096 = 4 KB)
- 페이지 번호: 20비트 (2^20 = 프로세스당 100만 페이지)

**페이지 테이블**이 페이지 번호 → 프레임 번호를 매핑합니다:

```
논리: 페이지 3, 오프셋 100
페이지 테이블[3] = 프레임 7
물리: 프레임 7, 오프셋 100 → 주소 = 7 * 4096 + 100
```

각 프로세스는 자체 페이지 테이블을 가지며 메모리에 저장됩니다. CPU의 **페이지 테이블 기저 레지스터** (x86의 CR3)가 현재 프로세스의 페이지 테이블을 가리킵니다.

### 페이지 테이블 항목 (PTE)

각 항목은 프레임 번호 이상의 정보를 저장합니다:

| 비트 | 이름 | 의미 |
|---|---|---|
| 프레임 번호 | — | 물리 프레임 주소 |
| P | Present/Valid | 이 페이지가 현재 RAM에 있는가? |
| R/W | Read/Write | 쓰기 접근이 허용되는가? |
| U/S | User/Supervisor | 유저 모드에서 접근 가능한가? |
| A | Accessed | CPU가 읽기 또는 쓰기 시 설정 |
| D | Dirty | CPU가 쓰기 시 설정 (페이지가 수정됨) |
| NX | No-Execute | 이 페이지에서 명령어 가져오기 불가 |

P 비트는 가상 메모리의 핵심입니다 — 0이면 페이지가 RAM에 없고 (디스크에 있음), 접근 시 **페이지 폴트**가 발생합니다.

### 내부 단편화

페이징은 외부 단편화가 없지만 **내부 단편화**가 있습니다: 프로세스가 100바이트만 필요해도 4 KB 페이지 전체가 할당됩니다. 평균적으로 할당당 반 페이지가 낭비됩니다.

---

## TLB (Translation Lookaside Buffer)

순수 페이징은 명령어당 **두 번의 메모리 접근**이 필요합니다: 페이지 테이블 읽기 하나, 실제 데이터 접근 하나. 메모리 지연이 두 배가 됩니다.

**TLB**는 MMU 내부의 고속 하드웨어 캐시로 최근의 페이지-번호-to-프레임-번호 변환을 저장합니다:

```
CPU가 논리 주소 생성
    ↓
TLB 조회 (페이지 번호로 키 검색)
    ↓
TLB 히트 (>95%) → ~1 사이클만에 프레임 번호 획득
    ↓
TLB 미스 → 메모리에서 페이지 테이블 탐색 (수백 사이클) → TLB에 저장
```

컨텍스트 스위치 시 (다른 프로세스 = 다른 페이지 테이블), CPU가 ASID/PCID를 지원하지 않으면 TLB를 **플러시**해야 합니다.

---

## 다단계 페이지 테이블

32비트 프로세스에서 4 KB 페이지를 사용하면 $2^{20}$ = 100만 개의 페이지 테이블 항목이 필요합니다. 항목당 4바이트면 **프로세스당 4 MB** — 주소 공간의 대부분을 사용하지 않더라도.

**다단계 페이지 테이블**은 트리 구조를 사용합니다: 실제로 사용되는 주소 공간 부분에만 페이지 테이블 메모리를 할당합니다.

### x86-32 2단계:
```
가상 주소: | PD 인덱스 (10) | PT 인덱스 (10) | 오프셋 (12) |
```

### x86-64 4단계:
```
가상 주소: | PML4 (9) | PDPT (9) | PD (9) | PT (9) | 오프셋 (12) |
```
48비트 가상 주소 공간(프로세스당 256 TB) 지원. 테이블 탐색마다 4번의 메모리 접근.

### 역 페이지 테이블 (Inverted Page Table)

일반 페이지 테이블은 가상 페이지당 하나의 항목을 가집니다 — 64비트 주소 공간에서는 거대한 테이블이 됩니다. **역 페이지 테이블**은 매핑을 뒤집습니다: **물리 프레임**당 하나의 항목.

```
일반 페이지 테이블:  가상 페이지 → 물리 프레임   (프로세스당 하나)
역 페이지 테이블:    물리 프레임 → (PID, 가상 페이지)   (전역 하나)
```

크기: `물리_프레임_수` 항목으로 고정 — 가상 주소 공간 크기에 독립적.

조회: (PID, page_number)가 주어지면 일치하는 항목을 테이블에서 검색합니다. 선형 탐색은 너무 느리므로 ($O(n)$) **해시 테이블**을 사용합니다.

IBM PowerPC, UltraSPARC, IA-64 (Itanium)에서 사용됩니다. x86-64에서는 사용되지 않습니다 (계층적 페이지 테이블 사용).

**단점**: 프로세스 간 페이지 공유가 더 어렵습니다 — 각 물리 프레임이 하나의 (PID, 가상 페이지) 쌍에만 매핑됩니다.

## 세그멘테이션

**세그멘테이션**은 페이징의 대안으로, 주소 공간을 프로그램 구조에 해당하는 **가변 크기의 논리적 세그먼트**로 나눕니다:

- 코드 세그먼트
- 데이터 세그먼트
- 스택 세그먼트
- 힙 세그먼트

논리 주소: **(세그먼트 번호, 오프셋)**

**세그먼트 테이블**이 각 세그먼트를 물리 기저 주소와 한계로 매핑합니다:

| 세그먼트 | 기저 | 한계 |
|---|---|---|
| 0 (코드) | 4096 | 1024 |
| 1 (데이터) | 8192 | 512 |
| 2 (스택) | 16384 | 2048 |

변환: `물리 = base[segment] + offset` (단, `offset < limit`이어야 함, 그렇지 않으면 세그멘테이션 폴트)

### 페이징 vs 세그멘테이션

| | 페이징 | 세그멘테이션 |
|---|---|---|
| 분할 방식 | 고정 크기 페이지 | 가변 크기 세그먼트 |
| 외부 단편화 | 없음 | 있음 |
| 내부 단편화 | 있음 (마지막 페이지) | 없음 |
| 보호 | 페이지 단위 | 세그먼트 단위 (논리 구조에 부합) |
| 공유 | 페이지 단위 | 세그먼트 단위 (코드 공유 용이) |

현대 x86-64 Linux/Windows는 **플랫 세그멘테이션**(모든 세그먼트가 전체 주소 공간을 커버하여 세그멘테이션을 사실상 비활성화)과 페이징을 함께 사용합니다.

## 가상 메모리

**가상 메모리**는 논리 주소 공간을 물리 RAM으로부터 분리합니다. 프로세스의 주소 공간 전체가 동시에 RAM에 있을 필요가 없습니다. 현재 필요하지 않은 페이지는 디스크(**스왑 공간** 또는 **페이지 파일**)에 저장할 수 있습니다.

장점:
- 물리 RAM보다 큰 프로그램 실행
- 더 많은 프로세스가 동시에 실행 (각 프로세스는 활성 사용 중인 페이지만 사용)
- 빠른 프로세스 시작 (첫 몇 페이지만 로드)
- 코드 페이지의 요구 로딩

### 요구 페이징

페이지는 **접근될 때만** RAM에 로드됩니다. 시작 시 프로세스는 최소한의 페이지만 얻습니다. 로드되지 않은 페이지에 접근하면 **페이지 폴트**가 발생합니다.

### 페이지 폴트 처리

1. CPU가 가상 주소 참조
2. MMU가 PTE 확인: P(present) 비트 = 0
3. CPU가 **페이지 폴트 예외** 발생
4. OS 페이지 폴트 핸들러 실행:
   a. 주소가 유효한지 확인 (VMA — 가상 메모리 영역 내). 유효하지 않으면 → 세그멘테이션 폴트 (SIGSEGV)
   b. 물리 RAM에서 빈 프레임 찾기
   c. 빈 프레임이 없으면: **페이지 교체** (희생 페이지 선택, dirty면 디스크에 쓰기)
   d. 디스크에서 필요한 페이지를 프레임으로 읽기
   e. PTE 업데이트: 프레임 번호 설정, P 비트 = 1
   f. 폴트 발생 명령어 재시작
5. 프로세스 계속

페이지 폴트는 **매우 비쌉니다** — 디스크 I/O는 밀리초가 걸립니다. RAM 접근은 나노초. 페이지 폴트는 캐시 히트보다 약 10만 배 느립니다.

## 페이지 교체 알고리즘

모든 프레임이 가득 찼을 때 어떤 페이지를 **퇴출**할지 선택합니다.

### FIFO (선입선출)

메모리에 가장 오래 있었던 페이지를 퇴출합니다. 간단하지만 **벨라디 이상(Belady's anomaly)**이 있습니다: 프레임 수가 늘어도 페이지 폴트가 더 많이 발생할 수 있습니다.

### OPT (최적)

미래에 가장 오랫동안 사용되지 않을 페이지를 퇴출합니다. 미래를 알아야 하므로 구현 불가능하지만 벤치마크로 사용됩니다.

### LRU (최근 최소 사용)

가장 오랫동안 사용되지 않은 페이지를 퇴출합니다. 과거 동작으로 미래를 예측하여 OPT에 근접합니다. 벨라디 이상이 없습니다.

정확한 LRU 구현은 모든 접근마다 타임스탬프 업데이트가 필요하여 비용이 높습니다. 하드웨어가 **접근(A)** 및 **더티(D)** 비트를 제공하며 OS가 주기적으로 읽습니다.

**시계 알고리즘 (두 번째 기회)**: 참조 비트를 이용한 FIFO. 희생자 선택 시 원형으로 스캔 — 참조 비트 = 1이면 비트를 지우고 건너뜀 (두 번째 기회), 0이면 퇴출. LRU의 저렴한 근사.

### 알고리즘 비교

| 알고리즘 | 폴트율 | 구현 | 벨라디 이상 |
|---|---|---|---|
| FIFO | 높음 | 단순 | 있음 |
| OPT | 최저 (이론) | 구현 불가 | 없음 |
| LRU | 최적에 근접 | 비용 높음 (정확) | 없음 |
| 시계/NRU | 양호 | 보통 | 없음 |

## 스래싱 (Thrashing)

**스래싱**은 프로세스(또는 시스템 전체)가 유용한 명령을 실행하는 것보다 페이지를 스왑하는 데 더 많은 시간을 소비할 때 발생합니다.

원인: 너무 많은 프로세스가 너무 적은 물리 RAM을 놓고 경쟁합니다. 각 프로세스가 워킹셋보다 적은 프레임을 얻습니다. 모든 메모리 접근이 폴트를 발생시킵니다.

```
CPU 사용률 감소 → OS가 더 많은 프로세스 로드
→ 각 프로세스가 더 적은 프레임 → 더 많은 페이지 폴트
→ CPU 사용률 더 감소 → 스래싱으로 나선
```

### 워킹셋 모델 (Denning, 1968)

프로세스의 **워킹셋**은 최근 시간 창 Δ 내에 활성적으로 사용하는 페이지 집합입니다.

$$W(t, \Delta) = \{ \text{구간 } (t - \Delta, t] \text{ 에서 참조된 페이지들} \}$$

OS는 워킹셋을 추적하고 시스템이 워킹셋을 보유할 충분한 빈 프레임이 있을 때만 프로세스를 스케줄링합니다. 총 워킹셋 요구 > 가용 프레임이면 일부 프로세스를 중단합니다(다중 프로그래밍 정도 감소).

## 프로세스의 메모리 레이아웃

일반적인 Linux 프로세스 메모리 레이아웃 (낮은 주소 → 높은 주소):

```
0x0000000000000000   (미매핑 — NULL 포인터 트랩)
텍스트 세그먼트       (코드, 읽기 전용)
데이터 세그먼트       (초기화된 전역/정적 변수)
BSS 세그먼트         (초기화되지 않은 전역, 0으로 채워짐)
힙                   (위로 증가, brk/mmap으로)
...  (빈 공간)
메모리 매핑 파일      (공유 라이브러리, mmap된 파일)
...  (빈 공간)
스택                 (아래로 증가)
커널 공간            (0xFFFF800000000000 이상 — x86-64)
```

주요 시스템 콜:
- `brk()` / `sbrk()` — 힙 확장 (레거시)
- `mmap()` — 파일 또는 익명 메모리를 임의 주소에 매핑
- `munmap()` — 영역 해제
- `mprotect()` — 영역의 보호 속성 변경 (PTE의 R/W/X 플래그)

## Copy-on-Write (CoW)

**Copy-on-Write**는 `fork()`에서 많이 사용되는 메모리 최적화입니다. 프로세스가 fork하면 자식은 부모 메모리의 전체 사본을 받지 않습니다. 대신 두 프로세스가 같은 물리 페이지를 공유하며 양쪽 페이지 테이블에서 **읽기 전용**으로 표시됩니다.

어느 프로세스든 공유 페이지에 쓸 때:
1. CPU가 **보호 폴트** 발생 (읽기 전용 페이지에 쓰기)
2. OS 페이지 폴트 핸들러가 CoW 페이지임을 인식
3. OS가 페이지를 새 물리 프레임으로 복사
4. 쓰기 프로세스의 PTE를 새 프레임으로 업데이트 (쓰기 권한 부여)

쓰지 않는 페이지는 절대 복사되지 않습니다. 이것이 `fork()`를 프로세스 크기에 관계없이 거의 순간적으로 만듭니다.

## 메모리 매핑 파일

`mmap()`은 파일(또는 익명 메모리)을 프로세스의 가상 주소 공간에 직접 매핑합니다. `read()`/`write()` 시스템 콜 대신 일반 메모리 연산으로 파일 내용에 접근합니다.

```c
int fd = open("data.bin", O_RDWR);
void *addr = mmap(NULL, file_size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// 메모리를 통해 파일 내용에 직접 접근
int *data = (int *)addr;
data[0] = 42;          // 파일에 직접 쓰기 (결국)

munmap(addr, file_size);
```

**동작 원리**: OS가 매핑 영역의 PTE를 설정하지만 데이터를 로드하지 않습니다. 프로세스가 페이지에 접근하면 페이지 폴트가 발생하고, OS가 해당 파일 블록을 프레임에 읽어와 매핑합니다.

**플래그**:
- `MAP_SHARED` — 변경이 다른 프로세스에 보이고 파일에 기록됨
- `MAP_PRIVATE` — CoW 의미론, 변경이 이 프로세스에만 국한
- `MAP_ANONYMOUS` — 파일 배킹 없음, 0으로 채워짐 (힙 할당에 사용)
- `MAP_HUGETLB` — 대용량 페이지 사용 (2MB/1GB)

**공유 라이브러리** (.so 파일)는 `mmap()`으로 로드됩니다 — 코드 페이지는 해당 라이브러리를 사용하는 모든 프로세스에서 읽기 전용으로 공유됩니다.

## NUMA (비균등 메모리 접근)

멀티 소켓 서버에서 각 CPU 소켓은 자체 로컬 메모리 뱅크를 가집니다. 로컬 메모리 접근은 빠르고 (~80ns), 다른 소켓의 메모리 접근은 인터커넥트를 거쳐 느립니다 (~150ns).

```
소켓 0: [코어 0-7] ←→ [로컬 RAM 0]    (빠름)
               ↕ QPI/UPI 인터커넥트     (느림)
소켓 1: [코어 8-15] ←→ [로컬 RAM 1]   (빠름)
```

OS는 NUMA를 인식해야 합니다:
- **메모리 할당**: 프로세스가 실행 중인 NUMA 노드에서 페이지 할당
- **프로세스 스케줄링**: 프로세스를 메모리와 같은 노드에 유지
- **페이지 마이그레이션**: 프로세스가 다른 노드로 이동하면 페이지도 마이그레이션 고려

```bash
# NUMA 토폴로지 확인
numactl --hardware

# 특정 NUMA 정책으로 실행
numactl --membind=0 --cpunodebind=0 ./program
```

## OOM (메모리 부족) 킬러

시스템이 메모리 부족 상태가 되고 모든 복구 메커니즘(페이지 교체, 스왑)이 실패하면, Linux의 **OOM 킬러**가 프로세스를 종료하여 메모리를 확보합니다.

OOM 킬러는 다음을 기반으로 계산된 **OOM 점수** (0–1000)에 따라 희생자를 선택합니다:
- 메모리 소비량 (RSS — Resident Set Size)
- 프로세스 연령 (새 프로세스가 높은 점수)
- 특권 프로세스 여부 (root는 낮은 점수)
- 프로세스별 `oom_score_adj` (-1000 ~ 1000, 조정 가능)

```bash
# 프로세스의 OOM 점수 확인
cat /proc/<pid>/oom_score

# 중요한 프로세스를 OOM 킬러로부터 보호
echo -1000 > /proc/<pid>/oom_score_adj
```

**오버커밋**: Linux는 총 할당이 물리 RAM + 스왑을 초과해도 `malloc()`이 성공하도록 허용합니다 (오버커밋 모드). 프로세스들이 실제로 가용량보다 많이 사용하려 하면 OOM 킬러가 활성화됩니다.

## 스왑 공간 관리

**스왑 공간**은 물리 메모리를 디스크로 확장합니다. RAM이 가득 차면 OS가 덜 사용되는 페이지를 스왑으로 이동하여 더 활발한 페이지를 위한 프레임을 확보합니다.

```bash
# 스왑 파일 생성
dd if=/dev/zero of=/swapfile bs=1M count=4096
mkswap /swapfile
swapon /swapfile

# 스왑 사용량 확인
swapon --show
free -h
```

### Swappiness

`vm.swappiness` 매개변수 (0–200, 기본 60)는 커널이 페이지를 스왑으로 이동하는 적극성을 제어합니다:

- **swappiness = 0**: OOM 방지를 위해서만 스왑
- **swappiness = 100**: 페이지 캐시 회수와 익명 페이지 스왑을 동등하게 선호

### 페이지 캐시 vs 익명 페이지

Linux는 구분합니다:
- **페이지 캐시** (파일 기반 페이지): 파일의 내용. 퇴출 후 디스크에서 다시 읽을 수 있음. 회수 비용 낮음.
- **익명 페이지** (힙, 스택, mmap MAP_ANONYMOUS): 파일 배킹 없음. 퇴출 전 스왑에 써야 함. 회수 비용 높음.

커널은 두 유형 모두 **활성/비활성 리스트**를 유지합니다. 페이지는 비활성 리스트에서 시작하고 재접근 시 활성 리스트로 이동합니다. 회수는 비활성 리스트를 우선적으로 대상으로 합니다.

### SSD 고려사항

기존 HDD의 스왑은 고통스럽게 느립니다 (탐색 시간 ~10ms). SSD는 스왑을 훨씬 더 실현 가능하게 만듭니다 (~0.1ms 접근). 하지만 SSD에 대한 과도한 스왑은 쓰기 증폭으로 수명을 단축합니다. **zswap**은 스왑에 쓰기 전 RAM에서 페이지를 압축합니다 — 보통 2-3배 압축률로 실제 스왑 I/O를 줄입니다.
