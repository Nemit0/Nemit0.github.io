# TCP/IP 4계층 모델

## 개요

**TCP/IP 모델**(인터넷 모델 또는 DoD 모델이라고도 한다)은 현대 인터넷을 실질적으로 구동하는 프레임워크다. OSI 모델(참조 표준)과 달리 TCP/IP는 인터넷이 실제로 어떻게 작동하는지 설명한다.

OSI의 7계층을 **4개 계층**으로 압축한다:

```
OSI 모델          TCP/IP 모델
─────────          ────────────
7 응용     ─┐
6 표현      ├─► 응용
5 세션     ─┘
4 전송     ────► 전송
3 네트워크 ────► 인터넷
2 데이터링크 ─┐
1 물리     ─┘──► 네트워크 액세스
```

---

## 4개 계층

### 계층 4 — 응용 계층

사용자 대면 프로토콜이 위치하는 최상위 계층. 애플리케이션이 여기서 네트워크와 상호작용한다.

**프로토콜**: HTTP/HTTPS, FTP, SMTP/IMAP/POP3, DNS, SSH, DHCP, SNMP
**데이터 단위**: 메시지

### 계층 3 — 전송 계층

서로 다른 호스트의 프로세스 간 종단 통신을 제공한다. **포트 번호**로 식별된다.

**프로토콜**: TCP, UDP
**데이터 단위**: 세그먼트(TCP) / 데이터그램(UDP)

| | TCP | UDP |
|---|---|---|
| 연결 | 연결 지향 | 비연결 |
| 신뢰성 | 전달 보장 | 최선형 |
| 순서 | 순서 보장 | 순서 없음 |
| 속도 | 느림 | 빠름 |

### 계층 2 — 인터넷 계층

출발지에서 목적지까지 네트워크를 가로질러 패킷을 라우팅한다. 논리적 주소 지정에 **IP 주소**를 사용한다.

**프로토콜**: IP(IPv4/IPv6), ICMP, ARP, OSPF, BGP
**데이터 단위**: 패킷
**핵심 장비**: 라우터

### 계층 1 — 네트워크 액세스 계층 (링크)

단일 네트워크 링크를 통한 전송을 처리한다. OSI의 데이터링크와 물리 계층을 결합했다.

**프로토콜**: 이더넷, Wi-Fi(802.11), PPP
**데이터 단위**: 프레임
**핵심 장비**: 스위치, NIC, 케이블

---

## 스택을 통한 데이터 흐름

```
브라우저에서 웹 서버로의 HTTP GET 요청:

응용:      [ HTTP 요청 ]
전송:      [ TCP Hdr | HTTP 요청 ]         ← 포트 80/443
인터넷:    [ IP Hdr | TCP Hdr | HTTP ]     ← 출발/목적 IP
네트워크 액세스: [ MAC Hdr | IP Hdr | TCP | HTTP | FCS ]  ← 출발/목적 MAC
              ─────────── 물리적 전송 ──────────
```

서버에서 스택은 아래에서 위로 읽힌다:
1. NIC가 프레임을 추출하고 MAC을 확인
2. OS가 패킷을 읽고 IP를 확인
3. TCP 계층이 세그먼트를 읽고 포트 80 → 웹 서버 찾기
4. 웹 서버가 HTTP 요청을 읽음

## 계층별 핵심 프로토콜

### 응용 계층 프로토콜

| 프로토콜 | 포트 | 용도 |
|---|---|---|
| HTTP | 80 | 웹 (평문) |
| HTTPS | 443 | 웹 (암호화) |
| FTP | 20, 21 | 파일 전송 |
| SSH | 22 | 보안 셸 |
| SMTP | 25 | 이메일 발송 |
| DNS | 53 | 도메인 → IP |
| DHCP | 67/68 | IP 주소 할당 |
| SNMP | 161 | 네트워크 관리 |

### 전송 계층

**TCP** (전송 제어 프로토콜):
- 3-way 핸드셰이크 (SYN, SYN-ACK, ACK)
- 흐름 제어 (슬라이딩 윈도우)
- 혼잡 제어
- 손실 시 재전송

**UDP** (사용자 데이터그램 프로토콜):
- 연결 설정 없음
- 확인 응답 없음
- 헤더가 8바이트뿐 (TCP의 20바이트 이상 대비)

### 인터넷 계층

**IPv4**: 32비트 주소 (예: 192.168.1.1)
**IPv6**: 128비트 주소 (예: 2001:db8::1)
**ICMP**: 오류 메시지 및 진단 (`ping`, `traceroute`)

### 네트워크 액세스 계층

**이더넷**: 지배적인 유선 LAN 프로토콜. MAC 주소(48비트) 사용.
**Wi-Fi(802.11)**: 무선 LAN. 역시 MAC 주소 사용.

## 계층별 주소 체계

| 계층 | 주소 유형 | 예시 | 범위 |
|---|---|---|---|
| 응용 | URL/호스트명 | `example.com` | 전역 |
| 전송 | 포트 번호 | 443 | 호스트별 |
| 인터넷 | IP 주소 | 192.168.1.1 | 논리적 (전역) |
| 네트워크 액세스 | MAC 주소 | AA:BB:CC:DD:EE:FF | 로컬 링크 |

**IP 주소**는 네트워크 간 라우팅을 위해 변하지 않는다.
**MAC 주소**는 로컬 네트워크 세그먼트 내에서만 사용되며 각 홉에서 변경된다(라우터가 출발/목적 MAC을 교체).

## 라우터에서의 캡슐화

```
호스트 A ──── 스위치 ──── 라우터 ──── 스위치 ──── 호스트 B

                      라우터는 각 홉에서 MAC을 재작성:
호스트 A → 라우터:  [ MAC_A | IP_A | ... | MAC_라우터 ]
라우터 → 호스트 B:  [ MAC_라우터 | IP_A | ... | MAC_B  ]
                  IP 주소는 종단 간 동일하게 유지됨!
```

- **IP 주소** (계층 3): 전체 경로에서 변경되지 않음
- **MAC 주소** (계층 2): 모든 라우터 홉에서 변경됨

---

## 실습 예제: 두 라우터를 거치는 패킷 이동

**시나리오**: 호스트 A(`10.0.1.2`)가 두 라우터를 통해 서버 B(`10.0.3.5`)에 HTTP GET을 보낸다.

```
호스트 A         라우터 R1              라우터 R2          서버 B
10.0.1.2         10.0.1.1/10.0.2.1    10.0.2.2/10.0.3.1  10.0.3.5
MAC: A1          MAC: R1a/R1b          MAC: R2a/R2b         MAC: B1
```

### 홉 1: 호스트 A → 라우터 R1

**인터넷 계층** IP 헤더:
```
출발 IP: 10.0.1.2    목적 IP: 10.0.3.5
TTL: 64              프로토콜: TCP
```
IP 헤더는 전체 여정에서 동일하게 유지된다 (종단 간 주소).

**네트워크 액세스 계층** 이더넷 프레임 (첫 번째 홉):
```
출발 MAC: A1    목적 MAC: R1a    (10.0.1.0 네트워크의 R1 인터페이스)
```

### 홉 2: 라우터 R1 → 라우터 R2

R1이 프레임을 받아 이더넷 헤더를 벗기고 IP 헤더를 읽는다:
- 목적지 `10.0.3.5` → 라우팅 테이블: "R1b 인터페이스를 통해 10.0.2.2로 전달"
- **TTL 감소**: 64 → 63
- 다음 홉을 위한 **새 이더넷 프레임** 생성:

```
출발 MAC: R1b   목적 MAC: R2a
IP 출발: 10.0.1.2  IP 목적: 10.0.3.5  (IP는 변경 없음!)
TTL: 63
```

### 홉 3: 라우터 R2 → 서버 B

R2가 프레임을 벗기고 IP 헤더를 읽는다:
- 목적지 `10.0.3.5` → 10.0.3.0에 직접 연결됨
- TTL 감소: 63 → 62
- 새 이더넷 프레임:

```
출발 MAC: R2b   목적 MAC: B1     (서버 B의 MAC, ARP로 획득)
IP 출발: 10.0.1.2  IP 목적: 10.0.3.5   (여전히 변경 없음!)
TTL: 62
```

### 핵심: 무엇이 변하고 무엇이 유지되는가

| 필드 | 변경 여부 | 이유 |
|---|---|---|
| IP 출발/목적 | **변경 없음** — 종단 간 동일 | 라우팅이 IP를 사용; IP는 영구 주소 |
| TTL | **변경** — 각 라우터에서 감소 | 무한 루프 방지 |
| MAC 출발/목적 | **변경** — 각 홉에서 교체 | MAC 주소는 링크 로컬 전용 |
| TCP Seq/Ack | **변경 없음** | 종단 간, 라우터가 건드리지 않음 |

### 각 지점에서의 패킷 상태

```
                  Eth 헤더          IP 헤더                TCP
호스트A → R1:  [A1→R1a]  [10.0.1.2→10.0.3.5, TTL=64]  [...]
R1 → R2:       [R1b→R2a] [10.0.1.2→10.0.3.5, TTL=63]  [...]
R2 → 서버B:    [R2b→B1]  [10.0.1.2→10.0.3.5, TTL=62]  [...]
```

**IP 주소는 논리적(종단 간), MAC 주소는 물리적(링크 로컬)**임을 이해하는 것이 핵심이다.

## TCP/IP vs OSI: 왜 둘 다 알아야 하는가

| 목적 | 모델 |
|---|---|
| 실제 구현 | TCP/IP |
| 학습, 문제 해결, 벤더 중립적 논의 | OSI |
| 프로토콜 설계 참조 | OSI |
| 실제 프로토콜 스택 | TCP/IP |

OSI 모델은 "이론상 어떻게 동작해야 하는가"이고, TCP/IP는 "인터넷에서 실제로 어떻게 동작하는가"다.
