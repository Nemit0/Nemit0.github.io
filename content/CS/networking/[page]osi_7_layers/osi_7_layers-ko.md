# OSI 7계층 모델

## OSI 모델이란?

**OSI (Open Systems Interconnection)** 모델은 서로 다른 네트워크 시스템이 통신하는 방식을 표준화한 개념적 프레임워크다. 1984년 ISO가 개발했으며, 네트워크 통신을 **7개 계층**으로 나누고 각 계층에 특정 역할을 부여한다.

OSI 모델은 네트워크를 이해하고 문제를 해결하기 위한 **참조 모델**로 사용된다. 실제 프로토콜(TCP/IP)이 완벽히 일치하지는 않지만, OSI는 공통 어휘를 제공한다.

---

## 7개 계층

```
송신자                              수신자
  │                                   │
  7  응용      ─────────────►   응용   7
  6  표현      ─────────────►   표현   6
  5  세션      ─────────────►   세션   5
  4  전송      ─────────────►   전송   4
  3  네트워크  ─────────────►   네트워크 3
  2  데이터링크 ────────────►   데이터링크 2
  1  물리      ═════════════►   물리   1
                    (실제 케이블)
```

데이터는 송신자 측에서 **하위 계층으로 내려가고**, 수신자 측에서 **상위 계층으로 올라간다**.

---

## 계층별 설명

### 계층 7 — 응용 계층 (Application)

**역할**: 네트워크와 사용자 애플리케이션 사이의 인터페이스. 애플리케이션이 통신에 사용하는 프로토콜을 정의한다.

**PDU**: 데이터
**프로토콜**: HTTP, HTTPS, FTP, SMTP, DNS, SSH, TELNET
**장비**: —

예시: 브라우저의 웹 페이지 요청, 이메일 클라이언트의 메일 발송.

### 계층 6 — 표현 계층 (Presentation)

**역할**: 애플리케이션과 네트워크 사이의 데이터 변환. 인코딩, 압축, 암호화/복호화를 처리한다.

**PDU**: 데이터
**프로토콜**: SSL/TLS(부분), JPEG, MPEG, ASCII, Unicode
**장비**: —

예시: HTTPS 트래픽 암호화, 전송 전 파일 압축.

### 계층 5 — 세션 계층 (Session)

**역할**: 애플리케이션 간 세션(논리적 연결)의 설정, 관리, 종료.

**PDU**: 데이터
**프로토콜**: NetBIOS, PPTP, RPC, NFS
**장비**: —

예시: 로그인 세션 유지, 파일 전송 재개.

### 계층 4 — 전송 계층 (Transport)

**역할**: 프로세스 간 종단 통신. 신뢰적 전달(TCP) 또는 빠른 최선형 전달(UDP) 제공. 분할 및 재조립 처리.

**PDU**: 세그먼트(TCP) / 데이터그램(UDP)
**프로토콜**: TCP, UDP
**장비**: 방화벽, 로드밸런서
**핵심 개념**: 포트 번호, 흐름 제어, 오류 복구, 다중화

### 계층 3 — 네트워크 계층 (Network)

**역할**: 논리적 주소 지정과 라우팅 — 서로 다른 네트워크를 거쳐 출발지에서 목적지까지 최적 경로를 결정한다.

**PDU**: 패킷
**프로토콜**: IP(IPv4/IPv6), ICMP, OSPF, BGP, RIP
**장비**: 라우터
**핵심 개념**: IP 주소, 라우팅 테이블, TTL

### 계층 2 — 데이터링크 계층 (Data Link)

**역할**: 동일 네트워크 세그먼트 내 노드 간 통신. 물리 주소(MAC) 지정, 프레이밍, 오류 감지 처리.

**PDU**: 프레임
**프로토콜**: 이더넷, Wi-Fi(802.11), ARP, PPP
**장비**: 스위치, 브리지
**핵심 개념**: MAC 주소, CSMA/CD, VLAN

### 계층 1 — 물리 계층 (Physical)

**역할**: 물리적 매체를 통해 원시 비트를 전송. 전기/광학/무선 규격을 정의한다.

**PDU**: 비트
**프로토콜**: USB, 블루투스, DSL, SONET
**장비**: 허브, 리피터, 케이블, 광섬유
**핵심 개념**: 대역폭, 신호, 인코딩

## 캡슐화와 역캡슐화

데이터가 **하위 계층으로 내려갈수록** 각 계층이 자신의 **헤더**(때로는 트레일러)를 추가한다. 이것이 **캡슐화**:

```
응용:     [ 데이터                            ]
전송:     [ TCP 헤더 | 데이터               ]  → 세그먼트
네트워크: [ IP 헤더 | TCP 헤더 | 데이터    ]  → 패킷
데이터링크: [ MAC 헤더 | IP | TCP | 데이터 | FCS ]  → 프레임
물리:     비트 ──────────────────────────────────
```

수신자 측에서 각 계층은 헤더를 **제거**하고 페이로드를 상위 계층으로 전달한다 — **역캡슐화**.

| 계층 | 이름 | PDU | 주소 | 주요 프로토콜 | 장비 |
|---|---|---|---|---|---|
| 7 | 응용 | 데이터 | — | HTTP, FTP, SMTP, DNS | — |
| 6 | 표현 | 데이터 | — | TLS, JPEG | — |
| 5 | 세션 | 데이터 | — | NetBIOS, RPC | — |
| 4 | 전송 | 세그먼트/데이터그램 | 포트 | TCP, UDP | 방화벽 |
| 3 | 네트워크 | 패킷 | IP 주소 | IP, ICMP, OSPF | 라우터 |
| 2 | 데이터링크 | 프레임 | MAC 주소 | 이더넷, ARP | 스위치 |

## 기억법

**상→하 (7→1)**: **아** 줘 **폰** 이 **세** 상에 **전** 네 **데** 물
*응용, 표현, 세션, 전송, 네트워크, 데이터링크, 물리*

---

## OSI vs TCP/IP 모델

| OSI | TCP/IP |
|---|---|
| 7 응용 | 응용 |
| 6 표현 | 응용 |
| 5 세션 | 응용 |
| 4 전송 | 전송 |
| 3 네트워크 | 인터넷 |
| 2 데이터링크 | 네트워크 액세스 |
| 1 물리 | 네트워크 액세스 |

TCP/IP는 OSI 상위 3계층을 응용 계층 하나로, 하위 2계층을 네트워크 액세스 계층 하나로 합쳤다.

---

## OSI로 문제 해결

아래쪽부터 위쪽으로:
1. **물리**: 케이블이 연결되어 있나? 링크 표시등이 켜져 있나?
2. **데이터링크**: 스위치 포트가 활성화되어 있나? ARP가 해석되나?
3. **네트워크**: ping이 되나? 경로가 올바른가?
4. **전송**: 포트가 열려 있나? 방화벽이 차단하나?
5. **응용**: 서비스가 실행 중인가? 자격증명이 올바른가?

## 실습 예제: HTTP 요청의 7계층 통과

**시나리오**: 브라우저가 `http://example.com/index.html`(서버 IP: 93.184.216.34)을 가져온다.

### 계층 7 — 응용
브라우저가 HTTP GET 요청을 생성:
```
GET /index.html HTTP/1.1
Host: example.com
```
이것은 순수한 **애플리케이션 데이터** — 전송/네트워크 정보가 아직 없다.

### 계층 6 — 표현
평문 HTTP에서는 변환 없음.
HTTPS의 경우 TLS가 페이로드를 암호화하고 TLS 레코드 헤더를 추가한다.

### 계층 5 — 세션
현재 HTTP 연결이 어느 브라우저 탭/세션에 속하는지 추적한다. HTTP/1.1 keep-alive에서는 여러 요청이 세션을 공유한다.

### 계층 4 — 전송 (TCP 세그먼트)
TCP가 HTTP 데이터를 **세그먼트**로 감싼다:
```
출발 포트: 54321  (OS가 할당한 임시 포트)
목적 포트: 80     (HTTP 잘 알려진 포트)
Seq:       1001
Ack:       5001
Data:      [HTTP GET /index.html ...]
```

### 계층 3 — 네트워크 (IP 패킷)
OS가 TCP 세그먼트를 **IP 패킷**으로 감싼다:
```
출발 IP:  192.168.1.100   (내 컴퓨터)
목적 IP:  93.184.216.34   (example.com)
TTL:      64
프로토콜: 6 (TCP)
```

### 계층 2 — 데이터링크 (이더넷 프레임)
NIC가 패킷을 MAC 주소를 사용한 **이더넷 프레임**으로 감싼다:
```
목적 MAC:  AA:BB:CC:DD:EE:FF   (기본 게이트웨이의 MAC, ARP로 획득)
출발 MAC:  11:22:33:44:55:66   (내 NIC의 MAC)
EtherType: 0x0800             (IPv4)
FCS:       [4바이트 CRC]
```
목적 MAC은 **게이트웨이의 MAC**이지, example.com의 것이 아니다 — 프레임은 다음 홉까지만 이동한다.

### 계층 1 — 물리
프레임이 전기 신호(또는 광섬유의 빛 펄스, Wi-Fi의 전파)로 변환되어 전송된다.

### 수신자 측: 역캡슐화 (아래→위)

```
물리      → NIC가 비트를 받아 프레임으로 조립
데이터링크 → 목적 MAC 확인(일치) → 이더넷 헤더 제거
네트워크  → 목적 IP 확인(93.184.216.34 = 나) → IP 헤더 제거
전송      → 목적 포트(80) 확인 → 웹 서버 프로세스로 전달
세션      → 올바른 HTTP 세션으로 라우팅
표현      → (TLS이면 복호화)
응용      → 웹 서버가 "GET /index.html HTTP/1.1" 읽음 → 파일 제공
```

### 캡슐화 요약

```
[HTTP 데이터                                         ] ← 계층 7
[TLS 레코드 | HTTP 데이터                           ] ← 계층 6 (HTTPS)
[TCP Hdr: 54321→443 | TLS | HTTP 데이터        ]    ← 계층 4
[IP Hdr: 192.168.1.100→93.184.216.34 | TCP | ... ]  ← 계층 3
[Eth: MAC_A→MAC_GW | IP | TCP | 데이터 | FCS  ]     ← 계층 2
[01001101 00101100 ... 비트 ...]                      ← 계층 1
```
