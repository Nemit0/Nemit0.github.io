# TCP 3-way / 4-way 핸드셰이크

## TCP에 핸드셰이크가 필요한 이유

TCP는 **연결 지향** 프로토콜이다. 데이터를 교환하기 전에 양측이:
1. 통신에 동의해야 한다
2. **시퀀스 번호**를 동기화해야 한다 (손실 또는 순서가 바뀐 데이터를 감지하기 위해)
3. 송수신 버퍼를 설정해야 한다

이 설정은 **3-way 핸드셰이크**로 이루어진다. 연결 종료에는 **4-way 핸드셰이크**가 필요하다.

---

## TCP 헤더 필드 (관련 항목)

| 필드 | 설명 |
|---|---|
| **SYN** | 동기화 — 연결 시작 |
| **ACK** | 확인 응답 — 수신한 데이터 확인 |
| **FIN** | 종료 — 연결 닫기 요청 |
| **RST** | 리셋 — 연결을 즉시 중단 |
| **Seq** | 시퀀스 번호 — 바이트 스트림에서의 위치 |
| **Ack** | 확인 응답 번호 — 다음으로 기대하는 바이트 |

## 3-way 핸드셰이크 (연결 설정)

```
클라이언트                            서버
  │                                   │
  │──── SYN (Seq=x) ────────────────►│  클라이언트: "연결하자. 내 seq = x"
  │                                   │
  │◄─── SYN-ACK (Seq=y, Ack=x+1) ───│  서버: "좋아. 내 seq = y, 네 x 받았어"
  │                                   │
  │──── ACK (Ack=y+1) ─────────────►│  클라이언트: "받았어. 네 y 확인"
  │                                   │
  │         [연결 설정 완료]           │
```

### 단계별 설명

**1단계 — SYN (클라이언트 → 서버)**
- 클라이언트가 `SYN=1`과 무작위로 선택한 **초기 시퀀스 번호(ISN)**: `Seq=x`를 포함한 세그먼트를 전송
- 클라이언트가 `SYN_SENT` 상태에 진입
- 서버는 `LISTEN` 상태여야 한다

**2단계 — SYN-ACK (서버 → 클라이언트)**
- 서버가 `SYN=1, ACK=1`로 응답
- 서버의 ISN: `Seq=y`
- 클라이언트 확인: `Ack=x+1` (클라이언트로부터 기대하는 다음 바이트)
- 서버가 `SYN_RECEIVED` 상태에 진입

**3단계 — ACK (클라이언트 → 서버)**
- 클라이언트가 `ACK=1`, `Ack=y+1` 전송 (서버로부터 기대하는 다음 바이트)
- 양측이 `ESTABLISHED` 상태에 진입
- 이제 데이터 전송이 시작될 수 있다

### 왜 3단계인가?

2단계로는 서버가 클라이언트의 SYN을 받았다는 것만 확인할 수 있다. 3단계(클라이언트가 서버의 SYN을 ACK)는 서버→클라이언트 경로도 작동한다는 것을 확인한다. 없으면 서버는 SYN-ACK이 수신됐는지 알 수 없다.

### 왜 무작위 ISN인가?

무작위 ISN은 다음을 방지한다:
- **오래된 중복 세그먼트**: 이전 연결의 낡은 패킷이 현재 데이터로 오인되는 것
- **TCP 하이재킹 공격**: 시퀀스 번호를 추측해서 악의적 데이터를 주입하는 것

## 4-way 핸드셰이크 (연결 종료)

TCP 연결은 **전이중(full-duplex)** — 각 방향이 독립적이다. 각 측이 자신의 FIN을 전송해야 종료된다.

```
클라이언트                            서버
  │                                   │
  │──── FIN (Seq=u) ────────────────►│  클라이언트: "나는 전송 완료"
  │                                   │  서버가 CLOSE_WAIT 진입
  │◄─── ACK (Ack=u+1) ──────────────│  서버: "FIN 받았어"
  │                                   │  [서버가 아직 데이터를 보낼 수도 있음]
  │                                   │
  │◄─── FIN (Seq=v) ────────────────│  서버: "나도 전송 완료"
  │                                   │  서버가 LAST_ACK 진입
  │──── ACK (Ack=v+1) ─────────────►│  클라이언트: "FIN 받았어"
  │                                   │
  │   [클라이언트: TIME_WAIT → CLOSED]│  [서버: CLOSED]
```

### 왜 4단계인가 (3단계가 아닌)?

클라이언트가 FIN을 보내면 "더 이상 보낼 데이터가 없다"는 의미다. 그러나 서버는 아직 보낼 데이터가 있을 수 있다. 따라서:
- 서버가 먼저 클라이언트의 FIN을 ACK
- 서버가 남은 데이터를 계속 전송
- 그 다음 서버가 자신의 FIN 전송

이 때문에 서버의 ACK와 FIN이 분리된다(2단계와 3단계).

### TIME_WAIT 상태

클라이언트가 최종 ACK를 전송한 후 완전히 닫기 전에 **2 × MSL**(최대 세그먼트 수명, 보통 60~120초) 동안 **TIME_WAIT** 상태가 된다.

**이유**:
1. **최종 ACK 도착 보장**: 서버가 ACK를 받지 못했다면 FIN을 재전송한다. 클라이언트가 살아 있어야 다시 ACK할 수 있다.
2. **오래된 패킷 만료 허용**: 이 연결의 낡은 중복 패킷이 같은 포트의 새 연결을 방해하지 못하도록 한다.

---

## TCP 상태 머신

| 상태 | 설명 |
|---|---|
| `LISTEN` | 서버가 들어오는 연결을 기다림 |
| `SYN_SENT` | 클라이언트가 SYN을 보내고 SYN-ACK를 기다림 |
| `SYN_RECEIVED` | 서버가 SYN을 받고 SYN-ACK를 보냄 |
| `ESTABLISHED` | 연결 열림, 데이터 전송 중 |
| `FIN_WAIT_1` | FIN을 보내고 ACK를 기다림 |
| `FIN_WAIT_2` | FIN의 ACK를 받고 서버의 FIN을 기다림 |
| `CLOSE_WAIT` | FIN을 받았고, 애플리케이션이 아직 보낼 데이터가 있음 |
| `LAST_ACK` | FIN을 보내고 최종 ACK를 기다림 |
| `TIME_WAIT` | 최종 종료 전 2×MSL 대기 |
| `CLOSED` | 연결 완전히 닫힘 |

## 반닫기 (Half-Close)

TCP 연결은 **반닫기** 상태가 될 수 있다: 한쪽이 전송을 완료했지만(FIN 전송) 아직 수신 중이다. 이것은 4-way 핸드셰이크의 2단계와 3단계 사이에서 정상적이다.

```python
import socket

s = socket.socket()
s.connect(('server', 80))
s.send(b'GET / HTTP/1.0\r\n\r\n')
s.shutdown(socket.SHUT_WR)   # FIN 전송 — 더 이상 전송 없음
data = s.recv(4096)           # 서버 응답은 아직 읽음
s.close()
```

## SYN 플러드 공격

**SYN 플러드**는 3-way 핸드셰이크를 악용하는 DoS 공격이다:
1. 공격자가 가짜 출발 IP로 많은 SYN을 전송
2. 서버가 자원을 할당하고 SYN-ACK를 전송
3. ACK가 오지 않음 — 서버의 연결 큐가 가득 참
4. 정상적인 연결이 거부됨

**완화책**: SYN 쿠키 — ACK가 도착할 때까지 메모리를 할당하는 대신 서버가 ISN에 상태를 인코딩한다.

---

## RST — 급격한 종료

4-way 우아한 종료 대신 **RST(리셋)** 세그먼트가 연결을 즉시 종료한다. 일반적인 원인:
- 애플리케이션 충돌
- 포트가 수신 대기 중이지 않음
- 방화벽 또는 보안 정책
- 반열린 연결 감지

```
클라이언트 ──── SYN ────────► 서버 (포트 닫힘)
클라이언트 ◄─── RST ──────── 서버 ("여기는 아무도 없어")
```

## 실습 예제: 구체적인 시퀀스 번호로 전체 연결 생명주기

**설정**: 클라이언트 ISN = 1000, 서버 ISN = 5000. 클라이언트가 "Hello"(5바이트), 서버가 "World!"(6바이트)를 전송.

### 1단계: 3-way 핸드셰이크

```
클라이언트                               서버
상태: CLOSED                             상태: LISTEN
  │  SYN                                    │
  │  Seq=1000, Ack=0, Flags=SYN             │
  │ ──────────────────────────────────────► │  서버: LISTEN → SYN_RECEIVED
  │                                         │
  │                      SYN-ACK            │
  │                      Seq=5000, Ack=1001 │  (Ack = 클라이언트 ISN + 1)
  │ ◄────────────────────────────────────── │
  │  클라이언트: SYN_SENT → ESTABLISHED      │
  │                                         │
  │  ACK                                    │
  │  Seq=1001, Ack=5001, Flags=ACK          │  (Ack = 서버 ISN + 1)
  │ ──────────────────────────────────────► │  서버: SYN_RECEIVED → ESTABLISHED
```

### 2단계: 데이터 교환

```
클라이언트                               서버
  │  DATA: "Hello"                          │
  │  Seq=1001, Ack=5001, Len=5              │  5바이트 데이터
  │ ──────────────────────────────────────► │
  │                                         │
  │                      ACK                │
  │                      Seq=5001, Ack=1006 │  (1001 + 5 = 1006, 다음 기대 바이트)
  │ ◄────────────────────────────────────── │
  │                                         │
  │                      DATA: "World!"     │
  │                      Seq=5001, Len=6    │
  │ ◄────────────────────────────────────── │
  │                                         │
  │  ACK: Seq=1006, Ack=5007                │  (5001 + 6 = 5007)
  │ ──────────────────────────────────────► │
```

**시퀀스 번호 규칙**: `Ack = 받은 Seq + Len` (= 당신으로부터 기대하는 다음 바이트)

### 3단계: 4-way 종료

```
클라이언트                               서버
  │  FIN: Seq=1006                          │
  │ ──────────────────────────────────────► │  서버: ESTABLISHED → CLOSE_WAIT
  │  클라이언트: FIN_WAIT_1                  │
  │                      ACK: Ack=1007      │  (FIN이 1바이트 소비)
  │ ◄────────────────────────────────────── │
  │  클라이언트: FIN_WAIT_2                  │
  │                      FIN: Seq=5007      │
  │ ◄────────────────────────────────────── │  서버: LAST_ACK
  │  ACK: Ack=5008                          │
  │ ──────────────────────────────────────► │  서버: CLOSED
  │  클라이언트: TIME_WAIT (120초)           │
  │  클라이언트: CLOSED                      │
```

### 재전송: 세그먼트가 손실되면?

"Hello"에 대한 ACK가 클라이언트에 도착하지 않으면:

```
클라이언트 전송:    Seq=1001, Len=5   → 네트워크에서 손실
클라이언트 대기...  타임아웃 (RTO, 초기 보통 200ms~3초, 매번 두 배로 증가)
클라이언트 재전송:  Seq=1001, Len=5   → 서버에 도착
서버 ACK:          Ack=1006
```

TCP의 **RTO(재전송 타임아웃)**는 각 실패 시 두 배로 증가한다 (지수 백오프): 200ms → 400ms → 800ms → ... 최대 ~2분, 그 후 연결이 포기된다.
