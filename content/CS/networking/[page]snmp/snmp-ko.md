# SNMP — 단순 네트워크 관리 프로토콜

## SNMP란?

**SNMP(Simple Network Management Protocol)**는 라우터, 스위치, 서버, 프린터, UPS 등 네트워크 장치를 모니터링하고 관리하기 위한 응용 계층 프로토콜(UDP 161/162)이다.

SNMP를 통해 중앙 **관리 시스템**이 개별 장치에 일일이 로그인하지 않고도 많은 장치에서 정보를 수집하거나 명령을 보낼 수 있다.

---

## 아키텍처

```
     네트워크 관리 시스템 (NMS)
      ┌────────────────────────────┐
      │   SNMP 관리자             │
      │   - 에이전트 폴링          │
      │   - 트랩 수신              │
      │   - 표시/경고              │
      └────────────┬───────────────┘
                   │ SNMP (UDP 161/162)
        ┌──────────┴──────────┐
        │                     │
  ┌─────┴────┐         ┌──────┴────┐
  │  라우터  │         │  스위치   │
  │  에이전트│         │  에이전트 │
  │  + MIB   │         │  + MIB   │
  └──────────┘         └──────────┘
```

### 세 가지 구성요소

1. **SNMP 관리자(NMS)**: 에이전트를 폴링하고 트랩을 받는 중앙 스테이션. 예: Nagios, Zabbix, PRTG, SolarWinds.

2. **SNMP 에이전트**: 관리 장치에서 실행되는 소프트웨어. 관리자 쿼리에 응답하고 자발적인 트랩을 전송한다.

3. **MIB(관리 정보 베이스)**: 에이전트가 보고할 수 있는 변수의 계층적 데이터베이스. 각 변수에는 **OID**가 있다.

---

## MIB과 OID

MIB은 트리 구조다. 각 노드에는 **OID(객체 식별자)** — 점으로 구분된 숫자 시퀀스가 있다:

```
1.3.6.1.2.1.1.1.0
│ │ │ │ │ │ │ │
│ │ │ │ │ │ │ └── 인스턴스 (0 = 스칼라)
│ │ │ │ │ │ └──── sysDescr (1)
│ │ │ │ │ └────── system (1)
│ │ │ │ └──────── mib-2 (1)
│ │ │ └────────── mgmt (2)
│ │ └──────────── internet (1)
│ └────────────── dod (6)
└────────────────── iso (1)
```

**sysDescr** (`1.3.6.1.2.1.1.1.0`): 장치의 텍스트 설명.

### 일반적인 MIB 변수

| OID | 이름 | 설명 |
|---|---|---|
| `1.3.6.1.2.1.1.1.0` | sysDescr | 장치 설명 |
| `1.3.6.1.2.1.1.3.0` | sysUpTime | 마지막 재부팅 이후 시간 |
| `1.3.6.1.2.1.1.5.0` | sysName | 장치 호스트명 |
| `1.3.6.1.2.1.2.1.0` | ifNumber | 인터페이스 수 |
| `1.3.6.1.2.1.2.2.1.10` | ifInOctets | 인터페이스당 수신 바이트 |
| `1.3.6.1.2.1.2.2.1.16` | ifOutOctets | 인터페이스당 송신 바이트 |

## SNMP PDU 유형

| PDU | 방향 | 설명 |
|---|---|---|
| **GetRequest** | 관리자 → 에이전트 | 하나 이상의 OID 값 요청 |
| **GetNextRequest** | 관리자 → 에이전트 | MIB 트리의 다음 OID 가져오기 (워킹용) |
| **GetBulkRequest** | 관리자 → 에이전트 | 여러 OID를 효율적으로 가져오기 (v2c+) |
| **SetRequest** | 관리자 → 에이전트 | OID 값 설정 (장치 구성) |
| **Response** | 에이전트 → 관리자 | Get/Set에 대한 응답 |
| **Trap** | 에이전트 → 관리자 | 자발적인 경보 (이벤트 기반) |
| **InformRequest** | 에이전트 → 관리자 | 확인이 있는 트랩 (v2c+) |

## SNMP 버전

### SNMPv1

- 원래 버전 (RFC 1157)
- 인증으로 커뮤니티 문자열 사용 (평문)
- 기본 PDU (Get, Set, Trap)
- 32비트 카운터 (고속 링크에서 오버플로)
- 보안 없음: 커뮤니티 문자열이 평문으로 전송

### SNMPv2c

- 커뮤니티 기반 v2 (RFC 1901)
- GetBulk 추가 (효율적인 테이블 검색)
- 64비트 카운터 (고속 링크 문제 해결)
- 여전히 평문 커뮤니티 문자열 사용
- InformRequest (확인된 트랩)

### SNMPv3

- RFC 3411~3418 (완전한 보안 모델)
- **인증**: MD5 또는 SHA (메시지 무결성, 진위성)
- **암호화**: DES 또는 AES (메시지 프라이버시)
- **접근 제어**: 다른 뷰를 가진 사용자별 권한
- 세 가지 보안 수준:
  - `noAuthNoPriv`: 인증 없음, 암호화 없음
  - `authNoPriv`: 인증됨, 암호화 안 됨
  - `authPriv`: 인증되고 암호화됨

**운영 환경에서는 SNMPv3를 사용해야 한다.** SNMPv1/v2c는 안전하지 않다.

---

## 커뮤니티 문자열 (v1/v2c)

**커뮤니티 문자열**은 관리자와 에이전트를 그룹화하는 비밀번호 같은 것이다:

- `public`: 기본 읽기 전용 커뮤니티 (모니터링)
- `private`: 기본 읽기-쓰기 커뮤니티 (구성)

이것들은 모든 패킷에서 **평문**으로 전송된다. 네트워크를 스니핑하는 누구나 읽을 수 있다. 항상 기본값을 변경할 것.

---

## SNMP 트랩 vs 폴링

| | 폴링 (Get) | 트랩 |
|---|---|---|
| 시작 | 관리자 | 에이전트 |
| 타이밍 | 주기적 | 이벤트 기반 (즉시) |
| 신뢰성 | 에이전트가 항상 응답 | UDP — v1에서는 ACK 없음 |
| 오버헤드 | 높음 (지속적 폴링) | 낮음 (이벤트 발생 시만) |
| 지연 | 폴링 간격까지 | 즉시 |

대부분의 배포에서 두 가지를 결합한다: **폴링**으로 메트릭(대역폭, CPU, 메모리) 수집, **트랩**으로 경보(링크 다운, 임계값 초과, 재부팅).

## 일반적인 SNMP 명령어

```bash
# net-snmp 도구 설치
apt install snmp

# 단일 OID 가져오기 (시스템 설명)
snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.1.1.0

# 전체 MIB 워크
snmpwalk -v2c -c public 192.168.1.1

# 인터페이스 서브트리 워크
snmpwalk -v2c -c public 192.168.1.1 1.3.6.1.2.1.2.2

# 인터페이스 트래픽 카운터 가져오기
snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.2.2.1.10.1  # ifInOctets.1
snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.2.2.1.16.1  # ifOutOctets.1

# SNMPv3
snmpget -v3 -l authPriv -u admin -a SHA -A authpass -x AES -X privpass 192.168.1.1 sysDescr.0
```

## 실습 예제: SNMP으로 인터페이스 대역폭 모니터링

**목표**: `192.168.1.1` 라우터의 인터페이스 1에서 대역폭 활용률 계산.

SNMP 카운터는 장치 재부팅 이후의 **누적 바이트**다. 처리량을 계산하려면 두 번 샘플을 채취해 델타를 계산한다.

### 1단계: 첫 번째 폴링 (t=0초)

```bash
snmpget -v2c -c public 192.168.1.1 \
  1.3.6.1.2.1.2.2.1.10.1 \   # ifInOctets.1  (수신 바이트)
  1.3.6.1.2.1.2.2.1.16.1     # ifOutOctets.1 (송신 바이트)

# 결과:
# ifInOctets.1  = 1,234,567,890
# ifOutOctets.1 = 987,654,321
```

### 2단계: 두 번째 폴링 (t=60초)

```bash
# 결과:
# ifInOctets.1  = 1,249,567,890   (+15,000,000바이트 / 60초)
# ifOutOctets.1 = 990,654,321     (+3,000,000바이트 / 60초)
```

### 3단계: 처리량 계산

```
인바운드 처리량  = (1,249,567,890 - 1,234,567,890) / 60초
                 = 15,000,000 / 60
                 = 250,000 바이트/초
                 = 250,000 × 8 비트/초
                 = 2,000,000 bps = 2 Mbps

아웃바운드 처리량 = (990,654,321 - 987,654,321) / 60
                  = 3,000,000 / 60
                  = 50,000 바이트/초 = 400 Kbps
```

### 4단계: 인터페이스 속도 확인

```bash
snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.2.2.1.5.1
# ifSpeed.1 = 1000000000  (1 Gbps 인터페이스)

활용률 (인바운드)   = 2,000,000 / 1,000,000,000 × 100 = 0.2%
활용률 (아웃바운드) = 400,000 / 1,000,000,000 × 100 = 0.04%
```

### 카운터 랩 (32비트 오버플로)

32비트 카운터는 최대 4,294,967,295(≈4 GB)다. 100 Mbps 링크에서 다음 시간마다 랩핑된다:
```
4,294,967,295 바이트 / (100,000,000 / 8 바이트/초) = 343초 ≈ 5.7분
```
이 때문에 SNMPv2c는 고속 인터페이스용 **64비트 카운터**(`ifHCInOctets`, `ifHCOutOctets`)를 도입했다.

## 실습 예제: SNMP 링크 다운 트랩

인터페이스 2가 다운되면 에이전트가 NMS에 자발적인 트랩을 전송한다:

```
SNMPv2c 트랩 PDU:
  출처:         192.168.1.1 (UDP 포트 162 목적지)
  커뮤니티:     public
  Enterprise OID: 1.3.6.1.6.3.1.1.5.3  (linkDown)
  변수 바인딩:
    sysUpTime.0    = 43200 (마지막 재부팅 이후 12시간, 1/100초 단위)
    snmpTrapOID.0  = 1.3.6.1.6.3.1.1.5.3  (linkDown)
    ifIndex.2      = 2       (어떤 인터페이스)
    ifDescr.2      = "GigabitEthernet0/1"
    ifAdminStatus.2 = 1      (admin: up — 관리자가 끄지 않음)
    ifOperStatus.2  = 2      (oper: down — 물리적 문제)
```

NMS가 이를 수신하고 경보를 발생시킨다:
```
경보: 192.168.1.1 (Router-Core-1) — GigabitEthernet0/1 링크 다운
      관리 상태: UP (관리적으로 비활성화되지 않음)
      가동 시간: 12h 0m
      조치 필요: 물리적 케이블/SFP 확인
```
