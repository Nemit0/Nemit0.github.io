# ARP와 RARP

## ARP가 해결하는 문제

호스트 A가 같은 네트워크의 호스트 B에 패킷을 보내려 할 때:
- A는 B의 **IP 주소**를 알고 있다 (DNS나 구성에서)
- A는 이더넷 프레임을 구성하기 위해 B의 **MAC 주소**가 필요하다
- A는 어떻게 B의 MAC을 찾을까?

**ARP(Address Resolution Protocol)**가 이를 해결한다. 로컬 네트워크에서 IP 주소를 MAC 주소로 매핑한다 (레이어 3 → 레이어 2).

---

## ARP 과정

```
호스트 A (192.168.1.10)              호스트 B (192.168.1.20)
MAC: AA:BB:CC:11:22:33               MAC: AA:BB:CC:44:55:66

  │                                   │
  │── ARP 요청 (브로드캐스트) ───────►│ (및 다른 모든 호스트)
  │   "192.168.1.20는 누구?           │
  │    192.168.1.10에게 알려줘"        │
  │                                   │
  │◄── ARP 응답 (유니캐스트) ─────────│
  │    "192.168.1.20은              │
  │     AA:BB:CC:44:55:66에 있어"      │
  │                                   │
  │  [A가 ARP 테이블에 B의 MAC 캐시]  │
  │                                   │
  │── B에게 이더넷 프레임 ───────────►│
  │   목적 MAC: AA:BB:CC:44:55:66      │
```

### ARP 요청

- **출발 MAC**: 송신자의 MAC
- **출발 IP**: 송신자의 IP
- **목적 MAC**: `FF:FF:FF:FF:FF:FF` (브로드캐스트 — "아직 모름")
- **목적 IP**: 해석할 IP
- **이더넷 브로드캐스트**로 전송 — 세그먼트의 모든 호스트가 수신

### ARP 응답

- **출발 MAC**: 응답자의 MAC (정답!)
- **출발 IP**: 응답자의 IP
- **목적 MAC**: 요청자의 MAC (유니캐스트로 다시 전송)
- **목적 IP**: 요청자의 IP

---

## ARP 캐시

ARP 응답은 반복적인 브로드캐스트를 피하기 위해 **ARP 테이블**(ARP 캐시라고도 함)에 캐시된다:

```bash
# Linux에서 ARP 캐시 보기
arp -n
ip neigh show

# 예시 출력:
192.168.1.1  dev eth0  lladdr aa:bb:cc:dd:ee:ff  REACHABLE
192.168.1.20 dev eth0  lladdr aa:bb:cc:44:55:66  REACHABLE
```

캐시 항목은 타임아웃 후 만료된다 (보통 20~60초). 동적 항목은 사용될 때 갱신된다.

---

## ARP 패킷 형식

```
하드웨어 유형:       이더넷 (1)
프로토콜 유형:       IPv4 (0x0800)
하드웨어 주소 길이:  6 (MAC)
프로토콜 주소 길이:  4 (IPv4)
작업:                1 = 요청, 2 = 응답
송신자 하드웨어 주소: 송신자 MAC
송신자 프로토콜 주소: 송신자 IP
목적 하드웨어 주소:  목적 MAC (요청에서는 모두 0)
목적 프로토콜 주소:  목적 IP
```

ARP는 **레이어 2/3 경계**에서 작동한다. 라우팅 불가 — ARP 트래픽은 로컬 브로드캐스트 도메인 내에만 머문다.

---

## Gratuitous ARP

**Gratuitous ARP(GARP)**는 송신자가 자신의 IP 주소를 묻는 ARP 요청이다. 다음 용도로 사용된다:
1. **IP 변경 알림**: 새 MAC을 이웃에게 알린다
2. **중복 감지**: 누군가 응답하면 IP 충돌이 있다
3. **ARP 캐시 업데이트**: 장애 조치 후 (VRRP, HSRP) 트래픽을 새 MAC으로 유도

```
호스트 A: "192.168.1.10는 누구?" (자기 자신에 대해 질문)
       → 예상 응답 없음, 하지만 모든 이웃이 캐시를 업데이트
```

---

## 프록시 ARP

라우터가 다른 서브넷의 호스트를 대신해 ARP 요청에 응답한다. 기본 게이트웨이 없는 호스트가 원격 네트워크에 접근할 수 있게 한다.

**현대 사용**: 대부분 권장하지 않음; 적절한 라우팅이 선호됨.

---

## ARP 스푸핑 (ARP 포이즈닝)

공격자가 가짜 ARP 응답을 보내 ARP 캐시를 오염시킨다:

```
공격자 → 호스트 A: "192.168.1.1 (게이트웨이)은 ATTACKER_MAC에 있어"
공격자 → 게이트웨이: "192.168.1.10 (호스트 A)은 ATTACKER_MAC에 있어"

결과:
A → 게이트웨이 트래픽 → 공격자를 통과 (MITM)
게이트웨이 → A 트래픽 → 공격자를 통과
```

**방어**:
- **동적 ARP 검사(DAI)**: 스위치가 DHCP 스누핑 테이블에 대해 ARP 패킷을 검증
- **정적 ARP 항목**: 중요한 항목을 수동으로 설정 (확장성 없음)
- **포트 보안**: 포트당 MAC 제한
- **HTTPS/TLS**: 트래픽이 가로채지더라도 암호화됨

---

## RARP — 역방향 ARP

**RARP**는 ARP의 역방향: **MAC 주소를 IP 주소로 매핑**한다. MAC은 알지만 IP를 모르는 디스크 없는 워크스테이션이 사용했다.

```
클라이언트 (MAC은 알지만 IP는 모름):
  ──── RARP 요청 (브로드캐스트) ────────► RARP 서버
       "내 MAC은 AA:BB:CC:11:22:33이야.
        내 IP는 뭐야?"
  ◄─── RARP 응답 ─────────────────────
       "네 IP는 192.168.1.100이야"
```

RARP는 **구식**이다. 다음으로 대체됐다:
- **BOOTP** (부트스트랩 프로토콜)
- **DHCP** (동적 호스트 구성 프로토콜) — 현대 표준

---

## IPv6: ARP 대신 NDP

IPv6는 ARP를 사용하지 않는다. 대신 ICMPv6의 일부인 **NDP(Neighbor Discovery Protocol)**를 사용한다:

| ARP | NDP (ICMPv6) |
|---|---|
| ARP 요청 | 이웃 권유 (NS) |
| ARP 응답 | 이웃 광고 (NA) |
| Gratuitous ARP | 자발적인 NA |
| 브로드캐스트 | 멀티캐스트 (더 효율적) |

NDP는 또한 라우터 발견과 상태 비저장 주소 자동 구성(SLAAC)을 처리한다.

## 실습 예제: ARP 해석 단계별 과정

**시나리오**: 호스트 A(`192.168.1.10`, MAC `AA:BB:CC:11:22:33`)가 같은 서브넷의 호스트 B(`192.168.1.20`)에 데이터를 보내려 한다. A의 ARP 캐시는 비어 있다.

### 1단계: ARP 캐시 확인 — 미스

```bash
# 호스트 A가 ARP 테이블 확인
arp -n
# 주소              HWtype  HWaddress  Flags   Iface
# 192.168.1.1      ether   ...        C       eth0
# 192.168.1.20 → 찾을 수 없음
```

### 2단계: ARP 요청 브로드캐스트

호스트 A가 LAN 전체에 브로드캐스트 이더넷 프레임을 전송:

```
이더넷 프레임:
  목적 MAC:  FF:FF:FF:FF:FF:FF  ← 브로드캐스트 (모든 호스트가 수신)
  출발 MAC:  AA:BB:CC:11:22:33
  EtherType: 0x0806            ← ARP

ARP 페이로드:
  작업:               1 (요청)
  송신자 MAC:         AA:BB:CC:11:22:33
  송신자 IP:          192.168.1.10
  목적 MAC:           00:00:00:00:00:00  ← 모름
  목적 IP:            192.168.1.20       ← 누가 이걸 갖고 있나?
```

세그먼트의 모든 호스트(192.168.1.x)가 이를 수신한다. 모든 호스트가 "192.168.1.20이 내 IP인가?"를 확인한다.

### 3단계: 호스트 B가 응답 (유니캐스트)

호스트 B(`192.168.1.20`, MAC `AA:BB:CC:44:55:66`)만 응답:

```
이더넷 프레임:
  목적 MAC:  AA:BB:CC:11:22:33  ← 요청자에게 유니캐스트
  출발 MAC:  AA:BB:CC:44:55:66
  EtherType: 0x0806

ARP 페이로드:
  작업:               2 (응답)
  송신자 MAC:         AA:BB:CC:44:55:66  ← 내 MAC이야
  송신자 IP:          192.168.1.20
  목적 MAC:           AA:BB:CC:11:22:33  ← 너
  목적 IP:            192.168.1.10
```

### 4단계: A가 캐시 업데이트하고 데이터 전송

```bash
# 응답 후 호스트 A의 ARP 테이블:
# 192.168.1.20    ether   AA:BB:CC:44:55:66   C   eth0
#                                             TTL: ~20-60초
```

호스트 A는 이제 B의 MAC으로 IP 패킷을 보낼 수 있다:

```
B에게 보내는 이더넷 프레임:
  목적 MAC:  AA:BB:CC:44:55:66
  출발 MAC:  AA:BB:CC:11:22:33
  EtherType: 0x0800  ← IPv4

IP 페이로드:
  출발: 192.168.1.10
  목적: 192.168.1.20
```

### 서브넷 외부 목적지에 대한 ARP

A가 `8.8.8.8`(Google DNS) — 서브넷 외부 — 에 접근하려면?
```
A는 8.8.8.8에 대해 ARP하지 않음 (다른 서브넷 → 로컬 LAN에 없음)
대신:
  A가 기본 게이트웨이: 192.168.1.1에 대해 ARP
  게이트웨이 MAC 획득: DD:EE:FF:00:11:22
  프레임 전송: 목적 MAC=DD:EE:FF:00:11:22, IP 목적=8.8.8.8
  라우터가 나머지 처리
```

## 실습 예제: ARP 스푸핑 공격

**네트워크**: 192.168.1.0/24

| 호스트 | IP | MAC |
|---|---|---|
| 피해자 | 192.168.1.10 | `AA:BB:CC:11:22:33` |
| 게이트웨이 | 192.168.1.1 | `DD:EE:FF:00:11:22` |
| 공격자 | 192.168.1.99 | `EE:EE:EE:EE:EE:EE` |

### 공격: 공격자가 가짜 ARP 응답 전송

```
공격자 → 피해자 (자발적인 ARP 응답):
  "192.168.1.1 (게이트웨이)은 EE:EE:EE:EE:EE:EE에 있어"
  → 피해자의 ARP 캐시: 192.168.1.1 = EE:EE:EE:EE:EE:EE  (오염!)

공격자 → 게이트웨이 (자발적인 ARP 응답):
  "192.168.1.10 (피해자)은 EE:EE:EE:EE:EE:EE에 있어"
  → 게이트웨이의 ARP 캐시: 192.168.1.10 = EE:EE:EE:EE:EE:EE  (오염!)
```

### 결과: 중간자 공격

```
공격 전:
피해자 ──────────────────────────────────► 게이트웨이 ──► 인터넷

공격 후:
피해자 ──► 공격자 ──► 게이트웨이 ──► 인터넷
                  ◄──────────────────────
(공격자가 감지를 피하기 위해 트래픽을 전달하면서 모든 것을 읽고/수정)
```

### 탐지

```bash
# 피해자에서: 게이트웨이 IP가 예상치 않은 MAC을 갖는지 확인
arp -n | grep 192.168.1.1
# 192.168.1.1   ether   ee:ee:ee:ee:ee:ee  C  eth0   ← dd:ee:ff:00:11:22여야 함!
```

### VRRP 장애 조치를 위한 Gratuitous ARP

VRRP(가상 라우터 이중화 프로토콜)은 장애 조치를 알리기 위해 Gratuitous ARP를 사용한다:

```
활성 라우터(R1)가 실패.
백업 라우터(R2)가 VIP(192.168.1.254)를 인수:

R2가 Gratuitous ARP 전송:
  "192.168.1.254는 누구? 192.168.1.254에게 알려줘"
  (자기 자신에 대해 질문 — 응답 예상 없음)

LAN의 모든 호스트가 이를 수신하고 업데이트:
  ARP 캐시: 192.168.1.254 (VIP) = R2의 MAC

결과: GARP 패킷 1개로 1초 이내에 모든 호스트가 게이트웨이 트래픽을 R2로 전환.
      (ARP 캐시 만료를 기다리는 것 대비: 20-60초)
```

## 요약

| 프로토콜 | 기능 | 방향 |
|---|---|---|
| ARP | IP → MAC | 순방향 |
| RARP | MAC → IP | 역방향 (구식) |
| DHCP | IP, 게이트웨이, DNS 할당 | 서버 → 클라이언트 |
| NDP (IPv6) | IP → MAC + 더 많은 기능 | 순방향 (IPv6) |
