# 서브넷 마스크와 서브네팅

## IPv4 주소 구조

IPv4 주소는 32비트로, 점으로 구분된 4개의 옥텟으로 작성된다:

```
192.168.1.100

11000000 . 10101000 . 00000001 . 01100100
   192        168        1          100
```

모든 IPv4 주소는 두 부분으로 나뉜다:
- **네트워크 부분**: 네트워크를 식별
- **호스트 부분**: 네트워크 내의 장치를 식별

**서브넷 마스크**는 네트워크 부분이 어디서 끝나고 호스트 부분이 어디서 시작하는지를 정의한다.

---

## 서브넷 마스크

서브넷 마스크도 32비트다. 네트워크 비트 = 1, 호스트 비트 = 0.

```
255.255.255.0  =  11111111.11111111.11111111.00000000
                  ─────────────────────────  ────────
                        네트워크 부분         호스트

IP:   192.168.1.100  →  네트워크: 192.168.1.0  /  호스트: .100
마스크: 255.255.255.0
```

### AND 연산: 네트워크 주소 찾기

```
IP:    192.168.1.100    11000000.10101000.00000001.01100100
마스크: 255.255.255.0    11111111.11111111.11111111.00000000
AND                     ────────────────────────────────────
네트워크: 192.168.1.0    11000000.10101000.00000001.00000000
```

---

## CIDR 표기법

**CIDR(Classless Inter-Domain Routing)**은 슬래시 접미사를 사용해 네트워크 비트 수를 나타낸다:

```
192.168.1.100/24
              ↑
              24 네트워크 비트 → 서브넷 마스크 = 255.255.255.0
```

| CIDR | 서브넷 마스크 | 서브넷당 호스트 | 사용 사례 |
|---|---|---|---|
| /8 | 255.0.0.0 | 16,777,214 | 대형 조직 |
| /16 | 255.255.0.0 | 65,534 | 중형 조직 |
| /24 | 255.255.255.0 | 254 | 소규모 사무실/네트워크 |
| /25 | 255.255.255.128 | 126 | /24의 절반 |
| /26 | 255.255.255.192 | 62 | /24의 1/4 |
| /27 | 255.255.255.224 | 30 | 소형 세그먼트 |
| /28 | 255.255.255.240 | 14 | 매우 소형 세그먼트 |
| /29 | 255.255.255.248 | 6 | 포인트 투 포인트 링크 |
| /30 | 255.255.255.252 | 2 | 라우터 링크 |
| /31 | — | 2 (브로드캐스트 없음) | RFC 3021 P2P |
| /32 | 255.255.255.255 | 1 | 단일 호스트/루프백 |

**서브넷당 호스트** = 2^(32 − 프리픽스) − 2 (네트워크와 브로드캐스트 주소를 빼야 함).

---

## 서브넷의 핵심 주소

`192.168.1.0/24`의 경우:

| 주소 | 값 | 설명 |
|---|---|---|
| 네트워크 주소 | 192.168.1.0 | 첫 번째 주소; 서브넷을 식별 |
| 첫 번째 사용 가능 | 192.168.1.1 | 첫 번째 호스트 주소 |
| 마지막 사용 가능 | 192.168.1.254 | 마지막 호스트 주소 |
| 브로드캐스트 | 192.168.1.255 | 마지막 주소; 서브넷의 모든 호스트에게 전송 |

네트워크 주소와 브로드캐스트 주소는 호스트에 할당할 수 없다.

---

## 서브네팅 예제

**문제**: `192.168.10.0/24`가 있고 4개의 동등한 서브넷이 필요하다.

**해결**: 호스트 부분에서 2비트 빌리기 (2² = 4 서브넷):
- 새 프리픽스: /24 + 2 = **/26**
- 서브넷당 호스트: 2^(32-26) − 2 = 62개 호스트

| 서브넷 | 네트워크 | 첫 번째 호스트 | 마지막 호스트 | 브로드캐스트 |
|---|---|---|---|---|
| 1 | 192.168.10.0/26 | .1 | .62 | .63 |
| 2 | 192.168.10.64/26 | .65 | .126 | .127 |
| 3 | 192.168.10.128/26 | .129 | .190 | .191 |
| 4 | 192.168.10.192/26 | .193 | .254 | .255 |

## VLSM — 가변 길이 서브넷 마스킹

VLSM은 같은 주소 공간 내에서 다양한 크기의 서브넷을 허용한다. 서브넷의 호스트 수가 크게 다를 때 효율적인 할당이 가능하다.

**예시**: 192.168.1.0/24를 다음에 할당:
- 부서 A: 100개 호스트 → /25 (126개 호스트)
- 부서 B: 50개 호스트 → /26 (62개 호스트)
- 부서 C: 25개 호스트 → /27 (30개 호스트)
- 라우터 링크 (×2): /30 (각 2개 호스트)

```
192.168.1.0/24:
├── 192.168.1.0/25    (부서 A: 126개 호스트)
└── 192.168.1.128/26  (부서 B: 62개 호스트)
    └── 192.168.1.192/27  (부서 C: 30개 호스트)
        └── 192.168.1.224/30  (라우터 링크 1)
            └── 192.168.1.228/30  (라우터 링크 2)
```

## 사설 IP 범위

RFC 1918은 사설 주소(공인 인터넷에서 라우팅 불가)를 정의한다:

| 범위 | CIDR | 블록 크기 |
|---|---|---|
| 10.0.0.0 – 10.255.255.255 | 10.0.0.0/8 | 16,777,216개 주소 |
| 172.16.0.0 – 172.31.255.255 | 172.16.0.0/12 | 1,048,576개 주소 |
| 192.168.0.0 – 192.168.255.255 | 192.168.0.0/16 | 65,536개 주소 |

사설 주소는 인터넷에 접근하기 위해 **NAT(네트워크 주소 변환)**을 사용한다.

### 특수 주소

| 범위 | 설명 |
|---|---|
| 127.0.0.0/8 | 루프백 (127.0.0.1 = localhost) |
| 169.254.0.0/16 | 링크 로컬 (APIPA — DHCP 실패 시) |
| 0.0.0.0/0 | 기본 경로 (모든 네트워크) |
| 255.255.255.255/32 | 제한된 브로드캐스트 |
| 224.0.0.0/4 | 멀티캐스트 |

## 클래스풀 주소 지정 (역사적)

CIDR 이전에는 주소가 고정 클래스로 나뉘었다:

| 클래스 | 범위 | 기본 마스크 | 호스트 |
|---|---|---|---|
| A | 1–126.x.x.x | /8 | 16M |
| B | 128–191.x.x.x | /16 | 65K |
| C | 192–223.x.x.x | /24 | 254 |
| D | 224–239.x.x.x | — | 멀티캐스트 |
| E | 240–255.x.x.x | — | 예약 |

클래스풀 주소 지정은 낭비가 심했다 (1000개의 호스트가 필요한 회사가 클래스 B = 65K 주소를 받았다). CIDR(1993)이 유연한 프리픽스 길이로 이를 대체했다.

---

## 실용적인 서브네팅 팁

**서브넷 증분** = 256 − 서브넷 마스크의 마지막 옥텟:
- /26 → 마스크가 192로 끝남 → 증분 = 256 − 192 = **64**
- 서브넷: .0, .64, .128, .192

**빠른 호스트 수**: 2^(32 − 프리픽스) − 2

**IP가 서브넷에 있는가?**
```
IP:      192.168.1.100
서브넷:  192.168.1.64/26

마스크:    255.255.255.192
IP AND 마스크: 192.168.1.64    (= .64에서 시작하는 /26 서브넷의 네트워크 주소) ✓
```

## 실습 예제: 6개 부서를 위한 /22 서브네팅

**주어진**: `10.0.0.0/22` (1022개 사용 가능한 호스트). VLSM으로 6개 부서에 서브넷 할당.

### 1단계: 요구사항 결정

| 부서 | 필요 호스트 | 서브넷 크기 |
|---|---|---|
| 엔지니어링 | 200 | /24 (254개 호스트) |
| 영업 | 100 | /25 (126개 호스트) |
| 재무 | 50 | /26 (62개 호스트) |
| 인사 | 25 | /27 (30개 호스트) |
| 경영진 | 10 | /28 (14개 호스트) |
| 서버 링크 (×2) | 각 2개 | /30 (2개 호스트) |

**VLSM 규칙**: 단편화를 피하기 위해 가장 큰 서브넷부터 할당.

### 2단계: 10.0.0.0/22의 이진 레이아웃

/22 블록은 다음을 포함:
```
10.0.0.0  에서  10.0.3.255   (4 × /24 = 1024개 주소)
```

### 3단계: 가장 큰 것부터 할당

**엔지니어링 → /24**
```
서브넷:    10.0.0.0/24
범위:      10.0.0.1  –  10.0.0.254
브로드캐스트: 10.0.0.255
남은 것:   10.0.1.0  –  10.0.3.255
```

**영업 → /25**
```
서브넷:    10.0.1.0/25
범위:      10.0.1.1  –  10.0.1.126
브로드캐스트: 10.0.1.127
```

**재무 → /26**
```
서브넷:    10.0.1.128/26
범위:      10.0.1.129  –  10.0.1.190
브로드캐스트: 10.0.1.191
```

**인사 → /27**
```
서브넷:    10.0.1.192/27
범위:      10.0.1.193  –  10.0.1.222
브로드캐스트: 10.0.1.223
```

**경영진 → /28**
```
서브넷:    10.0.1.224/28
범위:      10.0.1.225  –  10.0.1.238
브로드캐스트: 10.0.1.239
```

**라우터 링크 × 2 → /30**
```
링크 1:    10.0.1.240/30   (10.0.1.241 – 10.0.1.242)
링크 2:    10.0.1.244/30   (10.0.1.245 – 10.0.1.246)
```

### 4단계: 요약

```
10.0.0.0/22  (전체 블록)
├── 10.0.0.0/24     엔지니어링  (254개 호스트)
├── 10.0.1.0/25     영업        (126개 호스트)
├── 10.0.1.128/26   재무        (62개 호스트)
├── 10.0.1.192/27   인사        (30개 호스트)
├── 10.0.1.224/28   경영진      (14개 호스트)
├── 10.0.1.240/30   라우터 링크 1 (2개 호스트)
├── 10.0.1.244/30   라우터 링크 2 (2개 호스트)
└── 10.0.1.248/29   (미사용 — 미래에 사용 가능)
    10.0.2.0/23     (미사용 — 성장을 위한 여유)
```

### 5단계: 검증 — 10.0.1.150이 재무 서브넷에 있나?

```
재무 서브넷: 10.0.1.128/26
마스크:      255.255.255.192  (11111111.11111111.11111111.11000000)

10.0.1.150 이진법:
  10.0.1.150 = 00001010.00000000.00000001.10010110

마스크 적용:
  00001010.00000000.00000001.10000000
  = 10.0.1.128   ← 재무 서브넷의 네트워크 주소 ✓

범위: 10.0.1.128 – 10.0.1.191
10.0.1.150은 범위 내에 있음 → 재무 서브넷에 속함 YES.
```
