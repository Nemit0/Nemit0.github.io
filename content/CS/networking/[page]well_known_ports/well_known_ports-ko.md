# 잘 알려진 포트 번호

## 포트 번호란?

**포트 번호**는 호스트의 특정 프로세스 또는 서비스를 식별하는 16비트 정수(0–65535)다. IP 주소와 결합하면 **소켓** 엔드포인트를 고유하게 식별한다.

```
192.168.1.100 : 443
   IP 주소     포트
```

포트 번호는 OS에게 어떤 애플리케이션이 들어오는 데이터를 받아야 하는지 알려준다. 다른 포트를 사용하는 한 동일한 IP에서 여러 서비스가 실행될 수 있다.

---

## IANA 포트 범위

IANA(인터넷 할당 번호 기관)는 포트를 세 가지 범위로 나눈다:

| 범위 | 이름 | 설명 |
|---|---|---|
| 0–1023 | **잘 알려진 포트** | 표준 서비스에 할당; 바인딩에 루트/관리자 필요 |
| 1024–49151 | **등록 포트** | 애플리케이션이 등록; 루트 불필요 |
| 49152–65535 | **동적/임시 포트** | 클라이언트 소켓에 자동으로 할당 |

브라우저가 웹 서버에 연결할 때, 서버는 443번 포트(잘 알려진)에서 수신 대기하고, 브라우저는 54321 같은 임의의 임시 포트를 출발 포트로 사용한다.

## 잘 알려진 포트 (0–1023)

### 웹과 보안

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **80** | TCP | HTTP (웹, 평문) |
| **443** | TCP | HTTPS (웹, TLS 암호화) |
| **8080** | TCP | HTTP 대체 (개발/프록시) |
| **8443** | TCP | HTTPS 대체 |

### 파일 전송

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **20** | TCP | FTP 데이터 |
| **21** | TCP | FTP 제어 |
| **22** | TCP | SSH, SFTP |
| **69** | UDP | TFTP (간단한 FTP, 인증 없음) |

### 이메일

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **25** | TCP | SMTP (이메일 발송, 서버 간) |
| **110** | TCP | POP3 (이메일 수신, 다운로드) |
| **143** | TCP | IMAP (이메일 수신, 동기화) |
| **465** | TCP | SMTPS (TLS를 통한 SMTP) |
| **587** | TCP | SMTP 제출 (클라이언트 간) |
| **993** | TCP | IMAPS (TLS를 통한 IMAP) |
| **995** | TCP | POP3S (TLS를 통한 POP3) |

### 이름과 디렉토리 서비스

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **53** | TCP/UDP | DNS |
| **389** | TCP | LDAP (디렉토리 서비스) |
| **636** | TCP | LDAPS (TLS를 통한 LDAP) |

### 네트워크 서비스

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **67** | UDP | DHCP 서버 |
| **68** | UDP | DHCP 클라이언트 |
| **123** | UDP | NTP (시간 동기화) |
| **161** | UDP | SNMP (쿼리) |
| **162** | UDP | SNMP 트랩 (경보) |
| **514** | UDP | Syslog |

### 원격 접근

| 포트 | 프로토콜 | 서비스 |
|---|---|---|
| **22** | TCP | SSH |

## TCP vs UDP 일반 서비스

대부분의 서비스는 TCP(신뢰성 필요)를 사용하지만, 일부는 UDP(지연)를 선호한다:

**UDP 서비스** (낮은 지연 또는 브로드캐스트 친화적):
- DNS (53) — 단순 쿼리/응답; 앱 계층에서 재시도
- DHCP (67/68) — 브로드캐스트 기반
- SNMP (161/162) — 모니터링
- NTP (123) — 시간 동기화 (신뢰성보다 정밀도가 중요)
- Syslog (514) — 발사 후 잊어버리는 로깅
- TFTP (69) — 단순 파일 전송
- DNS (53)는 큰 응답과 존 전송에 TCP도 지원

## 임시 포트

클라이언트가 연결을 시작할 때, OS가 동적 범위에서 임시 **임시 포트**를 할당한다:

```bash
# Linux에서 임시 포트 범위 확인
cat /proc/sys/net/ipv4/ip_local_port_range
# 32768   60999

# 활성 연결
ss -tn
# ESTAB  0  0  192.168.1.100:54321  10.0.0.2:443
#                       ↑ 임시          ↑ 잘 알려진
```

### 포트 소진

호스트가 같은 목적지에 많은 나가는 연결을 열면 임시 포트가 소진될 수 있다. 해결책:
- TIME_WAIT 감소 (`SO_REUSEADDR`, `SO_LINGER`)
- 임시 범위 확장
- 연결 풀링 사용

## 열린 포트 확인

```bash
# 수신 대기 포트 목록 (Linux)
ss -tlnp          # TCP
ss -ulnp          # UDP
netstat -tlnp     # 오래된 도구

# 예시 출력:
# tcp  LISTEN  0  128  0.0.0.0:22    0.0.0.0:*  users:(("sshd",pid=1234))
# tcp  LISTEN  0  128  0.0.0.0:443   0.0.0.0:*  users:(("nginx",pid=5678))

# 원격 호스트 포트 스캔 (허가 필요)
nmap -p 22,80,443 192.168.1.1
nmap -sU -p 53,161 192.168.1.1   # UDP 스캔

# 특정 포트 열려 있는지 확인
nc -zv 192.168.1.1 22             # TCP
nc -zuv 192.168.1.1 53            # UDP
```

## 보안 고려사항

- **미사용 포트 닫기**: 모든 열린 포트는 잠재적인 공격 표면
- **방화벽 규칙**: 필요한 출처에서 필요한 포트만 허용
- **기본값 변경**: SSH를 포트 22에서 실행하지 않도록 (노이즈를 줄이기 위해 높은 포트로 변경)
- **포트 스캔 감지**: 비정상적인 포트 스캔은 종종 공격을 예고
- **암호화된 대안**: HTTP(80) 대신 HTTPS(443), IMAP(143) 대신 IMAPS(993), Telnet(23) 대신 SSH(22)

## 실습 예제: URL 입력 시 — 포트 동작

**시나리오**: 브라우저에 `https://mail.google.com`을 입력한다. 관련된 모든 포트:

```
1단계: DNS 해석 (UDP 포트 53)
──────────────────────────────────
브라우저 → OS → 로컬 리졸버 (8.8.8.8:53)
  쿼리:  mail.google.com A 레코드?
  답변: 142.250.80.37
프로토콜: UDP, 목적 포트 53, 출발 포트 52631 (임시)

2단계: TCP + TLS 연결 (TCP 포트 443)
───────────────────────────────────────
브라우저 → 142.250.80.37:443
  TCP SYN: 192.168.1.100:54321 → 142.250.80.37:443
  TCP SYN-ACK
  TCP ACK
  TLS ClientHello → ServerHello → Certificate → Finished
  [암호화된 HTTP/2 또는 HTTP/3 시작]

3단계: TLS 내의 HTTP 요청 (포트 443)
──────────────────────────────────────
GET /mail/u/0/ HTTP/2
Host: mail.google.com
[모든 데이터가 포트 443에서 암호화됨]
```

**하나의 HTTPS 페이지 로드에 관련된 포트**:
| 동작 | 프로토콜 | 포트 | 방향 |
|---|---|---|---|
| DNS 조회 | UDP | 53 | 클라이언트 → DNS 서버 |
| HTTPS 연결 | TCP | 443 | 클라이언트 → 웹 서버 |
| (클라이언트 임시 포트) | TCP | ~54321 | 출발 포트, OS가 선택 |

## 실습 예제: 이메일 발송 경로 — 여러 포트

**시나리오**: Alice(`alice@company.com`)가 `bob@gmail.com`에게 이메일을 보낸다.

```
1단계: Alice의 이메일 클라이언트 → 회사의 SMTP 제출 서버
  포트: 587 (SMTP 제출, 인증됨, STARTTLS를 통한 TLS)
  인증: 사용자명/비밀번호
  발신: alice@company.com  수신: bob@gmail.com
  메시지: "안녕하세요 Bob!"

2단계: 회사의 SMTP 서버 → Gmail의 SMTP 서버 (서버 간)
  DNS MX 조회: gmail.com MX? → aspmx.l.google.com (포트 53)
  포트: 25 (SMTP 서버 간, 보통 STARTTLS 포함)
  인증 없음 — 신뢰할 수 있는 서버 간 릴레이

3단계: Bob의 Gmail 클라이언트가 이메일 가져오기
  옵션 A: IMAP (포트 993 — IMAPS)
    → 동기화, 서버에 메일 유지, 여러 기기 지원
  옵션 B: POP3 (포트 995 — POP3S)
    → 메일 다운로드, 보통 서버에서 삭제

완전한 이메일 여정의 포트 요약:
  587  → 클라이언트가 발신 서버에 제출
  25   → 서버 간 릴레이
  53   → 릴레이 중 DNS MX 조회
  993  → 수신자가 IMAP으로 가져옴 (또는 POP3의 경우 995)
```

## 실습 예제: 포트 스캔에서 서비스 식별

```bash
$ nmap -sV 192.168.1.100

PORT     STATE  SERVICE    VERSION
22/tcp   open   ssh        OpenSSH 8.9
25/tcp   open   smtp       Postfix SMTP
80/tcp   open   http       nginx 1.24
443/tcp  open   https      nginx 1.24
3306/tcp open   mysql      MySQL 8.0.32
8080/tcp open   http-proxy Squid proxy 5.7
```

**스캔 읽기**:
- 포트 22: SSH 서버 — 원격 관리에 유효
- 포트 25: SMTP 열림 — 이 서버가 직접 이메일 발송/수신
- 포트 80 + 443: 웹 서버 (nginx) — HTTP + HTTPS
- 포트 3306: MySQL 노출! **보안 위험** — 네트워크에 열려 있어서는 안 됨; 127.0.0.1만 바인딩
- 포트 8080: 프록시 서비스 실행 중

**보안 강화 조치**:
```bash
# MySQL을 로컬호스트만으로 제한
# /etc/mysql/mysql.conf.d/mysqld.cnf에서:
bind-address = 127.0.0.1

# 또는 방화벽으로:
ufw deny 3306
ufw allow from 10.0.0.0/8 to any port 3306  # 내부 네트워크에서만 허용

# 변경 후 수신 대기 주소 확인:
ss -tlnp | grep 3306
# tcp  LISTEN  0  80  127.0.0.1:3306  0.0.0.0:*  users:(("mysqld",pid=1234))
# ↑ 이제 로컬호스트만 — 외부 연결 차단됨
```
