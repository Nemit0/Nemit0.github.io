# 라우팅 기초

## 라우팅이란?

**라우팅**은 하나 이상의 네트워크를 거쳐 출발지에서 목적지까지 네트워크 트래픽의 경로를 선택하는 과정이다 (레이어 3). **라우터**는 **IP 주소**를 사용해 네트워크 간에 패킷을 전달하는 장치다.

라우팅이 답하는 질문: "이 패킷의 목적지 IP를 고려할 때 어느 인터페이스로 전송해야 하는가?"

---

## 라우팅 테이블

모든 라우터는 **라우팅 테이블** — 알려진 네트워크와 해당 네트워크에 도달하는 방법의 목록을 유지한다.

```bash
# Linux 라우팅 테이블
ip route show

# 출력:
default via 192.168.1.1 dev eth0
10.0.0.0/8 via 10.1.1.1 dev eth1
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.100
```

### 라우팅 테이블 항목 필드

| 필드 | 설명 |
|---|---|
| **목적지** | 네트워크 주소 / CIDR |
| **다음 홉** | 전달할 다음 라우터의 IP |
| **인터페이스** | 나가는 인터페이스 |
| **메트릭** | 이 경로의 비용 (낮을수록 선호) |
| **관리적 거리** | 라우팅 소스의 신뢰성 |

### 최장 프리픽스 매치

여러 경로가 목적지와 일치할 때, 라우터는 **가장 구체적인** (가장 긴 프리픽스)를 사용한다:

```
경로:
  10.0.0.0/8   via 10.1.1.1
  10.1.0.0/16  via 10.2.2.2
  10.1.2.0/24  via 10.3.3.3

10.1.2.5로의 패킷:
  /8와 일치 → 가능
  /16와 일치 → 더 구체적
  /24와 일치 → 가장 구체적 → 10.3.3.3 사용
```

---

## 경로 유형

### 연결된 경로

인터페이스에 IP가 구성되고 활성화되면 자동으로 추가된다.

```
192.168.1.0/24 dev eth0 (직접 연결)
```

### 정적 경로

관리자가 수동으로 구성한다. 단순하고 예측 가능하지만 장애에 적응하지 못한다.

```bash
# 정적 경로 추가
ip route add 10.0.0.0/8 via 192.168.1.1

# 기본 경로 (알 수 없는 모든 것을 이 게이트웨이로)
ip route add default via 192.168.1.1
```

**사용 사례**: 소형 네트워크, 특정 정책 라우팅, 최후 수단 경로.

### 동적 경로

**라우팅 프로토콜**을 통해 다른 라우터에서 학습된다. 토폴로지 변경 시 자동으로 적응한다.

---

## 정적 vs 동적 라우팅

| 측면 | 정적 | 동적 |
|---|---|---|
| 구성 | 수동 | 프로토콜을 통해 자동 |
| 확장성 | 나쁨 (많은 경로) | 좋음 |
| 수렴 | 즉각적 (프로토콜 불필요) | 프로토콜에 따라 다름 |
| 장애 허용 | 수동 수정 필요 | 자동 재계산 |
| 오버헤드 | 없음 | 프로토콜 메시지가 대역폭 소비 |
| 사용 사례 | 소형 네트워크, 특정 경로 | 크고 복잡한 네트워크 |

---

## 라우팅 프로토콜

라우팅 프로토콜은 라우터 간에 토폴로지 정보를 교환해서 라우팅 테이블을 자동으로 구축한다.

### 분류

**범위에 따라**:
- **IGP (내부 게이트웨이 프로토콜)**: 단일 조직의 네트워크(AS) 내부
- **EGP (외부 게이트웨이 프로토콜)**: 다른 조직/ISP 간

**알고리즘에 따라**:
- **거리 벡터**: 각 라우터가 목적지까지의 거리를 알고; 이웃에게 테이블을 공유
- **링크 상태**: 각 라우터가 완전한 토폴로지를 알고; SPF 알고리즘을 실행

---

## 거리 벡터: RIP

**RIP(Routing Information Protocol)** — 가장 단순한 라우팅 프로토콜.

- 메트릭: **홉 수** (최대 15; 16 = 도달 불가)
- 업데이트: 30초마다 전체 테이블을 브로드캐스트
- 수렴: 느림 (벨만-포드 알고리즘)
- 루프 방지: 스플릿 호라이즌, 경로 독화
- 버전: RIPv1 (클래스풀), RIPv2 (CIDR 지원), RIPng (IPv6)

```
라우터 A가 아는 것:
  네트워크 10.0.0.0 — 1홉
  네트워크 172.16.0.0 — 2홉

라우터 A가 이웃에게 알림:
  "나는 10.0.0.0을 1홉에 도달할 수 있어"
  "나는 172.16.0.0을 2홉에 도달할 수 있어"
```

**문제점**: 최대 15홉, 느린 수렴, 대형 네트워크에서 대역폭 낭비.

---

## 링크 상태: OSPF

**OSPF(Open Shortest Path First)** — 가장 널리 사용되는 IGP.

- 메트릭: **비용** (기본적으로 인터페이스 대역폭 기반; 낮은 비용 = 빠른 링크)
- 알고리즘: **다익스트라 SPF (최단 경로 우선)**
- 토폴로지: 각 라우터가 완전한 **링크 상태 데이터베이스(LSDB)**를 구축
- 업데이트: 토폴로지 변경 시만 전송 (이벤트 기반), 주기적이지 않음
- 수렴: 빠름
- 영역: 계층적 설계 (영역 0 = 백본)

```
OSPF 비용 = 참조 대역폭 / 인터페이스 대역폭
기본 참조: 100 Mbps
  패스트이더넷 (100 Mbps): 비용 = 1
  기가비트이더넷 (1 Gbps): 비용 = 1 (조정하지 않으면 패스트이더넷과 동일)
  T1 (1.544 Mbps): 비용 = 64
```

**OSPF 인접 상태**: Down → Init → 2-Way → ExStart → Exchange → Loading → **Full**

### OSPF 영역

- **영역 0 (백본)**: 모든 다른 영역이 영역 0에 연결해야 함
- **ABR (영역 경계 라우터)**: 여러 영역을 연결
- **ASBR (AS 경계 라우터)**: 외부 경로를 OSPF로 재분배

---

## BGP — 경계 게이트웨이 프로토콜

**BGP**는 인터넷의 라우팅 프로토콜 — 널리 사용되는 유일한 EGP다.

- **자율 시스템(AS)** 간 라우팅을 관리 — 자체 라우팅 정책을 가진 조직
- **경로 벡터** 프로토콜 (거리 벡터의 확장; 전체 AS 경로를 추적)
- 메트릭: 최선의 경로를 선택하기 위해 많은 **속성** 사용 (AS-PATH, LOCAL_PREF, MED 등)
- TCP 포트 **179** (신뢰적 전송)
- 유형: eBGP (AS 간), iBGP (AS 내)

### BGP 경로 선택 (간략화, 순서대로)

1. 가장 높은 LOCAL_PREF (내부 정책 선호)
2. 가장 짧은 AS-PATH (가장 적은 AS 홉)
3. 가장 낮은 MED (이웃의 선호 진입 선호)
4. eBGP가 iBGP보다 우선
5. 다음 홉까지 가장 낮은 IGP 메트릭
6. 가장 낮은 라우터 ID (타이브레이커)

```
AS 100 ──── eBGP ──── AS 200 ──── eBGP ──── AS 300
(ISP A)               (고객)              (ISP B)
```

## 관리적 거리

여러 라우팅 소스가 같은 네트워크를 알고 있을 때, 라우터는 **가장 낮은 관리적 거리(AD)**를 가진 소스를 신뢰한다:

| 소스 | AD |
|---|---|
| 연결된 경로 | 0 |
| 정적 경로 | 1 |
| EIGRP 요약 | 5 |
| eBGP | 20 |
| OSPF | 110 |
| RIP | 120 |
| iBGP | 200 |
| 알 수 없음 | 255 (사용 불가) |

OSPF와 RIP 모두 10.0.0.0/8 경로를 알고 있으면, OSPF가 이긴다 (110 < 120).

---

## 프로토콜별 라우팅 메트릭

| 프로토콜 | 메트릭 |
|---|---|
| RIP | 홉 수 |
| OSPF | 비용 (대역폭 기반) |
| EIGRP | 복합 (대역폭 + 지연 + 부하 + 신뢰성) |
| BGP | 경로 속성 (단순한 메트릭이 아님) |
| 정적 | 수동 할당 (기본값 0) |

---

## 라우터의 패킷 전달 방법

```
1. 인터페이스에서 패킷 수신
2. 목적지 IP 확인
3. 라우팅 테이블 조회 (최장 프리픽스 매치)
4. 일치 항목 발견 시:
     a. TTL 감소 (TTL=0이면 드롭 + ICMP 시간 초과 전송)
     b. 목적지 MAC을 다음 홉 MAC으로 업데이트 (필요시 ARP)
     c. 적절한 인터페이스로 전달
5. 일치 없고 기본 경로 있음: 기본 게이트웨이로 전달
6. 일치 없고 기본 없음: 드롭 + ICMP 목적지 도달 불가 전송
```

## 실습 예제: 최장 프리픽스 매치 상세

**라우터 R1의 라우팅 테이블**:

```
목적지               다음 홉       인터페이스   메트릭  AD   소스
0.0.0.0/0         203.0.113.1    eth0        1       1    정적 (기본)
10.0.0.0/8        10.1.0.1       eth1        10      110  OSPF
10.1.0.0/16       10.1.0.1       eth1        5       110  OSPF
10.1.2.0/24       10.2.0.1       eth2        2       110  OSPF
10.1.2.128/25     10.3.0.1       eth3        1       110  OSPF
192.168.1.0/24    —              eth0        0       0    연결됨
```

**들어오는 패킷과 전달 결정**:

| 패킷 목적지 | 일치 항목 | 최장 매치 | 다음 홉 |
|---|---|---|---|
| `10.1.2.200` | /8, /16, /24, /25, 기본 | **/25** (가장 많은 비트 일치) | eth3 → 10.3.0.1 |
| `10.1.2.50` | /8, /16, /24, 기본 | **/24** | eth2 → 10.2.0.1 |
| `10.1.5.1` | /8, /16, 기본 | **/16** | eth1 → 10.1.0.1 |
| `10.5.0.1` | /8, 기본 | **/8** | eth1 → 10.1.0.1 |
| `172.16.0.1` | 기본만 | **/0** (기본) | eth0 → 203.0.113.1 |

### 10.1.2.200에 대한 이진 프리픽스 매칭

```
패킷:         10  .  1  .  2  .  200
이진:    00001010.00000001.00000010.11001000

/8 경로:  00001010.xxxxxxxx.xxxxxxxx.xxxxxxxx  ← 첫 8비트 일치 ✓
/16 경로: 00001010.00000001.xxxxxxxx.xxxxxxxx  ← 첫 16비트 일치 ✓
/24 경로: 00001010.00000001.00000010.xxxxxxxx  ← 첫 24비트 일치 ✓
/25 경로: 00001010.00000001.00000010.1xxxxxxx  ← 첫 25비트 일치 ✓
          (200 = 11001000, 첫 번째 비트 = 1 → /25의 128 경계 일치)

승자: /25 (25 > 24 > 16 > 8) → eth3 → 10.3.0.1로 전달
```

## 실습 예제: OSPF 비용 계산

**네트워크 토폴로지**:

```
R1 ──── GigE (1 Gbps) ──── R2 ──── FastE (100 Mbps) ──── R3
 └────── T1 (1.544 Mbps) ──────────────────────────────── R3
```

**OSPF 참조 대역폭**: 100 Mbps (기본값)

```
비용 공식: 참조 대역폭 / 인터페이스 대역폭

R1→R2 GigE:  100 Mbps / 1000 Mbps = 0.1 → 올림하면 1
R2→R3 FastE: 100 Mbps / 100 Mbps  = 1
R1→R3 T1:   100 Mbps / 1.544 Mbps = 64.77 → 64로 반올림
```

**경로 비용**:
```
R1→R2→R3 경로: 비용 = 1 (GigE) + 1 (FastE) = 2
R1→R3 T1 경로: 비용 = 64

OSPF가 R1→R2→R3를 선택 (비용 2 << 비용 64). 올바른 동작.
```

**문제**: 기본 참조로 GigE와 100 Mbps FastEthernet 모두 비용=1.

```
수정: 현대 네트워크에 맞게 참조 대역폭 증가
ip ospf auto-cost reference-bandwidth 10000  (10 Gbps)

GigE:   10000 / 1000  = 10
FastE:  10000 / 100   = 100
10GigE: 10000 / 10000 = 1

이제 OSPF가 FastEthernet보다 GigE를 올바르게 선호함.
```

## 실습 예제: 관리적 거리 타이브레이커

라우터 R1이 두 소스에서 동시에 `192.168.50.0/24`를 학습한다:

```
OSPF: 192.168.50.0/24 via 10.0.0.2 (인터페이스 eth1), 메트릭 30
RIP:  192.168.50.0/24 via 10.0.1.5 (인터페이스 eth2), 메트릭 3홉
```

RIP의 홉 수(3)가 OSPF의 비용(30)보다 수치적으로 낮더라도, **AD가 결정**한다:

```
OSPF AD = 110
RIP  AD = 120

110 < 120  →  OSPF 경로가 이기고, 라우팅 테이블에 설치됨.
RIP 경로는 RIP 자체 데이터베이스에 백업으로 유지됨
(OSPF 경로가 사라지면 RIP 경로가 승격될 수 있음).
```

동일 프로토콜의 경로만 메트릭을 비교한다. 다른 프로토콜은 AD를 먼저 사용한다.
